## 0. 总体定位

本项目面向“工控安全研究员 / 工业互联网安全研究员 / 工控漏洞分析工程师”岗位，目标不是只会背概念，而是建立一套能够解释真实工业控制系统运行逻辑、协议行为、攻击路径和防护方法的知识体系。

- DCS：分布式控制、控制站、操作站、工程师站、历史站、冗余、无扰切换。

- 工控安全：资产识别、协议分析、未授权访问、寄存器篡改、程序下载、反馈欺骗、Stuxnet 思路、MITRE ATT&CK for ICS 思维。
- 实操能力：Wireshark 抓包、Python socket/Scapy/pymodbus、OpenPLC/CODESYS/Factory I/O/PLCSIM、简单逆向分析。

全书写作原则：

5. 每节都要有“面试回答模板”和“可能追问”。

# 第一册 自动化与工业控制基础

---

## 第一章 控制（Control）的本质

### 1.1 为什么世界需要自动控制

自动控制不是为了“让设备自己动”，而是为了让工业过程在扰动下保持稳定、安全、连续运行。工业对象不是静止的，它们持续受到煤质变化、负荷变化、环境温度变化、阀门磨损、传感器漂移、工艺耦合等扰动。如果没有控制系统，锅炉压力、反应器温度、精馏塔液位、压缩机转速都无法长期维持在安全范围。

核心认知：

- 控制系统真正控制的是过程变量（PV），而不是 PLC 或阀门本身。
- PLC、DCS、调节阀、变送器只是实现控制的工具。
- 控制系统的基础结构是：设定值 SP → 控制器 → 执行机构 → 被控对象 → 传感器 → 反馈 PV。
- 工控安全研究员必须理解物理变量，因为一次“写寄存器”在 IT 系统中可能只是改数据，在工控系统中可能意味着改变反应器温度、压力或设备转速。

面试表达：

> 自动控制的核心目标不是单纯替代人工，而是让工业过程在扰动下保持稳定运行。对于煤化工、电厂、炼化等连续过程工业，关键变量如压力、温度、液位、流量具有动态特性和安全边界，因此必须通过传感器、控制器和执行机构形成闭环控制。工控安全研究员要关注的不只是 PLC 是否被入侵，而是控制闭环是否被破坏、反馈是否真实、执行是否完整。

---

### 1.2 反馈为什么优于开环

开环控制只执行预设动作，不知道结果。例如打开水阀 30 分钟后关闭，它不知道水是否足够、是否泄漏、外界条件是否改变。闭环控制通过测量结果形成误差 e(t)=SP-PV，并不断修正。

反馈的本质：

- 反馈并不需要知道扰动是什么，只需要知道结果偏离了目标。
- 反馈让系统具备纠错能力。
- 反馈系统必须在响应速度和稳定性之间权衡：修正太快容易振荡，修正太慢无法及时抑制扰动。

安全视角：

- 攻击 PV：让控制器看到假的世界。
- 攻击 SP：改变控制目标。
- 攻击 MV：控制器计算正确，但执行错误。
- 攻击反馈链路：最危险，因为系统可能在“自认为正常”的状态下持续失控。

---

### 1.3 工业对象为什么是动态系统

工业对象几乎都不是“输入立即变输出”的静态对象，而是具有能量、质量或动量存储的动态系统。

典型动态特性：

- 惯性：锅炉、反应釜、管道、液位、电机转子都不会瞬时变化。
- 延迟：热量传递、物料运输、传感器采样、网络通信都会引入滞后。
- 非线性：阀门开度和流量不是线性关系，煤质与发热量不是固定关系，反应速率随浓度和温度变化。
- 耦合：精馏塔调回流不仅影响液位，还影响塔顶温度、塔底温度、产品纯度和能耗。

研究员思维：

> 看到一次攻击动作时，不要只问“改了哪个寄存器”，还要问这个寄存器作用于哪个动态对象、对象响应多快、是否有滞后、影响是否会传播到其他控制回路、最终是收敛、振荡还是失稳。

---

### 1.4 状态、微分方程、拉普拉斯和传递函数为什么出现

工业对象具有记忆，当前输出不仅取决于当前输入，还取决于过去输入。因此仅看当前 PV 不足以预测未来，还必须知道系统内部状态。

关键概念：

- 状态：能够决定系统未来行为的一组最小变量。例如摆的状态是角度和角速度；锅炉的状态可能包括温度、压力、水位、燃料流量、蒸汽流量。
- 微分方程：控制理论研究变量随时间如何变化，所以必须用导数描述变化速度。
- 拉普拉斯变换：把微分运算变成代数运算，方便工程计算和控制器设计。
- 传递函数：一个系统对于输入信号的动态响应“身份证”，描述输入如何变成输出。

安全意义：

- 真正的攻击者关心对象如何响应，不只是能不能写入数值。
- Stuxnet 的高级之处不是“能写 PLC 程序”，而是理解了离心机动态响应，并在特定时间段改变频率，同时伪造 HMI 反馈。

---

### 1.5 一阶系统为什么呈指数响应

一阶系统的本质是：系统变化速度与距离平衡点的误差成比例。离目标越远，变化越快；越接近目标，变化越慢。液位、温度、压力、热交换器、简单反应器等很多工业过程都可以近似为一阶系统。

核心概念：

- 平衡：进入量等于流出量，或输入能量等于损失能量。
- 时间常数 τ：系统遗忘初始状态、接近新稳态所需的特征时间。
- 一阶对象响应慢不代表控制器差，可能是对象自身热容、容量或物料积累决定的。

安全意义：

- 对大时间常数对象，攻击影响可能延迟出现。
- 高频快速篡改对慢对象可能无效，因为对象还没来得及响应。
- APT 攻击通常会根据对象动态特性设计持续扰动，而不是随机改值。

---

### 1.6 二阶系统为什么会振荡

二阶系统的本质是两个能量存储器之间交换能量，例如弹簧-质量系统中的势能和动能、电机中的电磁能和机械动能、离心机中的驱动与转子惯性。二阶系统天然可能振荡。

关键概念：

- 自然频率 ω：系统倾向于振动的速度。
- 阻尼比 ζ：系统能否把多余能量消耗掉。
- 过阻尼：稳定但慢。
- 临界阻尼：最快且不振荡。
- 欠阻尼：有超调和振荡。
- 共振：输入频率接近系统自然频率时，能量不断叠加，可能造成机械破坏。

PID 关联：

- P 增强拉回力度，提高速度，但可能增大超调。
- I 消除稳态误差，但可能导致积分饱和和过冲。
- D 提供趋势预测和阻尼，但对噪声敏感。

安全意义：

- 攻击 PID 参数可能本质上是在降低阻尼或改变稳定裕度。
- 周期性扰动比单次改值更危险，因为它可能激发系统动态特性。

---

### 1.7 极点为什么决定系统命运

极点不是考试中的“特征方程的根”这么简单，它代表系统的自然响应方式。扰动之后，系统是衰减、持续振荡还是发散，取决于极点位置。

核心认知：

- 极点实部小于 0：响应衰减，系统稳定。
- 极点实部等于 0：不衰减，临界稳定或持续振荡。
- 极点实部大于 0：响应发散，系统不稳定。
- 虚部对应振荡频率，实部对应衰减速度。
- 阻尼本质上是系统耗散多余能量的能力。

PID 关联：

- 调 PID 参数本质上是在移动闭环极点。
- Kp 太大、Ki 太强、延迟太大，都可能把极点推向右半平面，使系统由稳定变成振荡或发散。

安全意义：

- 攻击者修改 PID 参数、滤波参数、采样周期、延迟补偿，可能不是立即改 PV，而是改变闭环极点，让系统慢慢失稳。

## 第二章 工业控制系统架构

### 2.1 为什么现代工厂一定采用分层控制

现代工业控制系统不是一台大电脑控制全厂，而是按照实时性、控制粒度、可靠性和管理需求进行分层。

典型层级：

| 层级 | 响应时间 | 关注对象 |
|---|---:|---|
| 传感器/执行器 | μs~ms | 物理信号 |
| PLC/DCS 控制层 | 10~100ms | 控制回路 |
| HMI/SCADA | 0.5~2s | 操作监视 |
| Historian | 秒~分钟 | 历史数据 |
| MES | 分钟~小时 | 生产调度 |
| ERP | 天~月 | 企业经营 |

关键认知：

- 越往下，实时性越强，控制权限越高。
- 越往上，数据价值越高，实时性要求越低。
- MES/ERP 不能直接控制阀门，因为它们不是实时系统。
- 工控安全中的分区分域，本质是避免上层信息系统直接破坏底层控制闭环。

---

### 2.2 4~20mA 信号如何贯穿工业闭环

4~20mA 是工业连续变量的编码方式，不是单纯电气标准。

完整链路：

```text
压力/温度/液位等物理量
  ↓
变送器
  ↓
4~20mA
  ↓
AI 模块
  ↓
ADC 转换
  ↓
工程量 PV
  ↓
PID 计算
  ↓
AO 模块
  ↓
4~20mA 输出
  ↓
阀门定位器
  ↓
调节阀动作
  ↓
工艺过程变化
  ↓
再次反馈
```

为什么是 4~20mA：

- 4mA 代表真实零点，0mA 可表示断线故障。
- 电流信号适合长距离传输，抗干扰能力强。
- 便于做故障诊断和量程映射。

安全意义：

- 攻击变送器、信号链路、AI 模块、PLC 数据区、AO 输出、阀门定位器都可能破坏闭环。
- 最危险的是反馈欺骗：PLC 以为 PV 正常，实际物理过程已经异常。

---

### 2.3 PLC 扫描周期的本质

PLC 不采用普通计算机的事件驱动，而采用循环扫描，核心目标是确定性。

典型流程：

```text
输入采样 → 冻结输入映像区 → 执行程序 → 修改输出映像区 → 输出刷新 → 通信/诊断 → 下一轮
```

为什么冻结输入：

- 如果程序前半段看到 I0.0=1，后半段看到 I0.0=0，同一扫描周期内逻辑会不一致。
- 输入映像区相当于扫描开始时给现场世界“拍照”。

为什么输出映像区存在：

- 程序执行过程中可能多次写 Q0.0。
- 现场输出只在扫描末尾统一刷新，避免继电器/阀门/电机在一次扫描中抖动。

安全意义：

- 扫描周期决定 PLC 看到世界的节奏。
- 高级攻击可能利用采样时机、扫描周期和任务调度，让 HMI 看到正常数据、PLC 执行异常逻辑。

---

### 2.4 PLC 程序执行模型：梯形图背后的 CPU

梯形图不是 PLC 真正执行的东西，它只是面向工程师的人机交互语言。PLC 内部最终执行的是布尔表达式、内部指令或机器码。

梯形图：

```text
I0.0 ---- I0.1 ---- ( Q0.0 )
```

等价于：

```c
Q0_0 = I0_0 && I0_1;
```

再底层是：

```assembly
LOAD I0_0
LOAD I0_1
AND
STORE Q0_0
```

关键认知：

- 常开触点本质是读取变量是否为 1。
- 常闭触点本质是读取变量是否为 0。
- 线圈本质是写变量。
- M 区本质是共享状态缓存，不是真实继电器。
- 梯形图采用左到右、上到下执行，是为了保证确定性。

安全意义：

- 攻击 OB/FB/FC、DB、M 区、输入映像区、输出映像区，都会改变程序执行结果。
- PLC 逆向不能只看图形，还要理解代码、状态和物理过程。

---

### 2.5 PLC 内存模型

PLC 内存分区的根本原因不是“功能分类”，而是“数据生命周期不同”。

主要区域：

| 区域 | 本质 | 生命周期/用途 |
|---|---|---|
| I 区 | 输入过程映像 | 一次扫描的现场快照 |
| Q 区 | 输出过程映像 | 一次扫描后统一输出 |
| M 区 | 共享逻辑缓存 | 程序运行状态、中间结果 |
| DB 区 | 业务状态/对象实例 | PID、配方、设定值、历史误差、积分 |
| L 区 | 局部变量 | 函数/块执行期间临时存在 |
| T/C | 定时器/计数器 | 时间/次数状态 |
| Retain | 掉电保持数据 | 重启后仍保留，危险性较高 |

DB 的重要性：

- PID 的积分、上次误差、模式、输出状态必须保存在 DB。
- HMI 修改 SP、报警限、配方，本质上通常是写 DB。
- 攻击 Retain DB 可以实现持久化影响，PLC 重启后恶意参数仍在。

面试表达：

> M 区主要用于共享和缓存运行时逻辑状态；DB 区用于保存控制对象的业务状态和实例数据，例如 PID 积分、设定值、配方和模式。攻击 M 区通常影响当前逻辑状态，攻击 DB 区尤其是 Retain DB 可能长期改变控制行为。

---

### 2.6 PLC Runtime：OB、FB、FC 的本质

现代 PLC 有实时运行时系统，负责任务调度、内存管理、I/O 刷新、通信处理和异常处理。

核心概念：

- OB：Runtime 调度的实时任务，相当于 RTOS Task。
- OB1：循环任务。
- OB35：周期中断任务，适合固定周期 PID。
- OB40：硬件中断。
- OB100：启动初始化，可被用于持久化初始化逻辑。
- FC：无状态函数，执行结束局部变量释放。
- FB：有状态对象，必须配实例 DB 保存内部状态。

为什么 PID 不宜放 OB1：

- OB1 扫描周期可能随程序复杂度和通信负载变化。
- PID 计算依赖固定 Δt，积分项 Integral += e × Δt。
- 固定周期 OB 能保证采样周期稳定。

安全意义：

- 攻击 OB1 是改普通逻辑。
- 攻击 OB35 可能影响周期控制任务。
- 攻击 OB100 可能实现启动持久化。
- Hook Runtime 或调度器可让某些安全逻辑不运行。

---

### 2.7 I/O：PLC 如何连接真实世界

PLC 只能看到电信号，不能直接看到温度、压力、液位。I/O 模块负责把物理世界编码成数字世界，再把数字世界解码成物理动作。

DI：数字量输入，适用于按钮、限位、开关量。内部通常包括浪涌保护、滤波、光耦隔离、施密特触发、去抖处理。

DO：数字量输出，适用于继电器、接触器、指示灯、电磁阀。

AI：模拟量输入，适用于 4~20mA、0~10V 等连续量。内部包括采样电阻、滤波、放大、ADC、工程量换算。

AO：模拟量输出，适用于阀门定位器、变频器给定等。内部包括数字量、DAC、输出驱动、电流/电压转换。

高速 I/O：普通 DI/DO 不能处理编码器和高速脉冲，因此需要高速计数器 HSC、PWM、PTO 等硬件单元。

安全意义：

- 攻击可发生在传感器、信号链路、I/O 模块、ADC/DAC、PLC 数据区、执行机构。
- 工控安全不是保护 PLC 一个点，而是保护“物理量 → 信号 → 数字变量 → 控制算法 → 执行动作 → 物理过程”的完整性。

---

### 2.8 DCS 为什么存在

DCS 不是大型 PLC，而是面向连续过程工业的系统级控制架构。

PLC 更偏向 Machine 控制，DCS 更偏向 Plant 控制。PLC 常讨论 I/Q/M/DB，DCS 更关注塔顶压力、塔底温度、氧煤比、回流比、汽包液位等工艺变量。

DCS 的核心设计哲学：

- 控制责任分布：每个控制站负责一个工艺区域。
- 控制与监控分离：控制器只负责控制，不负责画面、历史和报表。
- 工程师站、操作员站、历史服务器、控制站分工明确。
- 支持在线修改、在线下载、趋势、报警、权限和历史数据。
- 高可用性和连续运行优先。

面试表达：

> PLC 和 DCS 的区别不只是离散控制和过程控制，而是设计哲学不同。PLC 以单设备或单元控制为中心，DCS 以整个工艺装置为中心，通过分布式控制站、统一数据库、操作站、工程师站、历史站和冗余机制实现连续过程工业的高可用控制。

---

### 2.9 DCS 冗余与无扰切换

DCS 冗余的目标不是简单备用，而是高可用性。主备 CPU 通常同时运行同一程序并同步关键状态，当主 CPU 故障时，备用 CPU 以无扰方式接管。

关键概念：

- Reliability：可靠性，故障概率。
- Availability：可用性，系统在线比例。
- Failover：故障切换。
- Hot Standby：热备，备用 CPU 一直运行并同步。
- Bumpless Transfer：无扰切换，切换前后控制输出连续。
- Heartbeat：心跳检测。
- Split Brain：脑裂，两个控制器都认为自己是主。
- Arbiter：仲裁机制，确保任何时刻只有一个主控制器有输出权。

同步内容：

- 程序。
- DB/实例状态。
- PID 积分和历史误差。
- 定时器、计数器。
- 报警状态。
- I/O 状态。
- 运行上下文。

安全意义：

- 攻击心跳可能导致误切换。
- 攻击同步链路可能导致备机状态落后。
- 攻击仲裁可能引发脑裂。
- 冗余机制本身也是攻击面。

# 第二册 工业通信与协议分析

---

## 第三章 工业通信原理

### 3.1 工业协议与互联网协议的设计思想差异

互联网协议面向开放网络、吞吐量、兼容性和尽力交付；工业协议面向控制网络、确定性、低时延、可靠控制和简单实现。

工业协议解决四件事：

1. 设备识别：谁和谁通信。
2. 数据表示：压力、温度、寄存器、浮点数如何编码。
3. 通信时序：谁先说、谁后说、是否允许冲突。
4. 错误恢复：CRC、异常码、超时和重发。

为什么工业协议喜欢寄存器：

- PLC 内部变量需要被远程访问。
- 协议通过地址或节点表示变量。
- Modbus 是寄存器模型，S7 是内存区/DB 模型，OPC UA 是节点模型，本质上都在访问变量。

安全意义：

- 工业协议传输的是控制变量，不是网页。
- 早期协议默认控制网络可信，因此普遍缺少认证、加密、访问控制。
- 协议攻击本质是远程修改 PLC/DCS 状态和控制变量。

---

### 3.2 Modbus 为什么能活 40 多年

Modbus 的成功不是技术先进，而是工程适配。它简单、开放、资源占用低，抽象出了统一的寄存器访问模型，使不同厂家的 PLC、仪表、变频器、远程 I/O 能够互联。

Modbus 关键认知：

- RTU 通常运行在 RS485 上，采用 Master/Slave，避免总线冲突。
- RS485 采用差分通信，抗干扰、距离远、支持多点总线。
- 40001 等地址是人类文档地址，报文中实际通常从 0 开始。
- 四类数据区本质上是数据类型和访问权限组合：Coil、Discrete Input、Input Register、Holding Register。

四种数据区：

| 类型 | 数据 | 权限 | 用途 |
|---|---|---|---|
| Coil 0xxxx | 1 bit | 可读写 | DO/线圈 |
| Discrete Input 1xxxx | 1 bit | 只读 | DI |
| Input Register 3xxxx | 16 bit | 只读 | AI/PV |
| Holding Register 4xxxx | 16 bit | 可读写 | SP/参数/AO/配方 |

常见功能码：

| 功能码 | 含义 | 安全关注 |
|---|---|---|
| 01 | 读 Coil | 读输出状态 |
| 02 | 读 Discrete Input | 读输入状态 |
| 03 | 读 Holding Register | 读参数/设定值 |
| 04 | 读 Input Register | 读过程值 |
| 05 | 写单个 Coil | 改输出 |
| 06 | 写单个 Holding Register | 改参数/SP |
| 0F | 写多个 Coil | 批量改输出 |
| 10 | 写多个 Holding Register | 批量改参数/配方 |

安全意义：

- 03/04 大量出现通常是正常读取。
- 06/10/05/0F 是写操作，更值得关注。
- 判断异常不能只看功能码，还要看来源 IP、时间、频率、目标地址和工艺语义。

---

### 3.3 Modbus TCP 抓包分析

Modbus TCP = Modbus PDU + MBAP Header，运行在 TCP 502 端口。TCP 负责连接和可靠传输，Modbus 负责控制变量读写。

典型请求：读取 40001 起始的两个 Holding Register。

```text
00 01  00 00  00 06  01  03  00 00  00 02
```

字段解释：

| 字段 | 字节 | 含义 |
|---|---|---|
| Transaction ID | 00 01 | 请求/响应配对 |
| Protocol ID | 00 00 | Modbus 固定为 0 |
| Length | 00 06 | 后续字节长度 |
| Unit ID | 01 | 网关/从站地址 |
| Function Code | 03 | 读 Holding Register |
| Start Address | 00 00 | 起始地址 0，对应 40001 |
| Quantity | 00 02 | 读 2 个寄存器 |

典型响应：

```text
00 01  00 00  00 07  01  03  04  04 D2  16 2E
```

含义：Transaction ID 对应请求，返回 4 字节数据，两个寄存器分别为 0x04D2=1234、0x162E=5678。

Wireshark 分析思路：

1. 过滤 `tcp.port == 502` 或 `modbus`。
2. 查看 TCP 三次握手是否正常。
3. 看功能码：读还是写。
4. 看地址和数量：访问哪些寄存器。
5. 看来源和目的 IP：HMI、工程师站、未知主机？
6. 看时间规律：是否出现异常高频写入。
7. 结合变量映射、HMI 画面、趋势曲线恢复寄存器语义。

面试表达：

> Wireshark 只是协议解析工具，我更关注通信行为。分析 Modbus TCP 时，我会先确认 TCP 502 通信，再解析 MBAP Header 和功能码，重点关注 06、16 等写寄存器操作。如果出现写操作，会进一步分析来源 IP、目标寄存器、写入时间和频率，并结合 PLC 变量映射、HMI 操作记录和工艺趋势判断是正常操作、误操作还是恶意行为。

# 第三册 实验与实操建议

## PID 仿真实验路线

为了把控制理论变成实际经验，建议用以下软件或工具：

1. MATLAB + Simulink：控制理论和 PID 整定首选，适合一阶、二阶、大时滞对象建模，使用 PID Tuner 观察响应速度、超调、稳态误差和稳定裕度。
2. Scilab + Xcos：免费替代方案，可做基础控制系统仿真。
3. Factory I/O：建立工业现场直觉，如水箱、传送带、传感器、执行器，与 PLC 仿真联动。
4. CODESYS：可模拟 PLC，支持 ST、LD、FBD、SFC，可编写 PID、任务和 Modbus 通信。
5. OpenPLC：开源 PLC Runtime，适合工控安全实验、Python Modbus 读写、Wireshark 抓包。
6. Siemens TIA Portal + PLCSIM：学习西门子生态、S7 通信、OB/FB/DB/PID 组织方式。

建议实验：

- 实验 1：只调 Kp，观察上升时间、稳态误差和超调。
- 实验 2：只调 Ki，观察稳态误差消除和积分饱和。
- 实验 3：只调 Kd，观察阻尼增加和噪声放大。
- 实验 4：PI 控制一阶液位对象。
- 实验 5：PID 控制二阶欠阻尼对象。
- 实验 6：大时滞对象 PID 整定，观察振荡。
- 实验 7：在 OpenPLC 中建立 Modbus Holding Register，把 SP/Kp/Ki/Kd 映射出来。
- 实验 8：用 Python 修改 SP，Wireshark 抓包，观察 PV 和 MV 曲线。
- 实验 9：模拟反馈欺骗，给 PLC 假 PV，观察控制器错误动作。
- 实验 10：模拟写 PID 参数攻击，观察系统从稳定到振荡。

面试可描述经验：

> 我用仿真方式建立过一阶和二阶控制对象，观察过 Kp、Ki、Kd 对响应速度、超调、振荡和稳态误差的影响；同时计划/已经结合 OpenPLC 或 CODESYS 将 PID 参数映射到 Modbus 寄存器，通过 Python 和 Wireshark 分析写寄存器对控制回路的影响。这让我能够从控制理论、协议行为和安全风险三个维度理解工控系统。
