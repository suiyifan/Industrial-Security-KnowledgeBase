# 安全评估方法论

## 1. 工控安全评估目标

安全评估不是为了扫出一堆漏洞，而是识别真实风险并给出可落地改进建议。

评估目标：

- 明确资产和拓扑。
- 找出关键攻击路径。
- 发现高风险配置和漏洞。
- 评估防护和监测能力。
- 给出分阶段整改方案。

## 2. 评估流程

1. 范围确认：系统、区域、时间、授权边界。
2. 资料收集：拓扑、资产、账号、策略、制度。
3. 被动发现：流量镜像、日志、资产识别。
4. 低风险验证：版本、配置、弱口令、暴露面。
5. 风险分析：攻击路径和工控影响。
6. 报告输出：问题、证据、风险、建议。
7. 复测：确认整改效果。

## 3. 工控评估注意事项

- 避免高强度主动扫描。
- 避免对 PLC 做未授权写入测试。
- 避免在生产时段进行可能影响服务的验证。
- 所有测试必须授权并留痕。
- 优先被动识别和低风险验证。
- 关键操作前与现场工程师确认。

## 4. 检查项

| 方向 | 检查内容 |
| --- | --- |
| 资产 | 清单、版本、责任人、重要性 |
| 网络 | 分区、ACL、远程维护、暴露端口 |
| 主机 | 补丁、杀软、白名单、日志 |
| 账号 | 默认口令、共享账号、权限 |
| 工程 | 项目文件、下载权限、备份 |
| 协议 | 明文、越权、异常命令 |
| 设备 | 固件、运行模式、硬件钥匙 |
| 监测 | IDS、日志、告警处置 |
| 恢复 | 备份、演练、应急流程 |

## 5. 报告写法

每个问题写：

- 问题描述。
- 影响资产。
- 证据。
- 风险分析。
- 工控影响。
- 整改建议。
- 优先级。

## 6. 面试表达模板

> 工控安全评估我会先确认授权范围和生产约束，然后以资产、拓扑和数据流为基础，优先采用被动识别和低风险验证。评估重点不是单纯扫漏洞，而是找攻击路径，比如远程维护到工程站、工程站到 PLC、网关到控制网。报告中我会把技术问题映射到工控影响，并给出可分阶段落地的整改建议。
