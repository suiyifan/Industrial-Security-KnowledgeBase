# 密码学、PKI 与身份基础

工业系统需要确认“数据是否被修改、通信对象是谁、命令是否来自授权主体、固件是否可信”。密码学提供构建这些属性的工具，PKI 管理公钥身份，身份与访问管理再把人、设备、服务和权限连接起来。算法名称本身不能保证安全，密钥、随机数、实现、配置和生命周期同样重要。

## 学习目标

- 区分编码、哈希、MAC、加密、签名和密钥协商；
- 理解随机数、Nonce、Salt、KDF 和口令存储；
- 理解对称/非对称密码与 AEAD 的使用边界；
- 掌握证书、信任锚、证书链、撤销和私钥生命周期；
- 区分身份、认证、授权、会话和审计；
- 理解 MFA、Kerberos、服务账户和机器身份；
- 评估 OPC UA、VPN、远程运维和固件签名中的信任链；
- 将密码状态映射到工程操作和物理过程。

## 1. 密码学解决什么问题

密码学常用于提供：

- 机密性：未授权方无法理解数据；
- 完整性：数据变化能够被检测；
- 身份认证：确认通信主体或消息来源；
- 不可否认性的技术基础：签名可支持来源证明，但仍依赖密钥控制和流程证据；
- 新鲜性：结合随机数、序列号或时间抵御重放。

密码学不能自动解决对象授权、业务逻辑、终端失陷和物理安全。拥有有效证书的服务仍可能有漏洞，经过正确签名的危险配置仍可能破坏过程。

## 2. 编码不是加密

Base64、十六进制、URL 编码和字符编码用于表示或传输数据，任何人都可逆转换，不提供机密性。

```text
编码：改变表示形式
哈希：生成定长摘要，通常不可逆
加密：持有密钥者可解密
签名：私钥签名、公钥验证来源与完整性
```

把 Base64 口令称为“加密存储”是常见错误。安全评估必须识别真正使用的算法、密钥和模式。

## 3. 随机数、熵、Nonce 与 Salt

密钥、令牌、会话标识和挑战值依赖不可预测随机数。通用伪随机函数如果种子可预测，就不适合生成安全秘密。应使用操作系统提供的密码学安全随机源。

Nonce 是“一次使用的值”，常用于防止同一密钥下重复状态；它不一定需要保密，但必须满足算法要求的唯一性或随机性。某些 AEAD 模式重复 Nonce 会严重破坏安全。

Salt 是口令派生时与口令一起处理的随机值，目的是让相同口令产生不同结果并抵抗预计算。Salt 通常不需保密，但应为每个口令独立生成。

## 4. 哈希函数

密码学哈希把任意长度输入映射为固定长度摘要。理想性质包括原像抗性、第二原像抗性和碰撞抗性。

典型用途：

- 文件和固件完整性比对；
- 数字签名中的消息摘要；
- 内容寻址和去重；
- 取证证据校验；
- 作为 MAC/KDF 等构造的组件。

```bash
sha256sum firmware.bin
```

哈希相同能在算法和采集可信前提下支持内容一致；哈希不同证明内容不同。哈希本身不证明文件来自厂商，因为攻击者可以替换文件并重新计算摘要。来源验证需要签名或独立可信渠道。

### 4.1 口令不能只做快速哈希

普通快速哈希便于攻击者高速猜测口令。口令应使用专用、可配置成本的口令哈希/KDF，并保存独立 Salt。系统应同时使用速率限制、多因素、泄露监测和安全重置流程。

## 5. MAC 与消息完整性

消息认证码（MAC）使用共享秘密生成认证标签，验证方持有同一秘密。HMAC 是基于哈希的标准构造。

```text
tag = MAC(shared_key, message)
```

MAC 提供消息完整性和共享密钥持有者认证，但无法区分多个共享同一密钥的主体，也不提供公开可验证签名意义。把同一密钥复制到整批设备会扩大泄露范围并降低可归责性。

CRC、校验和用于检测传输错误，不具备对抗主动攻击者的密码学完整性，不能替代 MAC。

## 6. 对称加密

对称加密使用同一秘密密钥完成加密和解密，速度快，适合大量数据。分组密码需要工作模式；现代系统优先使用经过验证的认证加密（AEAD），同时保护机密性和完整性。

AEAD 输入常包含：密钥、Nonce、明文和附加认证数据（AAD）。AAD 不加密但受完整性保护，适合放协议版本、序号或头部。

风险包括：

- 使用不提供完整性保护的模式；
- Nonce/IV 重复或可预测性不符合要求；
- 解密后在认证成功前处理明文；
- 密钥长期不轮换或多用途复用；
- 错误信息泄露认证差异；
- 自行设计密码协议。

应使用成熟库和协议，不自行拼装“加密 + 哈希”方案。

## 7. 非对称密码与密钥对

非对称密码使用公钥和私钥。公钥可以公开，私钥必须受保护。它常用于数字签名、密钥协商和少量数据封装，而非直接加密大量业务数据。

安全性依赖算法、参数、随机数、实现和私钥保护。公钥本身不带可信身份，必须通过证书、预配置指纹或其他可信渠道绑定主体。

## 8. 数字签名

签名者用私钥对消息摘要相关结构签名，验证者用公钥验证。签名可证明内容在签名后未变且签名私钥参与，但不能单独证明：

- 签名人亲自执行了操作；
- 私钥从未泄露或被自动服务调用；
- 被签内容安全、正确或适合当前设备；
- 时间戳一定可信。

固件签名验证必须覆盖完整镜像、版本和目标设备，并结合防回滚机制。否则攻击者可能安装签名有效但存在旧漏洞的版本。

## 9. 密钥协商与前向安全

密钥协商让双方在不直接传输会话密钥的情况下得到共享秘密。临时密钥协商可提供前向安全：长期私钥日后泄露时，过去会话不一定能够被解密。

密钥协商必须同时认证对端，否则可能遭遇中间人。TLS、SSH 和 VPN 把协商、身份、完整性和会话管理组合成完整协议，不能只抽取其中一个算法自行实现。

## 10. KDF 与密钥分离

密钥派生函数（KDF）从共享秘密、口令或主密钥派生一个或多个子密钥。不同用途应使用上下文标签派生独立密钥，避免同一原始密钥同时用于加密、MAC、设备认证和备份。

```text
master secret + context
  -> KDF -> encryption key
         -> integrity key
         -> session/export key
```

密钥分离可降低一种用途泄露对其他用途的影响。

## 11. PKI 的组成

公钥基础设施（PKI）通常包含：

- 根 CA：信任锚，通常高保护、低频使用；
- 中间/签发 CA：签发终端证书；
- 注册/审批流程：验证主体身份和申请权限；
- 证书模板与策略：定义用途、期限和约束；
- 证书库、撤销服务和审计；
- 私钥生成、存储、轮换、恢复与销毁流程。

```text
根 CA
  -> 中间 CA
      -> 工程站证书
      -> OPC UA 服务器证书
      -> VPN 用户/设备证书
```

根 CA 不应直接在线签发所有设备证书。分层可隔离风险和便于撤销中间 CA。

## 12. X.509 证书字段

证书包含主体、公钥、签发者、序列号、有效期、签名算法和扩展。重点扩展：

- Subject Alternative Name：DNS、IP、URI 等身份；
- Basic Constraints：是否为 CA、路径限制；
- Key Usage：签名、密钥协商、证书签发等；
- Extended Key Usage：服务器、客户端、代码签名等用途；
- Authority/Subject Key Identifier：辅助链构建。

现代主机名验证主要依赖 SAN，而不是只看 Common Name。证书“在有效期内”只是检查之一。

```bash
openssl x509 -in certificate.pem -noout -text
openssl verify -CAfile ca-chain.pem certificate.pem
```

命令验证结果依赖给定信任库、时间和策略，不能代表所有客户端行为一致。

## 13. 证书申请、签发与部署

典型流程：

```text
设备/服务生成密钥对 -> 生成 CSR -> 身份与权限审批
-> CA 签发证书 -> 安全部署证书/链 -> 应用启用
-> 监测到期/异常 -> 轮换或撤销 -> 安全销毁旧密钥
```

优先在密钥最终使用位置生成私钥，避免私钥通过邮件或共享目录传输。自动注册应绑定受信设备身份和最小模板权限。

工业设备常难以更换证书，项目设计阶段就应确认支持的算法、链长度、证书容量、时钟、更新接口和停机影响。

## 14. 信任链与信任库

客户端从终端证书向上构建到本地信任锚，并检查签名、约束、用途、名称和时间。服务器发送一个证书文件不等于链完整；客户端信任根也不意味着信任所有用途。

在工控环境中，厂商自签证书、企业 CA 和产品私有 CA 可能并存。把未知根证书大范围加入信任库会扩大整个环境的签发权。信任库变更应审计、审批和基线化。

## 15. 撤销、过期与时钟

证书过期用于限制长期暴露，撤销用于私钥泄露、身份失效或错误签发。CRL 与 OCSP 是常见撤销机制，但离线控制网可能无法实时访问。

系统必须明确撤销不可达时是失败关闭还是继续运行。过度严格可能中断生产，失败开放则延长风险。可采用本地 CRL 分发、短期证书、维护窗口预更新和失效监测。

错误系统时间会导致有效证书被判过期或未来生效，也会破坏 Kerberos 和日志关联，因此时间同步属于身份基础设施的一部分。

## 16. 私钥、HSM 与 TPM

私钥应限制导出、访问和用途。可使用系统密钥库、TPM、智能卡或 HSM 提高保护。

- TPM：绑定设备、测量启动、保护本机密钥等；
- HSM：集中高保证密钥生成、使用和审计；
- 智能卡/安全令牌：持有用户密钥并支持强认证。

硬件保护不能阻止已授权应用滥用签名接口。还需操作审批、速率限制、用途约束和审计。

备份私钥会提高可恢复性，也扩大暴露面。签名密钥与解密密钥的备份策略应区分；某些签名私钥不应托管恢复。

## 17. 身份、认证、授权与审计

```text
身份标识：我声称是谁
认证：证明该身份
授权：该身份能做什么
会话：认证后持续交互状态
审计：记录谁在何时做了什么
```

认证成功不能替代授权。一个经过证书认证的工程师可能有权查看趋势，但无权下载 PLC 程序。授权应作用到站点、资产、对象和操作。

### 17.1 人、设备和服务身份

- 人员身份：与个人生命周期、MFA 和职责关联；
- 设备身份：绑定硬件/设备证书、库存和所有者；
- 服务身份：用于应用到应用访问，禁止多人共用；
- 工作负载身份：容器/任务的短期身份。

共享“admin”账户破坏可归责性。临时维护应使用个人身份获取限时角色，而非共享永久密码。

## 18. 密码认证与 MFA

密码安全依赖长度、唯一性、安全存储、速率限制、泄露检测和重置流程。复杂度规则若导致可预测模式，效果有限。

MFA 组合不同因素：所知、所有、所是。两个密码不是两个因素。抗钓鱼的密码学认证优于仅依赖可转发的一次性验证码，但选型需考虑离线现场、手套操作、应急账户和设备兼容性。

应急“break-glass”账户必须高保护、双人控制、使用告警、定期测试和事后复盘，不能成为日常绕过路径。

## 19. Kerberos 与域身份原理

Kerberos 使用可信 KDC 和票据，避免每次向服务传递用户密码。简化流程：

```text
用户认证 -> 获取 TGT
TGT -> 请求服务票据
服务票据 -> 访问目标服务
```

它高度依赖时间、服务主体名称、密钥和域信任。服务账户密码长期不轮换、委派配置过宽和票据权限过大都会扩大横向移动风险。

Kerberos 提供认证，不自动完成应用对象授权。工控域应分层管理账户，避免办公域高权限自然继承控制系统权限。

## 20. 机器到机器身份与秘密

应用连接数据库、消息总线或 API 时需要机器身份。优先使用短期证书、工作负载身份或受控秘密管理，避免在代码、镜像、注册表和配置库中保存长期明文秘密。

秘密生命周期包括生成、分发、使用、轮换、撤销和销毁。轮换必须考虑双方过渡窗口，避免“一改密码，采集全断”。可支持双密钥/双证书短期重叠并监测旧凭据停止使用。

## 21. 工业协议与密码身份

### 21.1 OPC UA

OPC UA 可使用应用实例证书建立安全通道，并在会话层认证用户。应用证书信任和用户授权是不同层。首次自动信任任何证书会让身份验证失效。

证书 Subject/SAN、Application URI、信任列表、拒绝列表、Endpoint 安全模式和用户权限需一致管理。证书更新不能破坏客户端信任链和生产连接。

### 21.2 VPN 与远程运维

VPN 可用证书认证用户或设备，但证书有效不代表获准进入所有控制区。还需 MFA、目标白名单、堡垒机、限时角色和会话审计。

### 21.3 固件与工程文件签名

固件签名应验证来源、完整性、设备型号、版本和防回滚。工程文件签名还应关联审批人、工具版本和目标控制器。签名正确不替代仿真、联锁审查和现场验证。

## 22. 密码敏捷性与遗留设备

工业设备生命周期长，算法和证书可能在设备退役前过时。密码敏捷性要求能够在不中断业务的情况下更新算法、密钥长度、证书链和信任锚。

资产清单应记录：协议版本、算法、证书到期、私钥位置、更新方法和依赖客户端。无法升级的设备应通过分区、网关、受控访问和监测补偿，并制定替换计划。

## 23. 数据流、控制流与状态流

### 数据流

明文经过加密和认证形成受保护数据，证书与信任库提供身份元数据，日志记录握手和授权结果。密钥不应沿普通配置和日志数据流泄露。

### 控制流

信任锚决定接受哪些证书，认证建立主体，授权策略决定允许操作，签名/审批服务决定配置能否发布。

### 状态流

```text
证书有效 -> 安全通道建立 -> 用户/设备认证 -> 对象授权
-> 命令被接受 -> PLC 状态变化 -> 现场反馈
```

每一步独立。TLS 握手成功不能证明用户有写权限，签名验证成功不能证明配置适合当前工艺。

## 24. 工控场景案例：共享设备证书

一批工业网关使用同一客户端证书和私钥访问数据平台。任一网关被攻破后，攻击者可复制私钥并模拟其他网关，平台日志只能看到同一证书主体，无法区分设备。

整改包括每设备唯一密钥、在设备或安全硬件中生成且限制导出、证书绑定资产编号与权限、泄露设备单独撤销、自动轮换和异常并发使用检测。数据平台还需校验设备可写的站点和点位范围。

## 25. 安全风险与攻击链

```text
私钥/服务秘密泄露 -> 冒充设备或服务 -> 建立合法加密通道
-> 利用过宽授权读取点表或提交配置 -> 发布链消费
-> PLC/SCADA 接收 -> 控制逻辑、联锁和反馈决定物理后果
```

加密通道会让攻击流量外观看起来“合法”。因此监测不能只看 TLS 是否成功，还要看证书身份、来源位置、使用时间、授权对象和业务行为。

## 26. 防护原则

- 使用成熟协议、库和经过审查的算法组合；
- 密钥由密码学安全随机源生成，按用途分离；
- 私钥限制导出，采用 TPM/HSM/系统密钥库；
- 人、设备和服务使用唯一身份，禁止共享私钥与账户；
- PKI 分层，模板最小化，签发、撤销和信任库变更受审计；
- 证书到期和撤销提前监测，演练不停产轮换；
- MFA、最小权限、对象级授权和短期会话共同使用；
- 签名验证包含版本、目标和防回滚；
- 日志记录身份与授权，但不记录秘密；
- 对遗留设备部署补偿控制和替换计划。

## 27. 隔离实验

### 实验一：哈希、MAC 与签名

对虚构配置文件分别计算哈希、HMAC 和数字签名，修改一个字节后比较验证结果，并解释三者对来源证明的差异。

### 实验二：本地 PKI

建立隔离测试根 CA 和中间 CA，为测试 Web/OPC UA 服务签发证书，验证名称、用途、链、过期和撤销状态。私钥不离开实验环境。

### 实验三：证书轮换

模拟旧新证书重叠、信任列表更新、客户端切换和旧证书撤销，记录如何避免服务中断和如何检测仍使用旧身份的客户端。

### 实验四：身份与对象授权

建立人员、设备和服务三类测试身份，分别授予只读趋势、写入测试点和审批权限，证明认证成功不等于所有对象可访问。

## 28. 面试表达

### 60 分回答

哈希用于内容摘要，MAC 用共享密钥验证完整性和来源，数字签名用私钥签名、公钥验证，对称加密适合大量数据，非对称机制常用于身份和密钥协商。PKI 用 CA、证书、信任库和撤销把公钥绑定身份。认证成功后仍要做对象级授权，TLS 也不能修复业务越权。

### 90 分回答

我会把密码学放进完整生命周期看：随机源生成密钥，私钥在 TPM/HSM 或受控密钥库中使用，证书通过分层 CA 和模板签发，客户端验证链、名称、用途、时间与撤销，认证后再按站点和操作授权。工业环境还要设计离线撤销、到期预警和不停产轮换。OPC UA 应区分应用证书安全通道与用户会话授权；固件签名还要绑定型号、版本和防回滚。即使攻击者用泄露私钥建立合法 TLS，会话行为和对象范围仍应被监测。

### 常见追问

**哈希能否证明文件来自厂商？** 不能。哈希只支持内容比对，来源需要可信签名或独立发布渠道。

**证书有效是否代表用户有控制权限？** 不代表。证书完成身份绑定，授权策略决定具体对象和操作。

**自签名证书是否一定不安全？** 不一定。关键是是否通过可信渠道固定其指纹并管理生命周期；盲目自动信任才会失去身份验证。

**加密后 IDS 是否无法检测？** 载荷可见性降低，但仍可分析证书、端点、时序、流量和业务审计，也可在受控终止点获取应用日志。

## 本章总结

密码学提供机密性、完整性和身份工具，PKI 与身份系统把工具变成可运营的信任体系。真正的安全取决于随机数、密钥保护、证书验证、唯一身份、最小授权、轮换撤销和业务审计。工控环境还必须考虑长生命周期、离线网络和连续生产，并始终验证密码状态之后的控制与现场状态。
