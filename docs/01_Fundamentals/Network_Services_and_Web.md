# 网络服务、Web 与远程管理基础

TCP/IP 解决数据如何到达另一台主机，网络服务则定义名称如何解析、地址如何配置、时间如何同步、用户如何认证以及管理命令如何执行。工控系统中的工程站、历史服务器、工业网关和远程运维平台都依赖这些服务。成熟工程师必须区分网络可达、服务可用、身份通过、对象授权和业务执行五种状态。

## 学习目标

- 理解 DNS、DHCP、NTP 的数据流、信任边界和故障表现；
- 掌握 HTTP 请求/响应、Cookie、Session、代理与 API 基础；
- 理解 TLS 握手、证书验证和 HTTPS 的安全边界；
- 理解 SSH、SMB、RDP、WinRM、WMI/RPC 的管理模型；
- 设计受控的供应商远程运维路径；
- 从日志、抓包和主机状态验证服务行为；
- 将远程管理操作映射到工程变更和物理过程。

## 1. 网络服务的分层原理

一次远程访问通常依赖多层服务：

```text
名称解析 -> 地址/路由 -> TCP/UDP -> TLS/SSH 等安全通道
         -> 用户认证 -> 会话建立 -> 对象授权 -> 业务操作
```

任一层成功都不能证明下一层成功。DNS 返回地址不代表 TCP 可达；TCP 握手成功不代表证书可信；登录成功不代表用户有权修改 PLC 项目；Web 页面返回 200 也不代表业务操作完成。

排障与安全分析应逐层确认输入、状态、日志和响应，避免把所有失败统称为“网络问题”。

## 2. DNS：名称到资源的映射

DNS 是分布式层次数据库。客户端通常向递归解析器查询，解析器通过缓存或向权威服务器获取记录。

```text
应用 -> Stub Resolver -> 递归 DNS -> 根/顶级域/权威 DNS
                    <- 缓存或最终答案 <-
```

常见记录：

| 类型 | 用途 | 安全关注 |
| --- | --- | --- |
| A/AAAA | 名称到 IPv4/IPv6 | 错误地址、双栈路径差异 |
| CNAME | 别名 | 依赖链和接管风险 |
| PTR | 地址到名称 | 仅作线索，不能证明身份 |
| MX | 邮件服务 | 工控区通常不应依赖外部邮件 |
| SRV | 服务发现 | AD、Kerberos 等高度依赖 |
| TXT | 文本策略/验证 | 内容与用途需结合上下文 |

### 2.1 缓存、TTL 与负缓存

TTL 控制缓存时间，但解析器可能受策略影响。修改记录后不同客户端短时间看到不同结果并不罕见。NXDOMAIN 等失败也可能被缓存。

排查要确认应用实际解析链：本地 hosts、NSS/Windows DNS Client、缓存、搜索域、VPN 和代理可能改变结果。

```bash
dig historian.example.local A
dig +trace example.com
```

```powershell
Resolve-DnsName historian.example.local
Get-DnsClientCache
```

### 2.2 DNS 安全风险与防护

风险包括未授权记录修改、缓存投毒、错误转发、区域传送暴露、恶意域名和 DNS 隧道。工控区应使用明确的内部解析器、限制递归和区域传送、保护管理权限、记录查询日志并监测新域名与异常外部解析。

DNSSEC 验证数据来源完整性，不能加密查询内容，也不能保证解析到的应用本身安全。DoT/DoH 加密传输，但若绕过企业解析器也可能削弱监测和策略。

## 3. DHCP：动态地址与网络参数

经典 DHCPv4 过程常概括为 DORA：Discover、Offer、Request、Acknowledge。服务器不仅分配地址，还可能提供网关、DNS、租期和其他选项。

```text
客户端广播 Discover -> 服务器 Offer
客户端 Request       -> 服务器 ACK
```

工业控制器常使用静态地址或保留租约，但工程笔记本、无线终端和管理设备可能依赖 DHCP。恶意或错误 DHCP 服务器可让客户端获得错误网关/DNS，地址冲突也会造成间歇通信。

防护包括端口信任、DHCP Snooping、地址规划、保留租约、日志与异常 Offer 监测。不能在生产网随意启动测试 DHCP 服务。

## 4. NTP/PTP 与时间可信度

准确时间支撑报警、SOE、身份认证、证书验证、日志关联和批次追溯。NTP 通过层级时间源同步通用系统，PTP 可在需要时提供更高精度。

时间问题包括：偏移、漂移、时区错误、闰秒处理、错误时间源和突然跳时。墙上时间被调整后，日志可能倒序；持续时间分析应使用单调时钟或专用指标。

```bash
timedatectl
chronyc tracking
chronyc sources -v
```

```powershell
w32tm /query /status
w32tm /query /source
```

工控网络应定义权威时间源、分层同步、允许的偏差与失步告警。安全设备、SCADA、PLC 和历史库时间不同步会使攻击链分析失真。

## 5. HTTP 请求与响应模型

HTTP 是无状态的请求/响应协议。典型请求包括方法、目标、版本、头部和可选消息体：

```http
GET /api/alarms?site=1 HTTP/1.1
Host: historian.example.local
Accept: application/json
```

响应包含状态码、头部和内容。方法表达意图，但服务器是否遵循语义取决于实现。`GET` 不应改变状态，`PUT` 常用于替换，`PATCH` 常用于部分更新，`DELETE` 表达删除；不能只凭方法名判断实际影响。

### 5.1 状态码不是业务结果

- `2xx`：HTTP 层处理成功，不保证业务目标正确；
- `3xx`：重定向，需检查目标和缓存；
- `4xx`：客户端请求、认证或授权问题；
- `5xx`：服务器处理失败或上游依赖异常。

一个配置发布接口返回 200，可能只表示任务已入队。最终还需查看任务状态、发布日志、PLC 响应和现场反馈。

### 5.2 头部、内容类型与缓存

`Host`、`Content-Type`、`Authorization`、`Cookie`、`Origin`、`Cache-Control` 等影响路由、解析、认证和浏览器行为。后端不能只相信客户端声明的内容类型，仍需验证实际结构和业务字段。

敏感报警、资产和配置响应应限制缓存。反向代理和浏览器缓存可能让注销后的数据仍可见。

## 6. Cookie、Session、Token 与会话状态

HTTP 无状态，应用通常用 Cookie 中的会话标识或 Token 关联服务端/自包含状态。

```text
登录凭据 -> 身份验证 -> 创建会话/签发 Token
请求携带标识 -> 校验有效性 -> 加载用户与权限 -> 执行业务
```

安全会话需要：足够随机的标识、TLS 保护、合理过期、注销失效、权限变化后的重评估、防固定和防重放。Cookie 常用属性包括 `Secure`、`HttpOnly`、`SameSite`。

JWT 等自包含 Token 不是天然更安全。必须验证算法、签名、签发者、受众、时间和密钥，并处理撤销与轮换。Token 中不应放置不必要敏感数据。

### 6.1 认证与授权

认证回答“你是谁”，授权回答“你能对哪个对象做什么”。工控平台还需对象和站点级授权：有权查看 A 厂站不代表可查看 B 厂站，有权确认报警不代表可改配方。

## 7. TLS 与 HTTPS

TLS 提供传输机密性、完整性和对端身份验证。简化握手：

```text
ClientHello（版本、套件、随机数、扩展）
ServerHello + 证书 + 密钥协商信息
客户端验证证书并完成密钥协商
双方使用会话密钥保护应用数据
```

现代 TLS 通常通过非对称机制完成身份与密钥协商，再用对称加密传输大量数据。具体细节取决于协议版本和套件。

### 7.1 证书验证

客户端应验证：

- 证书链是否到达可信根；
- 主机名/IP 是否匹配；
- 有效期是否成立；
- Key Usage/Extended Key Usage 是否适用；
- 签名算法和密钥是否满足策略；
- 撤销和企业策略是否要求检查。

“页面能打开”不代表验证正确，客户端可能被配置为忽略错误。工业设备常见自签名证书、共享证书、过期证书和私钥复制。应建立资产证书清单、唯一身份、轮换和失效告警。

### 7.2 TLS 的边界

TLS 不能修复 SQL 注入、越权和危险配置发布；代理终止 TLS 后内部链路仍需保护；终端失陷后攻击者可在加密前后读取数据。HTTPS 只说明传输通道，不证明业务可信。

## 8. 正向代理、反向代理与负载均衡

正向代理代表客户端访问外部资源，反向代理代表后端服务接受客户端请求。反向代理可终止 TLS、做认证、限速、路由和日志。

```text
客户端 -> 反向代理/WAF -> Web/API -> 数据库/发布服务
```

代理会改变源地址和协议层。应用若信任 `X-Forwarded-For` 等头部，必须只接受来自受控代理的值。负载均衡下会话状态、健康检查和日志需跨节点关联。

工业 DMZ 可使用代理隔离外部访问与内部服务，但不能让代理成为绕过身份和对象授权的万能入口。

## 9. REST、API 与机器身份

REST 是架构风格，不是安全协议。API 设计应明确资源、方法、状态、幂等性、版本和错误语义。

机器到机器访问应使用独立服务身份、最小权限、短期凭据或证书，避免共享管理员账号。API Key 只是秘密字符串，必须通过 TLS 传输、限制范围、轮换并审计。

对配置和控制类 API，还需：

- 参数类型、范围和变化率校验；
- 幂等键或事务标识；
- 双人审批与维护窗口；
- 可追溯版本和回滚；
- 最终设备状态验证。

## 10. SSH：加密远程 Shell 与隧道

SSH 提供服务端身份验证、用户认证、加密会话和可选端口转发。认证方式包括密码、公钥、证书和多因素组合。

```bash
ssh -G user@host
ssh-keygen -lf /path/to/host_key.pub
```

`known_hosts` 绑定主机名与主机密钥。首次连接盲目接受密钥会失去抵御中间人的能力；主机重装或密钥轮换应通过独立渠道确认。

### 10.1 密钥管理

用户私钥应受权限和口令保护，使用 agent 时控制转发。公钥授权文件要限制来源、命令和转发能力。共享私钥会让责任无法区分。

### 10.2 端口转发风险

本地、远程和动态转发可穿越网络边界。运维中很有用，也可能绕过防火墙和监测。工控区应按角色限制转发，记录会话并在任务结束后撤销通道。

## 11. SMB：文件共享与远程管理

SMB 支持文件、打印、命名管道和部分远程管理。现代环境应禁用不必要旧版本并启用适当签名/加密策略。

共享权限与 NTFS ACL 共同决定网络访问。隐藏共享和管理共享不是安全边界。工程项目通过共享目录流转时，应控制写权限、版本、签名、锁定和离线备份。

```powershell
Get-SmbShare
Get-SmbSession
Get-SmbOpenFile
Get-SmbServerConfiguration
```

SMB 横向移动风险常来自共享管理员凭据、过宽共享、NTLM 使用和远程服务管理。工控区应限制跨区 SMB，只允许明确源、目的和用途。

## 12. RDP：交互式远程桌面

RDP 提供完整图形会话，适合需要厂商工程软件的远程运维，但权限和影响面大。

关键控制包括：

- 通过 VPN/堡垒机进入，不直接暴露；
- 使用 NLA、强认证和独立账户；
- 限制剪贴板、驱动器、打印机和设备重定向；
- 审批、限时开放、会话录屏/审计；
- 禁止多人共用账户；
- 会话结束后确认注销和权限撤销。

“断开连接”可能保留会话和程序运行，不等同于注销。工程软件在断开后继续保持 PLC 在线连接，可能影响维护安全。

## 13. WinRM、PowerShell Remoting、WMI 与 RPC

WinRM 是 Windows 远程管理基础，PowerShell Remoting 可通过它建立远程会话。域环境常用 Kerberos，非域或跨边界配置需谨慎处理 HTTPS、TrustedHosts 和凭据委派。

```powershell
Get-Service WinRM
winrm get winrm/config
Get-PSSessionConfiguration
```

Just Enough Administration（JEA）可限制远程端点可用命令与身份，但配置和脚本本身需保护。

WMI/CIM 可查询和管理进程、服务、设备等。RPC/DCOM 还支撑大量 Windows 功能，端口和权限复杂。防火墙开放应基于具体管理需求，而不是粗放允许整个网段。

## 14. VPN、堡垒机与供应商远程运维

成熟远程运维路径：

```text
供应商终端 -> 强认证 VPN -> 工业 DMZ 堡垒机
           -> 审批会话 -> 指定工程站/设备
           -> 操作审计 -> 任务结束撤销
```

控制要点：

- 供应商终端安全基线和设备身份；
- 每人独立账户，禁止共享；
- 工单、维护窗口、目标白名单和最短时限；
- 远程会话录屏、命令/文件传输审计；
- 禁止直接分流和非必要本地资源映射；
- 敏感文件经受控中转、查杀、哈希和审批；
- 任务结束注销、断开 VPN、撤销临时权限；
- 重大操作由现场人员确认并验证过程状态。

远程运维不是简单“开一个 VPN”。VPN 只建立网络通道，仍需目标级授权、主机隔离和工程操作控制。

## 15. 服务发现与身份边界

mDNS、LLMNR、NetBIOS 名称服务和厂商发现协议可简化局域网设备发现，也可能泄露资产或被伪造响应。控制网应根据需要关闭或限制非必要发现，不能让临时工程笔记本自动信任名称解析结果。

服务身份最终应由证书、主机密钥、Kerberos 或其他可信凭据建立，而非只凭 IP、DNS 名或 MAC 地址。

## 16. 日志与跨层诊断

服务调查至少关联：

- DNS 查询与响应；
- TCP/UDP 会话和 TLS 握手；
- 代理、VPN 和防火墙日志；
- SSH/RDP/WinRM 认证与会话；
- Web/API 访问、应用审计和数据库记录；
- 工程软件操作、PLC 事件和现场状态。

```text
用户身份 -> VPN 会话 -> 堡垒机会话 -> 目标主机登录
        -> 工程软件进程 -> 工业协议操作 -> PLC 状态
```

时间必须统一。NAT、代理和跳板会改变源地址，需用会话 ID、账户、证书、时间和目标关联，而非只靠 IP。

## 17. 数据流、控制流与状态流

### 数据流

DNS 提供目标，DHCP/NTP提供网络与时间参数，TLS/SSH保护传输，应用再把账号、配置和工业数据交给后端或控制设备。

### 控制流

认证决定主体，会话承载请求，授权决定可操作对象，审批和发布服务再决定命令是否进入控制链。任何代理或远程入口都不应跳过对象授权。

### 状态流

```text
名称解析成功 -> 网络可达 -> 安全通道建立 -> 身份认证
-> 对象授权 -> 操作提交 -> 任务执行 -> PLC/现场反馈
```

每一步都应独立记录。远程桌面登录不代表执行了工程操作，API 返回成功也不代表现场动作完成。

## 18. 工控场景案例：供应商远程维护

某供应商通过 VPN 和共享账户连接工程站，使用 RDP 修改网关配置。VPN 日志能看到供应商出口地址，但共享账户无法区分具体人员；RDP 只记录登录，文件从本地磁盘重定向进入工程站，未经过受控中转。配置变更导致采集点映射错误，SCADA 趋势异常但 PLC 本地闭环保持正常。

整改不只是更换密码，而是建立个人身份、多因素认证、堡垒机、文件中转与哈希、维护审批、配置版本、操作录屏和变更后点表/趋势验证。共享账户和直接设备重定向应被取消。

## 19. 安全风险与攻击链

```text
供应商凭据失陷 -> VPN 进入工业 DMZ -> 利用过宽目标权限
-> RDP/SSH 登录工程站或网关 -> 修改项目/代理/配置
-> 工业协议或发布服务向 PLC 传播 -> 控制逻辑和联锁响应
```

调查必须验证凭据使用、VPN、跳板、目标会话、文件传输、进程、配置差异、协议操作和 PLC 状态。不能因为 VPN 登录成功就声称控制器被操纵。

## 20. 防护原则

- 服务最小化，只开放有明确业务负责人的端口和方向；
- 名称、地址、时间和证书服务采用受控权威源；
- 管理协议与业务数据路径分离；
- TLS/SSH 正确验证身份，不关闭证书或主机密钥检查；
- 每人独立账户、最小权限、多因素和短期授权；
- 所有远程访问经过 DMZ、堡垒机、审批和审计；
- 文件传输受控、查验、哈希并与工单关联；
- 配置和控制操作具备版本、双人确认和回滚；
- 用多源日志验证最终系统与过程状态。

## 21. 隔离实验

### 实验一：DNS 与 HTTP

在虚拟网络建立本地 DNS 与测试 Web 服务，观察正常解析、缓存、错误记录、TCP、HTTP 请求和代理日志。记录“名称成功但服务失败”等分层状态。

### 实验二：TLS 证书验证

使用本地测试 CA 签发服务器证书，分别验证正确主机名、错误主机名、过期证书和不受信根。不得对真实设备修改证书策略。

### 实验三：SSH/RDP 受控运维

通过跳板虚拟机访问目标虚拟机，配置个人账户、限时权限和日志。比较断开与注销，记录文件传输和端口转发控制。

### 实验四：远程变更证据链

模拟一次网关配置维护：工单、VPN/跳板日志、文件哈希、目标登录、配置差异、服务重载和业务验证。所有数据使用虚构资产。

## 22. 面试表达

### 60 分回答

远程服务分析要区分名称解析、网络可达、安全通道、身份认证、对象授权和业务执行。DNS、DHCP 和 NTP分别影响目标、网络参数和时间；HTTP/TLS、SSH、RDP、SMB 和 WinRM 承载不同访问。工控远程运维应经过 VPN、工业 DMZ、堡垒机、个人账户、审批和会话审计。

### 90 分回答

我会沿完整会话链分析远程运维：DNS/NTP 确保目标与时间，VPN 建立边界通道，堡垒机绑定个人身份和目标授权，RDP/SSH/WinRM 进入主机，工程软件或发布服务再接触 PLC。每层用账户、会话 ID、证书、时间和日志关联。TLS 只保护传输，VPN 也不等于目标授权；API 返回 200 或 RDP 登录成功都不能证明控制操作完成。最终要核对配置版本、工业协议操作、PLC 在线状态、联锁和现场反馈。

### 常见追问

**HTTPS 是否代表 Web 系统安全？** 不代表。它主要保护传输，越权、注入和危险业务逻辑仍可能存在。

**VPN 为什么不能直接连接 PLC 网段？** VPN 只建立网络通道，直接连接会扩大可达面并绕过目标级审批、跳板审计和文件控制。

**RDP 断开是否等于退出？** 不等于。会话和应用可能继续运行，应按流程注销并验证工程连接释放。

**DNS 名称能否证明设备身份？** 不能。名称解析只是地址映射，身份仍需证书、主机密钥或其他可信凭据。

## 本章总结

网络服务把基础连通性扩展为名称、时间、身份、会话和管理能力。成熟工控安全工程师必须逐层验证，从 DNS 和 TCP 到 TLS、认证、授权、工程操作和现场反馈；远程运维的安全来自个人身份、最小路径、审批、审计、文件控制和最终状态验证，而不是单独依赖 VPN 或加密协议。
