# 网络安全工程基础

安全工程不是罗列漏洞和采购产品，而是在业务与物理约束下识别资产、信任边界、威胁和失效模式，设计可验证、可运营、可恢复的控制。工控环境尤其强调安全生产、可用性、确定性和长生命周期，因此互联网系统中的最佳实践不能不经评估直接套用。

## 学习目标

- 准确定义资产、威胁、漏洞、暴露、攻击面、控制与风险；
- 理解 CIA、真实性、可归责性、安全生产与韧性的关系；
- 建立数据流图、信任边界、攻击路径和滥用场景；
- 应用最小权限、职责分离、安全默认值和纵深防御；
- 区分预防、检测、响应、恢复和补偿控制；
- 将漏洞严重性转换为具体工业场景风险；
- 设计安全需求、验证方法、监测指标和持续改进闭环；
- 区分网络系统失陷与真实物理后果。

## 1. 安全工程原理：解决什么问题

任何系统都无法只靠一个边界设备或一次加固获得永久安全。人员、软件、配置、网络、供应链和物理过程持续变化。安全工程的核心循环是：

```text
理解系统 -> 识别资产与边界 -> 建模威胁与失效
-> 选择控制 -> 实施与验证 -> 监测 -> 响应恢复 -> 复盘改进
```

每项控制都要回答：解决哪个风险、在什么假设下有效、如何证明生效、失效后怎样发现、怎样恢复。

## 2. 核心术语

### 2.1 资产

资产是需要保护且对组织有价值的对象，包括：

- 物理过程、人员安全、产能和产品质量；
- PLC、DCS、SIS、工程站、网络设备和现场仪表；
- 控制逻辑、点表、配方、证书、凭据和历史数据；
- 服务、通信链路、供应商支持和恢复能力；
- 声誉、合规和事故证据。

资产清单不能只列 IP。还要记录角色、所有者、位置、版本、依赖、数据、通信、关键性、备份和生命周期。

### 2.2 威胁、威胁主体与威胁事件

威胁是可能造成损害的潜在原因；威胁主体是可能实施行为的人或组织；威胁事件是具体发生的行为或自然/技术事件。内部误操作、供应链缺陷、勒索软件、设备故障和断电都可能形成安全风险。

威胁建模不应只假设“高级攻击者”，也要覆盖高概率的配置错误、共享账户、远程维护遗留和恢复失败。

### 2.3 漏洞、弱点与错误配置

漏洞是可被利用或触发的弱点，可能存在于代码、设计、配置、流程和人员。CVE 只是公开漏洞标识体系中的一部分；未公开缺陷、弱口令、过宽信任和缺失审批同样重要。

### 2.4 暴露与攻击面

暴露描述资产在特定路径上可被接触的程度。攻击面包括网络端口、Web/API、无线、USB、工程文件、更新包、供应商通道和人员流程。

端口存在不等于可利用，端口关闭也不等于资产安全。分析还需认证、权限、状态、版本和下游能力。

### 2.5 风险

风险是威胁利用弱点，在特定条件下造成后果的不确定性。可用简化形式帮助沟通：

```text
风险 ≈ 发生可能性 × 后果
```

这不是精确物理公式。评分必须记录场景、证据、假设和不确定性，不能用一个数字掩盖判断过程。

## 3. 安全目标：CIA 及其扩展

### 3.1 机密性

防止未授权披露。工业敏感信息包括拓扑、点表、配方、维护窗口、凭据和安全逻辑。泄露可能不立即停产，却会提高后续攻击精度。

### 3.2 完整性

确保数据、软件和状态未被未授权或意外改变。工控系统中完整性常比普通办公数据更直接影响控制：错误设定值、点表和程序可能改变过程。

### 3.3 可用性

确保授权主体在需要时获得功能与数据。可用性不仅是设备在线，还包括实时性、正确降级、冗余切换和恢复时间。

### 3.4 真实性、可归责性与不可否认性

真实性确认主体/数据来源，可归责性让操作能够关联责任主体。数字签名可支持不可否认性的技术基础，但仍依赖私钥控制、审批与审计。

### 3.5 Safety 与 Security

Safety 关注非蓄意或蓄意事件下人员、环境和设备免受不可接受伤害；Security 关注对抗恶意或未授权行为。二者目标相互影响：安全控制不能妨碍紧急停车，网络攻击也可能制造安全事故。

## 4. 系统、数据流与信任边界

安全建模从系统图开始：组件、外部主体、数据存储、数据流和信任边界。

```text
供应商 -> VPN -> DMZ 堡垒机 -> 工程站 -> PLC -> 执行器
                               |         |
                               v         v
                            项目库     现场反馈
```

信任边界是身份、权限、管理责任或安全假设发生变化的位置。穿越边界的数据应进行身份验证、授权、类型/范围校验和审计。

图应标注协议、方向、端口、身份、加密、频率、数据类型和业务负责人。只画设备图不够，因为攻击者利用的是交互关系。

## 5. 数据流、控制流与状态流

### 数据流

数据从传感器、用户或外部系统进入，经转换、存储和发布到消费者。机密性与完整性控制必须覆盖整个生命周期。

### 控制流

控制流描述谁能触发服务、审批、下装和执行。身份、角色、工单和设备模式决定路径是否允许。

### 状态流

```text
登录 -> 授权 -> 变更审批 -> 配置发布 -> PLC 接收
-> 输出逻辑 -> 执行器动作 -> 反馈与联锁
```

安全结论必须明确证据停在哪一层。发现账户登录不等于发生配置变更，配置入库也不等于现场动作。

## 6. 威胁建模方法

威胁建模不是一次性文档，而是随架构和业务变化更新。基本步骤：

1. 定义范围、目标和不可接受后果；
2. 识别资产、主体、组件和依赖；
3. 绘制数据/控制/状态流与信任边界；
4. 枚举滥用场景、故障和攻击路径；
5. 评估现有控制与残余风险；
6. 定义安全需求和验证证据；
7. 记录假设、未知项和责任人。

可以使用 STRIDE、攻击树、Kill Chain、ATT&CK 等帮助枚举，但框架不能替代现场知识。工业模型必须加入过程约束、控制回路和安全屏障。

## 7. 攻击树与攻击路径

攻击树从目标分解必要或可选条件：

```text
目标：未经授权改变 PLC 设定值
├─ 获得合法工程账户
│  ├─ 凭据泄露
│  └─ 供应商共享账户
├─ 利用工程站/发布服务漏洞
└─ 直接访问未认证协议

共同后续条件：网络可达、PLC 接受写入、地址语义正确、联锁未阻断
```

路径分析应同时寻找阻断点和检测点。最优控制不一定在最末端；关闭不必要供应商通道可能比为每台 PLC 部署复杂检测更有效。

## 8. 安全风险评估

风险评估至少记录：

- 资产与业务/物理后果；
- 威胁场景和攻击者能力；
- 漏洞与利用前提；
- 网络、身份和现场可达性；
- 现有预防、检测、响应和安全屏障；
- 可能性、影响、置信度和证据；
- 建议、负责人、期限与残余风险。

### 8.1 严重性不等于场景风险

CVSS 等严重性评分描述通用技术属性，现场风险还受资产角色、暴露、过程、冗余和 SIS 影响。高分漏洞位于隔离备用设备上可能优先级较低；中等权限缺陷若位于配置发布链可能风险更高。

### 8.2 定性、半定量与不确定性

高/中/低或数值矩阵都可使用，但必须统一定义。评分前先写场景，评分后保留依据。未知版本、缺失拓扑和未验证联锁应作为不确定性，而非默认为安全。

## 9. 安全控制分类

### 9.1 按功能

- 预防：身份、分区、白名单、参数校验；
- 检测：日志、IDS、完整性监测、异常告警；
- 响应：隔离、吊销凭据、切换安全模式；
- 恢复：备份、重建、回滚、灾备；
- 威慑/治理：政策、责任和审计。

### 9.2 按性质

- 管理控制：制度、培训、供应商合同、审批；
- 技术控制：防火墙、MFA、签名、EDR；
- 物理控制：机柜、门禁、摄像和端口封堵。

补偿控制用于无法直接实施首选控制时降低风险，必须验证有效性和持续运营，不能成为永久不整改的借口。

## 10. 最小权限与职责分离

最小权限要求主体只获得完成任务所需的对象、操作、范围和时间。职责分离避免同一人独立完成申请、审批、执行和审计。

工业例子：

- 操作员可启停批准设备，但不能改 PLC 程序；
- 工程师可编辑离线项目，但下装需维护审批和现场确认；
- Web 查询账户只读历史视图，发布服务使用独立身份；
- DBA 可恢复数据库，但不能单独批准配方变更。

及时权限（JIT）和限时授权可减少长期高权限，但必须设计断网应急和 break-glass 流程。

## 11. 安全默认值与失败方式

安全默认值意味着新资产、账户和接口默认最小开放；失败后进入可预测的安全状态。

工控系统中“失败关闭”不是简单断电。通信失联时阀门保持、关闭还是转手动取决于工艺危险分析。网络安全控制应与控制逻辑、失效安全位置、看门狗和 SIS 协调。

错误处理避免泄露敏感细节，同时应在受控日志中保留诊断信息。静默失败会让操作员误判状态。

## 12. 纵深防御

纵深防御使用相互独立、作用不同的多层控制：

```text
人员与流程
-> 物理安全
-> 分区分域/DMZ
-> 身份与远程访问
-> 主机/应用/协议安全
-> 监测与响应
-> 控制器限幅、联锁与 SIS
-> 备份与恢复
```

多层不等于堆叠同类产品。若所有层共享同一域账户、管理平台和网络路径，单点失陷仍可穿透。应分析独立性、共同原因失效和绕过路径。

## 13. 零信任思想与工控边界

零信任强调不因网络位置自动信任、持续验证身份与上下文、最小访问和显式策略。它不是购买某个产品，也不表示每个 PLC 都必须直接接入云身份系统。

在工控环境可体现为：

- 每个远程会话绑定个人和设备身份；
- 访问按目标、协议、操作和时间授权；
- 网络区仍保留，身份控制与分区互补；
- 对遗留设备通过代理、网关和跳板实现策略；
- 连续监测但避免对实时通信引入不可接受延迟。

## 14. 安全开发生命周期

安全需求应从设计开始：

```text
需求 -> 架构/威胁建模 -> 安全设计 -> 实现与评审
-> 测试 -> 签名发布 -> 部署加固 -> 监测响应 -> 退役
```

需求要可验证，例如“只有发布角色可修改批准配方，所有变更记录工单和旧新值”，而非空泛的“系统必须安全”。

实现阶段进行代码评审、静态/动态测试、依赖治理和秘密扫描；发布阶段保护构建与签名；运行阶段管理补丁、证书、配置和日志。

## 15. 资产、配置与变更管理

不知道资产和版本，就无法判断漏洞与补丁。配置基线应包含：

- 硬件/软件/固件、角色、所有者和位置；
- 账户、服务、端口、通信矩阵和信任关系；
- 工程项目、点表、配方、证书和备份；
- 批准偏差、补偿控制和生命周期状态。

变更必须有原因、范围、风险、测试、审批、实施、验证和回滚。紧急变更也要事后补录与复盘。

## 16. 漏洞管理

漏洞管理不是“扫描—打补丁”两步：

1. 维护准确资产与版本；
2. 收集厂商公告、CVE、配置弱点和现场发现；
3. 验证适用性、暴露和利用条件；
4. 评估过程风险与补偿控制；
5. 测试补丁/缓解的兼容性；
6. 安排维护、备份、回滚和验证；
7. 跟踪例外、复测和残余风险。

主动扫描可能影响老旧控制器，生产环境优先被动识别、配置清单和厂商证据。必要主动验证必须授权、限速和监控。

## 17. 安全监测与检测工程

检测由数据源、分析逻辑、上下文和响应组成。一个告警规则必须说明：

- 观察哪个威胁行为；
- 需要哪些日志/流量；
- 正常基线和例外是什么；
- 告警如何调查与升级；
- 数据缺失如何发现；
- 规则如何测试和维护。

工业高价值检测包括新资产/新会话、非维护时段工程操作、异常写功能码、程序下载、联锁旁路、远程账户变化和日志源失联。

检测不能只看攻击签名，也要把命令与工单、设备模式和现场状态对应。

## 18. 事件响应与恢复

工控响应优先保护人员、环境和过程。典型阶段：准备、发现分析、遏制、清除、恢复和复盘，但顺序可因安全生产调整。

遏制不是默认拔网线。断开工程站可能安全，断开冗余控制网络可能扩大事故。响应预案应预先定义资产角色、替代操作、授权人和安全状态。

恢复需要可信备份、重建介质、配置基线、证书/凭据轮换、在线程序校验、功能测试和加强监测。系统启动不等于生产恢复。

## 19. 人员、流程与供应链

人员不是简单“最薄弱环节”，而是系统设计的一部分。界面、审批和应急流程若难以使用，会促使共享账户和绕过。

供应商风险包括远程访问、更新包、开发依赖、服务账户、工程文件和生命周期支持。合同应明确漏洞通知、补丁、日志、远程审批、数据处理和事件协作，但合同条款仍需技术验证。

## 20. 安全指标与度量

指标应支持决策，不能只追求数量。示例：

- 关键资产清单完整率和未知资产发现时间；
- 远程访问中个人身份/MFA/审批覆盖率；
- 关键日志在线率与时间偏差；
- 高风险漏洞处置与例外老化时间；
- 配置备份恢复成功率与实际 RTO；
- 非维护时段工程操作调查闭环时间；
- 安全联锁和应急流程演练结果。

“扫描漏洞数量下降”可能来自扫描覆盖下降，必须同时监测数据质量。

## 21. 工控场景：远程运维风险评估

某工厂计划让供应商直接通过 VPN 访问工程站。通用方案只描述“VPN 已加密”，但完整评估发现：供应商共享账户、目标网段过宽、工程站本地管理员权限、文件可直接映射、无维护工单关联、PLC 下载无双人确认。

风险场景为凭据失陷或误操作后，供应商可修改项目并下装。整改采用个人 MFA、DMZ 堡垒机、单目标限时授权、受控文件中转、会话审计、项目差异评审、现场双人确认和 PLC 在线校验。VPN 只是其中一层。

## 22. 攻击链与物理后果

```text
初始入口 -> 凭据/主机失陷 -> 跨越区域边界
-> 获得工程/发布权限 -> 理解过程和点位
-> 修改配置/程序/命令 -> 隐藏或干扰反馈
-> 控制逻辑、设备与 SIS 响应 -> 物理后果
```

攻击链分析必须标出每一步条件、证据、控制和不确定性。网络层到达 PLC 不是终点，攻击者还需理解状态、地址和过程；安全联锁也不能被假定永远有效，需验证独立性和旁路状态。

## 23. 防护架构示例

```text
互联网/供应商
  -> MFA VPN
  -> 工业 DMZ 堡垒机、文件中转、代理
  -> 工业防火墙与通信白名单
  -> 工程站/SCADA 区：主机加固、应用控制、日志
  -> 控制器区：协议限制、工程操作审计、限幅
  -> 现场与 SIS：独立联锁、反馈、失效安全
```

每层都应有所有者、配置基线、监测、测试和恢复方案。

## 24. 隔离实验

### 实验一：威胁建模

对虚拟水箱控制系统绘制组件、数据流、信任边界和安全屏障，枚举至少一个保密性、完整性和可用性场景，并写出验证证据。

### 实验二：最小权限

设计操作员、工程师、发布服务和审计员角色，验证每个角色只能访问批准对象。记录拒绝事件和紧急授权流程。

### 实验三：纵深防御失效

在隔离网络模拟单层控制失效，例如 Web 凭据泄露，验证网络边界、数据库权限、发布审批和 PLC 限幅如何继续阻断路径。

### 实验四：响应与恢复桌面演练

给出工程站疑似失陷场景，团队按人员安全、证据、遏制、替代操作、重建、在线校验和复盘形成行动卡。使用虚构资产，不操作生产系统。

## 25. 面试表达

### 60 分回答

安全工程先识别资产、信任边界、威胁、漏洞和后果，再选择预防、检测、响应和恢复控制。CIA 中工控环境尤其重视完整性、可用性和 Safety。漏洞严重性不等于现场风险，还要看资产角色、网络可达、权限、控制逻辑和安全联锁。

### 90 分回答

我会从物理过程和不可接受后果反推安全需求，画出数据流、控制流、状态流和区域边界，建立带前提的攻击路径。控制设计采用最小权限、职责分离和纵深防御，并给每层定义验证与失效检测。例如供应商 VPN 只解决加密通道，还需个人 MFA、DMZ 堡垒机、单目标限时权限、项目差异、下装审批、PLC 在线校验和独立联锁。评估结果区分技术严重性、场景风险、证据置信度和残余风险。

### 常见追问

**高 CVSS 是否一定优先修复？** 不一定。还需资产关键性、暴露、利用前提、过程后果和补偿控制，但高严重性不能因缺少现场证据被长期忽略。

**纵深防御是不是多买几种产品？** 不是。重点是独立、不同功能的控制覆盖预防、检测、响应和恢复，并避免共同原因失效。

**工控系统为什么不能直接断网响应？** 隔离动作可能影响监控、冗余和安全操作，应依据资产角色、过程状态和预案选择遏制方式。

**零信任是否取消网络分区？** 不取消。身份与细粒度授权应和区域、通道、协议限制及现场安全屏障互补。

## 本章总结

网络安全工程以系统理解和风险场景为起点，以可验证控制和持续运营为闭环。成熟工控安全工程师不会把漏洞、端口、产品或评分孤立看待，而会沿信任边界和三类流确认利用条件、系统传播和物理后果，并把人员安全、生产连续性、检测与恢复纳入同一设计。
