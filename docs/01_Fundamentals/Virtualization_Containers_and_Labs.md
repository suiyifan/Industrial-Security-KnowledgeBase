# 虚拟化、容器与隔离实验环境

协议分析、漏洞复现和安全工具测试需要可重复实验，但任何主动请求、畸形报文或恶意样本都不应进入生产控制网。虚拟化与容器提供环境隔离、快照、资源控制和自动化能力，但配置错误仍可能桥接办公网、暴露服务或泄露凭据。安全实验环境必须从网络边界和恢复目标开始设计。

## 学习目标

- 理解虚拟机、Hypervisor、虚拟硬件与宿主机关系；
- 区分桥接、NAT、Host-only、Internal 等虚拟网络；
- 正确使用快照、克隆、模板、备份和恢复；
- 理解容器镜像、层、namespace、cgroup、volume 与网络；
- 识别容器并非强虚拟机边界的原因；
- 设计无生产连通、有限资源、可重建的工控靶场；
- 记录版本、配置、时间、输入和输出，保证实验可复现；
- 将实验现象与真实工业环境边界区分。

## 1. 虚拟化解决什么问题

虚拟化把物理资源抽象为多个隔离运行环境，使不同操作系统和实验角色共享硬件。安全研究中可用于：

- 保存特定版本 Windows/Linux 和厂商软件；
- 模拟工程站、SCADA、数据库、PLC/协议服务和监测节点；
- 在测试前创建恢复点；
- 控制 CPU、内存、磁盘与网络；
- 复制相同基线完成对照实验；
- 隔离不可信样本和危险配置。

隔离是有条件的。虚拟机仍共享 Hypervisor 与宿主硬件，Guest Tools、共享剪贴板、共享目录、USB 和桥接网络都会形成边界。

## 2. Hypervisor 与虚拟机原理

Hypervisor 管理虚拟 CPU、内存、设备和中断，将 Guest 操作系统的操作映射到物理硬件。

```text
虚拟机 A：工程站    虚拟机 B：PLC 模拟器
       \              /
       Hypervisor / 虚拟交换机
                |
       宿主 CPU、RAM、磁盘、网卡
```

常见分类：

- Type 1：直接运行在硬件或专用虚拟化层上，常用于服务器；
- Type 2：运行在宿主操作系统之上，常用于个人实验室。

分类有助理解管理边界，但具体安全性取决于产品、补丁、配置、管理面和宿主机。

### 2.1 虚拟 CPU 与内存

为虚拟机分配 vCPU 不等于获得独占物理核心。过度分配会产生调度等待；内存不足会触发宿主交换或回收，影响实验时序。

工业协议周期和性能实验需记录宿主负载、vCPU、内存、时间同步和虚拟化开销。普通虚拟机不适合作为严格硬实时结论的唯一证据。

### 2.2 虚拟磁盘

虚拟磁盘可固定或动态增长，支持差分链。动态磁盘节省空间，但宿主磁盘耗尽会损坏实验。长快照链会降低性能并增加恢复复杂度。

## 3. 虚拟网络模式

虚拟网络是实验安全的首要边界。

### 3.1 桥接网络

虚拟机像独立物理主机一样接入宿主所在二层网络。它可能获得企业/家庭地址并直接访问真实设备，风险最高。工控漏洞实验默认不使用桥接。

### 3.2 NAT

虚拟机通过宿主/NAT 访问外部网络，外部通常不能主动连接 Guest，除非配置端口转发。NAT 不是完整安全边界：Guest 仍可向互联网和宿主可达网络发起连接。

### 3.3 Host-only

虚拟机与宿主专用虚拟接口通信，通常不直接访问外部网络。适合宿主运行抓包/管理工具，但若宿主开启路由、ICS 或 VPN，仍可能形成转发路径。

### 3.4 Internal/Private Network

只有同一虚拟网络中的虚拟机互通，宿主通常无三层接口。适合最严格隔离的攻击者、目标和监测节点。需要通过受控方式导入工具和导出证据。

### 3.5 多网卡与意外路由

一台跳板虚拟机同时连接 NAT 和内部网，可能成为实验网到外部的路由器。即使没有显式转发，代理、端口转发和应用中继也会跨界。

```text
禁止默认拓扑：实验目标 -- Internal -- 双网卡攻击机 -- NAT/Bridge
```

若确需更新依赖，使用阶段化流程：断开实验目标、启用临时更新节点、下载并校验、关闭外联、恢复隔离，再启动靶场。

## 4. 实验网络安全检查

开始前确认：

- 所有虚拟网卡模式和 MAC/IP；
- 宿主路由、ICS、VPN、代理和端口转发；
- DHCP 来源，防止实验 DHCP 泄漏；
- IPv6 是否形成未预期路径；
- DNS/NTP 是否需要本地服务；
- 虚拟交换机混杂模式与抓包权限；
- Guest 防火墙和监听服务；
- 无真实 PLC、网关或生产地址可达。

可在靶场网关部署默认拒绝规则，并通过一次只读连通性验证证明生产网不可达。不要依赖“我记得选了 Host-only”。

## 5. 快照、克隆、模板与备份

### 5.1 快照

快照保存虚拟磁盘和可选内存状态的差异，适合短期回滚。它不是独立备份：基础磁盘损坏或宿主丢失时，快照链也可能不可用。

快照命名应记录：时间、系统版本、实验阶段和是否关机一致。例如：

```text
win10-lab-before-driver-install-2026-07-14
```

数据库和工程软件运行时快照可能产生应用不一致，优先使用应用支持的备份或关机快照。

### 5.2 克隆与模板

完整克隆独立占用空间，链接克隆依赖父磁盘。模板用于生成一致环境。克隆后要处理主机名、SID/机器身份、SSH 主机密钥、证书、MAC 和静态 IP，避免多个实例共享身份。

### 5.3 备份与恢复

关键实验基线应导出或备份到独立存储，并验证导入恢复。记录 Hypervisor 版本、虚拟硬件版本、磁盘格式和校验值。

## 6. Guest Tools 与共享功能

增强工具提供剪贴板、拖放、共享目录、时间同步和驱动。便利功能会扩大边界：

- 恶意样本可读取共享目录或剪贴板；
- 双向拖放可能把样本带回宿主；
- USB 直通接触真实设备；
- Guest Tools 漏洞可能影响宿主；
- 时间自动同步会干扰需要控制时钟的实验。

不可信实验默认关闭共享剪贴板、拖放、共享目录和 USB 自动连接。证据通过受控只读介质或隔离中转导出。

## 7. 容器运行模型

容器通常共享宿主内核，通过 namespace 隔离视图、cgroup 管理资源、文件系统层提供镜像根目录。

```text
容器进程
  -> namespaces：PID、网络、挂载、用户、IPC、UTS
  -> cgroups：CPU、内存、I/O、进程数
  -> capabilities / seccomp / MAC
  -> 共享宿主 Linux 内核
```

容器不是轻量虚拟机。内核漏洞、危险挂载和过高权限可能突破边界。运行 Windows 容器与 Linux 容器也有不同内核约束。

## 8. 镜像、层与容器生命周期

镜像由只读层和元数据组成，容器运行时增加可写层。删除容器会丢失可写层数据，持久数据应使用 volume 或绑定挂载。

```text
Dockerfile/构建上下文 -> 镜像层 -> 镜像摘要
-> 容器实例 + 可写层 -> 日志/volume/网络
```

标签如 `latest` 可移动，不能作为唯一版本证据。应记录镜像摘要、构建文件、基础镜像、依赖和扫描结果。

### 8.1 构建安全

- 使用最小可信基础镜像并固定版本/摘要；
- 多阶段构建减少编译工具进入运行镜像；
- 不把密码、私钥和客户数据复制进层；
- 使用 `.dockerignore` 限制构建上下文；
- 以非 root 用户运行；
- 扫描依赖并记录 SBOM；
- 签名/证明构建来源并保护 Registry。

即使后续删除文件，秘密也可能留在早期镜像层中。

## 9. 容器权限与隔离

危险配置包括：

- `--privileged`；
- 挂载宿主根目录或容器运行时 Socket；
- 共享宿主 PID/网络命名空间；
- 赋予不必要 Capabilities；
- 关闭 seccomp、SELinux/AppArmor；
- 以 root 运行并允许写入程序目录。

最小化能力，根文件系统尽量只读，设置 `no-new-privileges`、进程/内存/CPU 限制，并使用专用网络和秘密挂载。

容器中的 root 在用户命名空间未正确隔离时可能具有强宿主影响。不要把“进程在容器里”当作可信结论。

## 10. 容器网络

常见容器网络包括 bridge、host、none 和自定义网络。默认 bridge 可能通过 NAT 出网；host 网络让容器直接共享宿主网络栈，隔离较弱；none 适合完全不需要网络的任务。

发布端口会把容器服务暴露到宿主接口：

```text
宿主 0.0.0.0:502 -> 容器 172.x.x.x:502
```

绑定 `127.0.0.1`、指定管理接口和绑定所有接口意义不同。还要检查宿主防火墙、容器运行时自动规则和上游网络。

实验 PLC 模拟器不应发布到办公网。优先放入仅靶场组件可访问的自定义内部网络。

## 11. Volume、绑定挂载与证据

Volume 由运行时管理，绑定挂载直接映射宿主路径。绑定挂载的写权限可能让容器修改宿主证据或配置。

原始 PCAP/固件分析使用只读挂载：

```text
原始证据（宿主只读） -> 容器只读挂载 -> 分析输出到独立目录
```

记录输入和输出哈希、镜像摘要、命令参数和时间。容器删除不会自动清理 volume，敏感实验结束后按证据保留策略处理。

## 12. 编排与可复现配置

Compose 等声明式文件可以定义服务、网络、volume、健康检查和资源限制。一个可复现实验项目应包含：

```text
lab/
├─ README.md              # 目标、授权边界、拓扑和步骤
├─ compose.yaml           # 服务、网络、volume、资源
├─ configs/               # 脱敏配置
├─ fixtures/              # 固定测试向量
├─ expected/              # 预期结果
└─ evidence-manifest.txt  # 版本、哈希、时间
```

声明式配置不自动保证安全，应检查端口发布、权限、秘密、镜像来源和外部网络。

## 13. 工控实验拓扑设计

推荐最小拓扑：

```text
Internal Lab Network（无默认网关）

[工程/客户端] ---- [OpenPLC/协议模拟器]
       |                    |
       +------ [监测/抓包] -+

可选管理接口：仅 Host-only，默认关闭转发
```

角色分离便于观察：客户端产生请求，模拟器响应，监测节点被动采集。不要在同一主机上混合所有角色后推断真实网络时序。

### 13.1 物理过程模拟

PLC 模拟器可连接水箱、泵或温度等简单过程模型：

```text
过程模型 -> 虚拟传感器 -> PLC 逻辑 -> 虚拟执行器 -> 反馈
```

实验要记录扫描周期、过程步长、变量映射和初始状态。模拟结果用于理解机制，不直接证明特定真实设备行为。

## 14. 漏洞复现环境

复现应固定：

- 产品、固件/软件、架构和依赖版本；
- 操作系统、补丁和防护机制；
- 网络拓扑、地址、时间和资源；
- 输入样本、脚本和哈希；
- 触发条件、预期、实际与失败现象；
- 快照、清理和恢复步骤。

公开 PoC 视为不可信代码，先静态检查，在无外联环境运行。不要把真实凭据、许可证或厂商受限镜像公开提交。

## 15. 恶意样本与危险工具隔离

高风险样本实验应使用专用宿主或强隔离环境，而非日常办公电脑。至少：

- 无桥接、无共享目录/剪贴板/USB；
- 无真实企业 VPN、凭据和云同步；
- 独立本地账户与存储；
- 捕获 DNS/网络并提供本地模拟服务；
- 入口/出口文件经哈希、审批和受控中转；
- 实验结束回滚并验证宿主状态。

虚拟机逃逸属于低概率高影响风险，不能宣称普通虚拟机提供绝对隔离。

## 16. 时间、确定性与性能边界

快照恢复可能造成时间回拨，影响证书、Kerberos、日志和协议序号。实验时间线应记录宿主与 Guest 的时钟源，必要时使用本地 NTP 并禁用冲突同步。

虚拟 CPU 调度、动态磁盘和宿主负载会产生抖动。协议功能和安全逻辑可以在虚拟环境验证，但严格实时性、抗干扰和设备负载结论需要物理测试台或厂商环境。

## 17. 资源限制与拒绝服务安全

为实验虚拟机/容器设置 CPU、内存、磁盘、进程数和日志容量上限，防止测试拖垮宿主。日志采用滚动，PCAP 限制大小与数量。

拒绝服务实验的目标应是观察受控资源状态，而不是追求最大流量。定义停止条件：CPU、内存、错误率、响应延迟或模拟过程偏差达到阈值即终止。

## 18. 数据流、控制流与状态流

### 数据流

镜像、配置、样本和证据在宿主、虚拟机、容器与制品库之间流动。每次跨边界应验证来源、哈希和敏感性。

### 控制流

Hypervisor/容器运行时控制启动、网络、挂载、快照和资源；实验脚本触发协议行为；审批流程决定是否允许外联或导出。

### 状态流

```text
基线镜像 -> 实例启动 -> 网络隔离验证 -> 实验执行
-> 证据采集 -> 清理/回滚 -> 基线完整性复核
```

应用返回成功、模拟 PLC 变量变化和物理模型反馈是不同状态，需要逐层观察。

## 19. 工控场景案例：错误桥接导致实验外泄

研究人员复制了一台协议模拟器虚拟机，网卡从 Internal 误改为 Bridge。模拟器使用与生产 PLC 相同的静态 IP，接入办公/工程网络后形成地址冲突，导致工程站间歇连接错误。

根因不是模拟器漏洞，而是虚拟网络变更缺少检查。整改包括模板默认无网卡/内部网、启动前自动检查网络模式和生产地址、实验宿主禁止连接控制网、独立物理实验 VLAN，以及在交换机监测重复 IP 和新 MAC。

## 20. 安全风险与攻击链

```text
不可信镜像/样本 -> 在带共享目录和外联的虚拟机运行
-> 窃取宿主共享数据或通过桥接网络扩散
-> 获取工程凭据/访问真实设备 -> 修改控制系统
```

另一类风险来自容器：恶意镜像利用危险宿主挂载/运行时 Socket，获得宿主控制，再接触企业凭据和网络。防护必须覆盖镜像供应链、运行权限和网络隔离。

## 21. 防护与实验治理

- 实验环境与办公、生产和客户网络物理或逻辑隔离；
- 模板默认 Internal/none 网络，桥接需要显式审批；
- 宿主不保存生产 VPN、客户凭据和自动云同步；
- 禁用不必要共享、USB、端口转发和容器高权限；
- 镜像固定摘要、验证来源、扫描并记录 SBOM；
- 以代码定义拓扑、资源、配置和预期结果；
- 原始证据只读，分析输出分离并计算哈希；
- 快照用于短期回滚，独立备份用于恢复；
- 建立停止条件、清理、销毁和事故报告流程；
- 虚拟结论明确其与真实实时设备的差异。

## 22. 隔离实验

### 实验一：验证网络模式

创建 Bridge、NAT、Host-only 和 Internal 测试网，记录地址、路由、可达性和抓包。证明 Internal 网络无法到达宿主默认网关和生产地址。

### 实验二：快照与恢复

在关机一致状态创建快照，修改配置和文件后回滚，比较文件哈希、系统时间和服务状态。再验证独立导出备份能否在另一目录恢复。

### 实验三：容器最小权限

构建本地测试服务，使用非 root、只读根文件系统、删除不必要能力、资源限制和内部网络运行。验证无法写程序目录或访问宿主敏感路径。

### 实验四：OpenPLC 协议靶场

建立客户端、OpenPLC/协议模拟器和监测节点，产生只读 Modbus 流量，将事务、寄存器和虚拟过程变量对应。禁止连接真实 PLC 和生产网。

## 23. 面试表达

### 60 分回答

虚拟机通过 Hypervisor 虚拟硬件并运行独立 Guest 内核，容器通常共享宿主内核，用 namespace 和 cgroup 隔离。工控靶场默认使用 Internal 或严格 Host-only 网络，不使用桥接；关闭共享目录、剪贴板和 USB。快照用于短期回滚，不替代独立备份。

### 90 分回答

我会先从网络和恢复目标设计实验：客户端、PLC 模拟器和监测节点位于无默认网关的 Internal 网络，宿主管理接口不转发；镜像固定摘要，配置和测试向量版本化，原始证据只读挂载。容器使用非 root、只读根、最小 Capabilities、资源限制和专用网络，禁止挂载运行时 Socket。实验记录时钟、版本、输入哈希、停止条件和状态链，并明确虚拟调度和模拟过程不能替代真实设备实时性验证。

### 常见追问

**NAT 是否等于完全隔离？** 不等于。Guest 通常仍能主动访问外部和宿主可达网络。

**快照是否等于备份？** 不等于。快照依赖基础磁盘和宿主存储，需独立导出并验证恢复。

**容器为何不是虚拟机？** 容器通常共享宿主内核，隔离依赖 namespace、cgroup 和安全策略。

**虚拟 PLC 测试能否证明真实 PLC 安全？** 不能完全证明。它适合验证协议和逻辑假设，真实固件、时序、资源和 I/O 仍需专用测试台验证。

## 本章总结

安全实验的第一成果不是漏洞截图，而是一个不会误连生产、能够恢复且可被他人复现的环境。虚拟化和容器提供隔离工具，却也引入管理面、共享功能、网络和供应链风险。成熟工控安全工程师会用明确拓扑、最小权限、固定版本、证据哈希和停止条件控制实验，并诚实限定模拟结论的适用范围。
