# Git、GitHub 与研究证据管理

Git 不只是代码备份工具，而是一个内容寻址的版本数据库。对工控安全研究而言，它用于记录知识库、工具、配置、实验步骤、修复差异和报告演进；GitHub 等协作平台则增加评审、Issue、CI、发布和权限管理。成熟工程师必须知道历史能证明什么、不能证明什么，以及如何避免把凭据、客户数据和危险实验材料永久写入仓库。

## 学习目标

- 理解工作区、暂存区、对象数据库、引用与提交图；
- 熟练使用状态、差异、提交、分支、合并、远程与标签；
- 设计可审查、可回滚、可追溯的章节和工具工作流；
- 管理二进制工程文件、实验数据和大文件；
- 防止秘密、客户信息和敏感工控数据进入仓库；
- 理解签名、保护分支、CI 和依赖供应链风险；
- 将 Git 证据与工单、实验、日志和发布物对应。

## 1. Git 的工作原理

Git 把内容保存为对象，并用哈希标识。核心对象包括：

- blob：文件内容；
- tree：目录结构和文件名到对象的映射；
- commit：根 tree、父提交、作者、提交者、时间和说明；
- annotated tag：带说明和可签名的标签对象。

```text
branch -> commit -> tree -> blob
           |
           +-> parent commit -> ...
```

分支本质是指向提交的可移动引用，`HEAD` 表示当前检出位置。提交图而非文件夹副本构成历史。

内容哈希能发现对象内容变化，但 Git 仓库不是自动可信的取证系统：有写权限的人可以创建新历史、改写引用或修改作者字段。可信度还依赖访问控制、签名、远程审计、时间来源和外部工单。

## 2. 工作区、暂存区与仓库

```text
工作区 --git add--> 暂存区/index --git commit--> 本地对象库与引用
                                              --git push--> 远程仓库
```

工作区是正在编辑的文件；暂存区保存下一次提交的精确快照；提交把暂存快照与元数据写入历史。`git add` 不是上传，`commit` 也不是推送。

```bash
git status --short
git diff
git diff --staged
git diff HEAD
```

`git diff` 默认比较工作区与暂存区，`--staged` 比较暂存区与 `HEAD`。提交前至少检查两者，避免漏交或误交。

### 2.1 文件状态

文件可能是未跟踪、已跟踪未修改、已修改、已暂存或被忽略。`.gitignore` 只影响尚未跟踪文件，不会让已提交秘密从历史消失。

```bash
git ls-files
git check-ignore -v path/to/file
git status --ignored --short
```

## 3. 创建高质量提交

提交应表达一个可独立理解的变更目的。章节正文、无关格式化、生成报告和工具代码不应随意混成一个巨大提交。

```bash
git add docs/01_Fundamentals/Linux_Basics.md
git diff --staged
git commit -m "Expand Linux security fundamentals"
```

推荐说明回答：改了什么、为什么改、如何验证、有哪些限制。提交说明不应包含凭据、内部地址和客户名称。

### 3.1 选择性暂存

```bash
git add -p
git restore --staged path
git diff --check
```

`git add -p` 可按补丁块选择内容，适合把混合修改拆成逻辑提交。`restore --staged` 只取消暂存，默认不删除工作区内容。执行任何恢复或清理命令前都要确认作用对象。

### 3.2 作者、提交者与时间

提交记录作者和提交者，两者可以不同；时间来自本机并可被设置。它们是协作元数据，不应单独作为法律或取证意义的绝对证明。关键变更应关联评审、工单、签名和服务器审计。

## 4. 分支、合并与提交图

分支让不同工作沿同一历史基点独立推进：

```bash
git switch -c codex/linux-chapter
git branch --show-current
git log --graph --decorate --oneline --all
```

合并保留两条开发线的共同祖先与结果；快进合并只移动引用；rebase 通过重新应用补丁创建新提交，使历史线性但会改变提交 ID。

```text
A---B---C main
     \
      D---E topic
```

是否使用 merge 或 rebase 是团队策略。不要对其他人正在使用的公共历史随意 rebase/force push。必须强制更新时优先使用 `--force-with-lease` 并确认远程状态，但受保护分支通常应禁止强推。

### 4.1 冲突处理

冲突表示 Git 无法自动决定结果，不代表某一方一定错误。处理流程：

1. 查看 `git status`；
2. 理解双方意图和共同祖先；
3. 编辑为最终正确内容；
4. 运行格式、链接和测试检查；
5. 暂存并继续合并/rebase；
6. 审查最终差异，而非只删除冲突标记。

二进制 PLC 工程文件通常无法文本合并，应通过锁定、单人编辑、厂商比较工具、导出文本或项目级版本机制管理。

## 5. 远程仓库与协作

```bash
git remote -v
git fetch --prune
git branch -vv
git log --left-right --graph HEAD...origin/main
```

`fetch` 获取远程对象与引用而不自动修改当前工作；`pull` 通常组合 fetch 与 merge/rebase。成熟工作流先获取、查看差异，再决定整合方式。

远程 URL 可能使用 HTTPS 或 SSH。凭据应由系统凭据管理器、SSH agent 或短期令牌管理，不能写入脚本和远程 URL。

## 6. GitHub 协作模型

GitHub 提供 Pull Request、Issue、Review、Actions、Release 和权限控制。推荐流程：

```text
Issue/章节任务 -> 分支 -> 小提交 -> 自动检查
              -> Pull Request -> 技术/安全评审 -> 合并
              -> 标签/发布 -> 产物与校验值
```

保护主分支，要求评审与检查通过；限制管理员绕过；关键目录可设置 CODEOWNERS。评审要验证技术内容、实验边界、敏感信息、生成文件和链接，而不是只看 CI 绿灯。

### 6.1 Issue 与变更证据

Issue 可记录目标、范围、验收标准和风险。提交和 PR 引用 Issue，最终报告再引用发布版本。这样能够回答“为什么修改、谁审查、哪些文件变化、验证了什么”。

## 7. 标签、版本与发布物

轻量标签只是引用，附注标签具有说明、创建者和可签名对象：

```bash
git tag -a v1.0.0 -m "First reviewed release"
git show v1.0.0
```

版本号应有语义和发布策略。研究项目发布至少包含：源提交、环境说明、依赖版本、输入样本哈希、测试结果、已知限制和产物哈希。

标签指向源码提交，不自动证明发布二进制由该源码构建。可复现构建、受控 CI、签名和制品证明可进一步建立关联。

## 8. 签名与身份可信度

Git 可使用 GPG、SSH 或平台支持方式签名提交和标签。签名证明持有相应私钥者签署了该对象，不证明内容无漏洞，也不证明签名者正确理解变更。

密钥应独立保护、设置轮换与撤销流程。平台显示“Verified”依赖其信任配置，报告应说明信任根和验证方式。

## 9. `.gitignore`、属性与行尾

`.gitignore` 用于排除构建产物、本地环境、临时证据和秘密文件模板：

```gitignore
.venv/
__pycache__/
*.pcap
*.pcapng
.env
local-evidence/
```

忽略 PCAP 是默认防泄露策略，不代表所有 PCAP 永远不能版本化；经过授权、脱敏的小型教学样本可放在明确目录并记录来源。

`.gitattributes` 可统一文本、行尾、差异驱动和 LFS 行为：

```gitattributes
*.md text eol=lf
*.ps1 text eol=crlf
*.pcap binary
```

行尾批量变化会掩盖真实修改。引入规则时应单独做规范化提交，并让评审区分机械变更与内容变更。

## 10. 二进制文件、大文件与 Git LFS

PCAP、固件、虚拟机镜像和厂商工程文件可能体积大且敏感。Git 每个历史版本都可能保留内容，不适合作为无限制证据仓库。

Git LFS 用指针文件替代普通 Git 对象中的大内容，但远端 LFS 存储仍需权限、保留和备份策略。虚拟机镜像通常应通过受控制品库管理，不提交 Git。

对二进制工程文件建议：

- 保留原文件哈希和厂商版本；
- 同时导出可审查文本、变量表或报告；
- 记录创建工具、插件和硬件目录版本；
- 不在未经授权的公共仓库保存真实项目。

## 11. 敏感信息与秘密管理

禁止提交：密码、私钥、完整令牌、客户拓扑、真实 PLC 地址、配方、未公开漏洞细节和受限固件。

```bash
git grep -n -I -E 'password|token|secret|BEGIN .*PRIVATE KEY'
git log -p --all -- path/to/suspected-file
```

关键原则：秘密一旦提交，即使后来删除文件，也可能仍在历史、分叉、缓存、CI 日志和克隆中。处置顺序通常是：

1. 立即撤销和轮换秘密；
2. 评估泄露范围和访问日志；
3. 按团队流程清理历史；
4. 通知所有克隆和分叉同步；
5. 添加扫描与预防控制。

历史重写是破坏性协作操作，不能把“从当前分支看不到”误当成已经完全删除。

## 12. 供应链与 CI 安全

仓库中的依赖清单、构建脚本、GitHub Actions 和发布令牌都属于供应链攻击面。应：

- 固定第三方 Action 到可信提交或受控版本；
- 使用最小 `GITHUB_TOKEN` 权限；
- 将来自 Fork 的代码视为不可信；
- 避免让不可信 PR 获取部署秘密；
- 审查依赖更新、安装脚本和锁文件；
- 对制品生成哈希、签名并记录源提交；
- 保护环境审批和发布权限。

CI 通过只证明配置的检查通过，不证明测试覆盖完整或内容正确。验证器本身也需要版本管理和评审。

## 13. Git 与研究证据

Git 适合保存：Markdown 记录、脚本、配置模板、脱敏小样本、测试向量、报告源文件和校验清单。原始大体积或受限证据应放在受控证据库，并在仓库记录：

```text
证据编号
采集时间与时区
采集位置与人员
原始文件 SHA-256
受控存储位置
分析副本 SHA-256
使用工具及版本
```

不要提交后又编辑“原始证据”。分析在副本上完成，原件权限只读并由独立链路保全。

## 14. 工业工程变更管理

Git 可以管理 PLC 程序导出文本、点表、报警配置和通信矩阵，但需要补充工业语义：

- 变更对应哪个装置、控制器和维护窗口；
- 是离线项目变化还是已下装在线系统；
- 编译结果、硬件配置和库版本是否一致；
- 谁批准、谁执行、如何验证、如何回滚；
- 变更是否影响联锁、SIS 或安全状态。

```text
需求/工单 -> 离线工程变更 -> 差异评审 -> 仿真/测试
          -> 维护审批 -> 现场下装 -> 在线校验 -> 标签/归档
```

Git 提交成功不等于 PLC 下装成功，工程软件的在线比较、PLC 审计和现场测试才证明运行状态。

## 15. 数据流、控制流与状态流

### 数据流

工作区内容进入暂存区和对象库，再传输到远端、CI 和发布制品。秘密或客户数据一旦进入这条数据流，可能被多处复制。

### 控制流

分支策略、评审和 CI 决定什么变更可以合并与发布。权限过宽或工作流脚本被修改会绕过控制。

### 状态流

工作区已修改、提交已创建、远端已合并、发布已生成、工程项目已下载、PLC 在线程序已改变是不同状态。必须分别验证。

## 16. 工控场景案例：PLC 项目版本不一致

工程站本地项目、Git 主分支和 PLC 在线程序出现三个版本。调查发现工程师在维护窗口临时修改并下装，但只保存了本地项目，未提交评审。之后另一名工程师从 Git 拉取旧版本继续编辑。

风险不只是“少一次 commit”，而是配置管理失去单一可信基线，后续下装可能覆盖现场修复。整改包括冻结变更、在线比较、导出并审查差异、确认现场运行版本、补录工单、建立下装后校验与签名标签流程。

## 17. 安全风险与攻击链

```text
开发者凭据/令牌失陷 -> 篡改仓库或工作流
-> 恶意修改协议工具/工程导出 -> 绕过不足的评审
-> 生成并发布受污染产物 -> 工程站使用
-> 配置/程序进入控制系统 -> 联锁与验证决定物理后果
```

提交记录只能证明仓库状态变化。确认工业影响还需验证制品来源、工程站实际文件、下装日志、PLC 在线程序和现场状态。

## 18. 防护与工程工作流

- 主分支保护、强认证、最小仓库和发布权限；
- 小而清晰的提交，PR 强制技术与安全评审；
- 自动检查格式、链接、测试、依赖和秘密；
- 关键提交/标签签名并保留平台审计；
- CI 使用最小令牌，隔离不可信代码和发布环境；
- 敏感证据与源代码分库存储，公开前脱敏复核；
- 工程变更关联工单、维护窗口、在线校验和回滚；
- 定期验证备份和仓库恢复，而非只依赖远程平台。

## 19. 隔离实验

### 实验一：对象和提交图

创建测试仓库，生成多个分支和合并，使用 `git cat-file`、`git ls-tree` 和图形日志观察 blob、tree、commit 与引用。不要在当前项目仓库练习历史重写。

### 实验二：差异与冲突

在两个测试分支修改同一配置，制造冲突并记录共同祖先、双方差异、解决依据和验证结果。对比文本与二进制文件的处理差异。

### 实验三：秘密泄露响应

只使用虚构令牌，模拟提交、发现、撤销、历史清理和预防扫描。重点记录为什么“删除当前文件”不足以完成处置。

### 实验四：工业变更包

使用虚构 PLC 点表建立一次变更：Issue、分支、提交、PR 检查、标签、导出文件哈希、模拟下装记录和回滚说明。不得使用真实客户项目。

## 20. 面试表达

### 60 分回答

Git 用对象和提交图保存版本，工作区、暂存区、本地提交和远端是不同状态。我提交前会检查未暂存与已暂存差异，用小提交和分支组织工作，通过 PR 评审与自动检查后合并。凭据和真实工控数据不进入仓库，`.gitignore` 也不能清除已经提交的历史。

### 90 分回答

我会把 Git 放进研究和工业变更证据链中。提交哈希标识对象内容，但作者时间可配置、历史可重写，所以关键变更还要结合签名、保护分支、评审、CI 审计和工单。PLC 工程文件需记录厂商工具与库版本，尽量导出可审查文本；Git 合并只证明仓库状态，现场影响必须通过下装日志、PLC 在线比较和功能验证确认。供应链方面还要限制 Action 与令牌权限、隔离不可信 PR、扫描秘密并将发布物哈希关联源提交。

### 常见追问

**`.gitignore` 能否保护已经提交的密码？** 不能。应立即撤销轮换，再评估和清理所有历史副本。

**commit hash 是否等于文件 SHA-256？** 不是。提交对象包含 tree、父提交和元数据，并使用仓库配置的对象哈希算法计算。

**为什么不把所有 PCAP 和固件放 Git？** 它们可能体积大、不可差分且包含敏感信息，应由受控证据/制品库管理，Git 保存哈希和元数据。

**Git 版本是否代表 PLC 在线版本？** 不代表。还需工程软件和 PLC 的在线比较、下装审计与现场验证。

## 本章总结

Git 提供内容寻址、提交图和协作基础，但可靠变更管理还需要身份、签名、评审、CI、工单和现场校验。成熟工控安全工程师能够区分代码历史、发布制品、工程项目和控制器在线状态，并避免让秘密与真实工业数据沿仓库数据流永久扩散。
