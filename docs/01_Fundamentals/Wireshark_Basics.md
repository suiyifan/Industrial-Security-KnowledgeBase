# Wireshark 与工业流量分析基础

Wireshark 的价值不在于“会写几个过滤器”，而在于把不可见的通信转化为可验证证据。在工控场景中，成熟的安全工程师应知道在哪里抓、抓到的流量代表什么、哪些报文可能遗漏，以及如何把网络行为映射到控制命令、设备状态和物理过程。

## 学习目标

- 选择不会干扰生产的抓包位置与采集方式；
- 区分捕获过滤器、显示过滤器和着色规则；
- 通过会话、时序、重传和协议字段定位问题；
- 分析 Modbus TCP 等工业协议的读写行为；
- 建立网络包、应用日志、PLC 变量和现场反馈之间的证据链；
- 导出可复核且不泄露敏感信息的分析结果。

## 1. 抓包原理与证据边界

网卡接收到帧后，抓包驱动复制符合条件的数据并附加时间戳，Wireshark 再依据协议解析器逐层解码。抓包是对某个观测点流量的记录，不是全网真相。

```text
线路流量 -> 网卡/采集设备 -> 抓包驱动 -> PCAP/PCAPNG -> 协议解析器
```

未出现在抓包中的报文可能根本没有经过该点，也可能被交换机镜像、采集网卡、操作系统缓冲区或写盘过程丢弃。反之，抓包看到坏校验和也不一定代表线路错误，可能是网卡校验和卸载导致抓包发生在硬件补全之前。

结论必须限定为“在该采集点、该时间范围内观察到……”。需要证明端到端行为时，应结合多个抓包点、交换机计数器、主机日志和设备状态。

## 2. 工业现场如何选择抓包点

### 2.1 交换机端口镜像

SPAN/端口镜像把指定端口或 VLAN 的流量复制到监测端口，部署方便，但镜像出口带宽不足时会丢包，也可能看不到交换机内部未经过镜像源的流量。应记录镜像源、方向、配置时间和监测口速率。

### 2.2 网络 TAP

TAP 在链路上复制流量，通常比临时镜像更稳定。聚合型 TAP 仍可能在双向流量总和超过输出带宽时丢包。关键证据采集要确认供电、故障旁路、链路速率与时间同步。

### 2.3 主机本地采集

在工程站或服务器本地抓包能观察该主机实际收发的数据，但安装驱动、占用磁盘或修改系统可能不符合生产变更要求。老旧 Windows、实时服务器和厂商保修设备尤其需要审批。

### 2.4 抓包位置与问题匹配

```text
工程站 --交换机 A-- 防火墙 --交换机 B-- PLC
            A 点                         B 点
```

A 点有 SYN、B 点没有，说明问题位于两点之间；两点都有 SYN 而 PLC 无回应，才继续检查 PLC 端口、返回路径或设备负载。不要在错误观测点反复猜测。

## 3. 安全采集与防护原则

- 生产环境优先采用被动复制，不使用 ARP 欺骗把流量引向分析主机；
- 采集前明确范围、时间、设备、责任人和存储位置；
- 评估镜像带宽、磁盘空间、CPU 和敏感数据暴露；
- 分段滚动保存，避免单个超大文件和磁盘耗尽；
- 统一时区与时间源，记录采集主机时钟偏差；
- 原始 PCAP 只读保存并计算哈希，分析在副本上进行；
- 分享前脱敏 IP、账号、工艺点名、配方和认证材料。

## 4. 界面与数据模型

Wireshark 的三个主要窗格分别是数据包列表、协议树和原始字节。成熟分析应在三者之间切换：列表用于时序，协议树用于字段，原始字节用于验证解析器是否正确。

常用列包括时间、源、目的、协议、长度和 Info。工业分析建议增加 TCP Stream、Modbus Transaction ID、功能码等自定义列，便于对请求与响应。

时间显示可选绝对时间、相对首包时间和相对前一包时间。事件关联用绝对时间，周期和延迟分析用相对时间。报告必须说明时区和时间基准。

## 5. 捕获过滤器与显示过滤器

捕获过滤器在写入文件前筛选数据，能降低存储与处理压力，但过滤掉的数据无法恢复。显示过滤器只改变界面展示，不修改原始文件，更适合探索分析。

```text
捕获过滤器（BPF）：host 192.168.10.20 and tcp port 502
显示过滤器：      ip.addr == 192.168.10.20 && tcp.port == 502
```

首次采集不宜设置过窄的捕获条件，否则可能遗漏 ARP、ICMP、DNS、NTP 或异常端口。已经明确目标且数据量巨大时，再使用经过验证的捕获过滤器。

## 6. 常用显示过滤器

### 6.1 地址、方向与端口

```text
ip.addr == 192.168.10.20
ip.src == 192.168.10.20
ip.dst == 192.168.10.20
tcp.port == 502
tcp.dstport == 102
udp.port == 123
```

`ip.addr` 同时匹配源和目的，适合查看资产全部通信；分析命令方向时应使用 `src`/`dst`。

### 6.2 TCP 状态与异常线索

```text
tcp.flags.syn == 1 && tcp.flags.ack == 0
tcp.flags.reset == 1
tcp.analysis.retransmission
tcp.analysis.duplicate_ack
tcp.analysis.zero_window
```

分析标记是 Wireshark 根据当前抓包推断的结果。镜像丢包会制造“重传”或“前序段未捕获”假象，应结合序列号、两端证据和交换机丢包计数判断。

### 6.3 常见基础协议

```text
arp
icmp
dns
http.request
tls.handshake
```

协议名称是否可用取决于 Wireshark 版本、端口和解码结果。非默认端口上的协议可使用“Decode As”临时指定，但报告要说明这是分析者假设，不是自动识别事实。

### 6.4 Modbus TCP

```text
modbus
modbus.func_code == 3
modbus.func_code == 16
```

不同版本的字段名称可能略有变化，应通过协议树自动补全确认，不能机械复制过滤器后忽略“无结果”。

## 7. 建立基线：先回答谁与谁通信

分析未知 PCAP 时不要立刻搜索“攻击特征”，先建立环境基线：

1. 查看捕获时长、包数、平均速率和是否存在明显丢包；
2. 用 Endpoints 统计资产，用 Conversations 统计会话；
3. 根据 MAC 厂商、IP、端口、周期和载荷推断资产角色；
4. 区分周期采集、事件上报、操作员请求、工程操作和基础服务；
5. 标记首次出现、通信量突变、非预期方向和新协议。

统计结果是线索。例如通信量最高的设备可能是历史采集服务器，也可能是环路或扫描源，需要结合拓扑和业务确认。

## 8. TCP 会话分析

### 8.1 握手与失败模式

观察 SYN、SYN/ACK、ACK 是否完整，并检查握手耗时。反复 SYN 无回应通常指向过滤、丢包或不可达；立即 RST 通常表明端口未监听或被主动拒绝；握手完成后无应用数据则要检查客户端逻辑和服务状态。

### 8.2 重组与 Follow Stream

TCP 是字节流，一个应用消息可能跨多个段。Follow TCP Stream 可重组双向内容，适合查看 HTTP 或未知协议会话，但必须注意：

- 内容视图可能隐藏原始分段与时间关系；
- TLS 加密流量不能凭普通抓包直接读取明文；
- 导出的流可能包含口令、令牌、配方或工程数据；
- 应回到原始包验证关键字段和响应时序。

### 8.3 延迟、重传与窗口

使用 RTT、重传、重复 ACK、乱序和窗口变化判断性能。零窗口表示接收端暂时无法继续接收，常与应用阻塞或资源不足有关。网络异常必须与服务器 CPU、磁盘、应用队列和交换机端口计数共同分析。

## 9. 工业协议分析方法

### 9.1 请求与响应配对

工业协议常具有事务号、序列号或会话标识。分析时要配对请求与响应，确认：发起者、操作类型、目标对象、参数值、响应结果和往返延迟。

### 9.2 Modbus TCP 示例

```text
以太网 -> IPv4 -> TCP -> MBAP -> PDU
                         |       |
                         |       +-> 功能码、地址、数量、数据
                         +-> 事务标识、协议标识、长度、单元标识
```

看到功能码 03 可描述为读取保持寄存器，但不能立即知道寄存器代表液位还是设定值。语义需要结合点表、HMI 标签、控制逻辑或隔离实验确认。功能码 05、06、15、16 等写操作应结合来源资产、维护窗口、地址和值判断是否异常。

异常响应说明服务端拒绝或无法完成请求，不等于写入已经生效。即便协议响应成功，也要继续核对 PLC 变量、输出和现场反馈。

### 9.3 周期与状态变化

正常 SCADA 轮询常呈稳定周期。新客户端、轮询频率突变、非常用写功能、地址范围扩张和大量异常响应是值得调查的偏差，但也可能由工程调试、主备切换或设备恢复引起。检测规则必须结合工单和运行状态。

## 10. 数据流、控制流与状态流映射

### 数据流

从协议字段识别变量如何从 PLC 进入 SCADA、历史库和报表。记录编码、字节序、缩放、工程单位和质量状态，避免只看到十六进制值。

### 控制流

区分谁主动发起连接、谁发出读取或写入、谁确认执行。工程站下载、HMI 操作和周期采集具有不同的控制权限和审计要求。

### 状态流

```text
TCP 已连接 -> 协议会话成立 -> 命令被接受 -> PLC 变量改变
           -> 输出逻辑允许 -> 执行器动作 -> 现场反馈变化
```

PCAP 通常只能证明前几步。后续状态需要 PLC 在线记录、SCADA 事件、控制逻辑和现场反馈补证。报告中必须区分已观察事实和推断。

## 11. 安全风险与攻击链分析

流量分析可发现资产侦察、端口扫描、未授权连接、异常写入、重放迹象、会话中断和数据外传。完整攻击链不能只靠单个包判断，应按时间线关联：

```text
新资产出现 -> 多端口探测 -> 工业协议建连 -> 地址枚举
           -> 非维护时段写操作 -> 过程变量异常 -> 报警/联锁响应
```

每一步都应寻找反证：新资产可能是更换的工程站，地址枚举可能是正常驱动初始化，写操作可能有工单。成熟分析以证据降低不确定性，而非迎合预设结论。

## 12. 工业案例：异常泵启停调查

某站点报告夜间泵组短时启动。监控口 PCAP 显示，非日常 HMI 地址向 PLC 建立 Modbus TCP 连接并发送单寄存器写请求，PLC 返回正常响应。随后周期读取显示相关命令变量变化，但运行反馈寄存器未变化。

进一步检查发现 PLC 联锁因低液位阻止启动。网络证据证明异常客户端发出了被协议接受的写请求，却不能证明泵实际动作。最终风险结论为“未授权控制尝试，控制逻辑屏障有效”，并推动封禁异常路径、轮换远程凭据、核查工程站和加强写命令监测。

## 13. 证据导出与报告

建议保留：原始 PCAPNG、SHA-256、采集说明、显示过滤器、关键包编号、流导出副本、截图和分析笔记。截图应显示时间、源、目的、协议字段和上下文，不能只截取高亮的一行。

报告描述示例：

```text
2026-07-13 02:14:31.245（UTC+8），在 PLC 接入交换机镜像口观察到
10.10.5.23:49122 向 10.10.8.15:502 建立 TCP 会话。第 1842 包为
Modbus 写单寄存器请求，目标地址与值见附件；第 1843 包返回正常响应。
本抓包未包含执行器反馈，不能单独证明现场设备动作。
```

## 14. 隔离实验

1. 在本机访问测试 HTTP 服务，观察 DNS、TCP 握手、请求、响应和关闭。
2. 停止服务后再次连接，对比无响应与 RST 的差异。
3. 在 OpenPLC 或其他隔离仿真中产生 Modbus 读取，配对事务号和响应。
4. 修改虚拟测点，验证报文值、HMI 标签和模拟过程状态的对应关系。
5. 设置窄捕获过滤器和宽捕获过滤器，对比遗漏的 ARP、ICMP、DNS 或 NTP。
6. 对原始 PCAP 计算哈希，并制作包含包号和过滤器的简短报告。

实验不得连接生产控制器，不实施 ARP 欺骗、批量写入、畸形报文或压力测试。

## 15. 面试表达

### 60 分回答

我使用 Wireshark 时会先确认抓包点和采集范围，再通过 Endpoints、Conversations 建立资产与会话基线。之后按 Ethernet、IP、TCP/UDP 和应用协议逐层分析，用请求响应、重传、RST 和时序定位异常。工业协议还要看功能码、地址、值与响应，而不能只凭端口判断。

### 90 分回答

我会先说明证据边界：镜像口抓包只代表该观测点，可能受镜像带宽、网卡卸载和时间偏差影响。分析时先建立资产、方向、周期和五元组基线，再配对协议请求与响应，把写操作映射到点表、PLC 变量、输出逻辑和现场反馈。例如 Modbus 写请求返回成功只证明协议层接受，不能直接证明泵动作；还要核对联锁和运行反馈。生产环境优先被动采集，原始 PCAP 只读保存并计算哈希，报告区分事实、推断和待验证项。

### 常见追问

**抓包里没有报文，是否说明设备没发送？** 不一定，可能是抓包点错误、镜像配置遗漏或采集丢包。

**为什么显示 TCP 重传但线路正常？** 镜像丢包、网卡卸载、乱序或接收端处理阻塞都可能导致该标记，需要多源验证。

**如何判断 Modbus 写入是否影响现场？** 配对响应后继续核对地址语义、PLC 变量、控制逻辑、输出与反馈，并确认联锁状态。

**能否直接在生产网用主动方式验证？** 默认不能。应优先使用已有流量或隔离复现，主动测试必须经过书面授权和影响评估。

## 本章总结

Wireshark 是证据工具，不是自动结论生成器。可靠分析由正确观测点、完整采集、分层解码、时序关联、协议语义和过程验证共同构成。工控安全工程师最终要回答的不是“这个包是什么”，而是“谁在什么时间对什么对象做了什么，系统如何响应，现场是否真的发生变化”。
