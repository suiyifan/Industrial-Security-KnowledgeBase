# 日志、可观测性与数字取证基础

安全事件不会自动形成完整故事。操作系统日志、应用日志、网络流量、数据库审计、PLC 事件和过程趋势分别观察系统的一部分，而且时间、格式和保留周期不同。成熟工控安全工程师需要明确每个数据源能证明什么、可能遗漏什么，并在不破坏生产与证据的前提下建立可复核时间线。

## 学习目标

- 区分日志、指标、追踪、事件、审计和网络流量；
- 设计包含主体、动作、对象、结果和关联标识的事件模型；
- 理解时间源、时区、漂移、记录时间和采集时间；
- 构建采集、缓冲、传输、解析、存储和告警链；
- 关联 Windows、Linux、应用、数据库、网络和工控数据；
- 理解数字证据完整性、易失性、哈希和保管链；
- 完成主机与工业事件的分层时间线；
- 区分技术活动、控制系统状态与物理结果。

## 1. 可观测性的工作原理

可观测性是根据系统外部输出推断内部状态的能力。常见三类遥测：

- 日志：离散事件和上下文；
- 指标：随时间变化的数值，如 CPU、连接数、周期延迟；
- 追踪：一次请求跨组件的因果路径。

安全还使用 PCAP、文件哈希、内存、进程、配置快照和工业过程数据。数据越多不等于可观测性越好；缺少统一时间、身份、资产和关联 ID 时，大量日志仍无法回答问题。

```text
事件源 -> 本地缓冲/Agent -> 传输 -> 解析/规范化
       -> 存储/索引 -> 规则/关联 -> 告警 -> 调查/响应
```

链路每一步都可能丢失、延迟、重复或篡改数据，因此监测系统本身也需要健康监测。

## 2. 日志、审计和业务事件

普通运行日志帮助诊断，审计日志用于记录安全相关活动，业务事件描述应用状态变化。三者可能重叠，但目的不同。

一个高质量事件至少考虑：

```text
何时：事件时间、时区、精度
何处：主机、服务、站点、资产标识
谁：用户、设备、服务、会话
做什么：动作、目标对象、参数摘要
结果：成功/失败、错误码、状态变化
关联：请求 ID、事务 ID、工单、父事件
来源：提供程序、日志通道、版本
```

不要记录密码、私钥、完整令牌和不必要的配方数据。敏感字段可使用受控脱敏或标识符，但必须保留调查需要的关联能力。

## 3. 事件时间与时间可信度

至少区分：

- Event Time：事件实际发生时间；
- Record Time：源系统写入日志时间；
- Receive Time：采集器收到时间；
- Ingest Time：平台解析入库时间；
- Analysis Time：分析者处理时间。

网络中断和本地缓冲会让 Receive/Ingest 晚于 Event。若平台按接收时间排序，攻击链可能错序。

### 3.1 时区、UTC 与夏令时

跨系统存储优先使用 UTC，并保留源时区。报告面向现场人员时可转换本地时间，但需明确 `UTC+8` 等偏移。只写“02:15”没有取证意义。

### 3.2 时钟漂移与跳变

NTP/PTP 同步异常、虚拟机快照恢复和手工改时会产生漂移或回拨。调查前记录各系统相对可信时间源的偏差，必要时对时间线做校正，同时保留原始时间。

PLC、SOE 和保护设备可能使用不同精度。先后关系只有在误差范围小于事件间隔时才可靠。

## 4. 日志格式与结构化

纯文本便于阅读但难稳定解析；JSON、CEF、LEEF、syslog 等可携带结构化字段。结构化不代表语义统一，同名字段在不同产品中含义可能不同。

推荐字段：

```json
{
  "timestamp": "2026-07-14T08:15:31.245+08:00",
  "asset_id": "lab-gateway-01",
  "event_type": "config_change",
  "actor": "engineer-a",
  "object": "modbus-map-v3",
  "result": "success",
  "correlation_id": "CHG-2026-001",
  "source": "gateway-admin"
}
```

字段定义、枚举和版本需文档化。解析失败的原始事件不能静默丢弃，应进入隔离队列并告警。

## 5. 采集、缓冲与传输

采集方式包括本地 Agent、syslog、Windows Event Forwarding、API、文件读取、数据库订阅和网络镜像。

设计要点：

- 断网时本地缓冲容量和溢出策略；
- 至少一次/至多一次传输带来的重复或丢失；
- TLS 和双向身份认证；
- Agent 权限和自身日志；
- 背压、速率和带宽；
- 日志轮转、断点续传和源文件标识；
- 采集配置变更审计。

工控区带宽和设备资源有限，不能因日志采集拖慢控制通信。优先在服务器、网关和监测节点集中采集，老旧控制器采用被动或厂商支持方式。

## 6. 存储、保留与完整性

日志平台应根据事件价值、法规、存储成本和调查周期制定保留。关键工程操作与安全事件保留期通常应覆盖维护、审计和事故发现窗口。

保护措施包括：

- 写入账户与查询账户分离；
- 源主机外独立存储；
- 传输与静态加密；
- 防篡改/不可变存储和访问审计；
- 容量、索引、保留任务监测；
- 备份与恢复验证。

日志发送到 SIEM 不代表源日志可删除。原始格式、解析后字段和证据导出各有价值，应明确权威副本。

## 7. Windows 数据源

常见来源：

- Security、System、Application 日志；
- PowerShell、Task Scheduler、RDP、SMB 等专用通道；
- Defender/EDR、Sysmon（若部署）；
- 服务、注册表、文件系统和 WMI 遥测；
- 工程软件与厂商应用日志。

事件 ID 必须结合 Provider、通道、版本和 XML 字段解释。安全日志质量依赖审计策略；日志为空可能表示未启用、轮转覆盖、服务异常或被清理。

```powershell
Get-WinEvent -FilterHashtable @{
  LogName='System'
  StartTime=(Get-Date).AddHours(-1)
}
```

导出时保留 EVTX 或原始 XML，不只截屏消息文本。

## 8. Linux 数据源

常见来源：

- systemd journal 与内核日志；
- syslog/rsyslog 文件；
- auditd；
- SSH、sudo、PAM 与账户日志；
- systemd、cron、容器运行时；
- 应用、Web、数据库和厂商服务日志。

```bash
journalctl -o json --since '2026-07-14 08:00:00'
ausearch --start today
```

journal 可能只保存在内存，重启后丢失；传统日志可能轮转压缩。调查前确认实际配置和保留范围。

## 9. 网络与安全设备数据源

PCAP 提供包级字段和载荷（若未加密且合法可见），NetFlow/IPFIX 提供会话摘要，防火墙/IDS/VPN/堡垒机提供策略与身份上下文。

```text
PCAP：细节强、存储大、观察点局限
Flow：覆盖广、载荷少
防火墙：允许/拒绝与规则
VPN/堡垒机：用户、会话、目标和审批
```

NAT、代理和跳板会改变源地址。关联应使用转换日志、会话 ID、账户、证书、时间和目标，不能只靠 IP。

## 10. 应用与 Web 数据源

Web 访问日志通常记录方法、路径、状态、大小、User-Agent 和来源。反向代理与应用应使用关联 ID，把请求继续连接到身份、数据库和发布任务。

安全日志应记录认证结果、对象授权、配置变更、旧新版本、审批、导出和高风险管理操作。不要只记录“请求 200”，因为业务可能异步执行或失败。

应用堆栈、SQL 和内部路径应进入受控日志，不直接返回客户端。日志中的用户输入需要转义，防止日志注入和伪造行。

## 11. 数据库与数据平台数据源

数据库可提供登录、权限、结构变更、关键表写入、慢查询、锁、备份恢复和复制状态。应用最终用户与共享数据库账户之间要通过可信上下文关联。

历史库还应监测采集缺口、质量码、压缩、队列、时间偏差和归档延迟。趋势缺失不一定代表 PLC 没有产生数据，可能是采集、队列、数据库或查询层故障。

## 12. 工业控制数据源

### 12.1 PLC、DCS 与工程软件

可能记录：运行模式、程序下载、在线修改、用户登录、配置变化、诊断缓冲区、通信连接和固件事件。不同产品能力差异大，日志容量可能很小且循环覆盖。

### 12.2 SCADA/HMI

记录操作员登录、画面操作、报警产生/确认、设定值修改、配方、脚本和通信质量。报警确认表示人员处理状态，不等于过程变量恢复正常。

### 12.3 SOE 与过程趋势

Sequence of Events 提供高精度状态变化，趋势提供连续过程值。它们能验证命令之后是否有输出与反馈变化，但受时间同步、采样周期、死区、压缩和质量码影响。

### 12.4 SIS 与安全屏障

SIS 事件、联锁触发和旁路记录对物理后果分析关键。应保持与基本控制系统的独立性和受控访问，不能为了集中日志破坏安全架构。

## 13. 资产、身份与上下文规范化

同一设备可能有 IP、主机名、资产编号、序列号和证书主体。DHCP、NAT 和更换网卡会改变网络标识。SIEM 需要时间有效的资产映射，而非永久将一个 IP 绑定一台设备。

身份也可能出现用户名、SID/UID、邮箱、证书和服务主体。规范化时保留原值，并通过目录与资产系统补充角色、站点、所有者和关键性。

## 14. 关联与因果关系

时间接近不等于因果。事件关联应寻找共同会话、事务、工单、对象和状态变化：

```text
VPN 会话 ID -> 堡垒机会话 -> RDP 登录
-> 工程软件进程 -> PLC 下载事务 -> PLC 程序版本
-> 输出变化 -> 现场反馈/SIS
```

只有时间相近时，应表述为“相关线索”而非“证明导致”。关联 ID 应由可信组件生成并防止客户端任意伪造。

## 15. 检测工程与告警质量

检测规则应包含：威胁假设、数据需求、逻辑、基线、例外、严重性、调查步骤和测试样本。

工业检测示例：

- 新工程站与 PLC 建立会话；
- 非维护时段程序下载或配置发布；
- 写命令地址/值超出批准范围；
- 远程账户登录后出现工程软件和文件传输；
- SIS 旁路与无工单设定值变化；
- 关键日志源中断或时间漂移。

告警要区分工具置信度和业务影响。规则命中不是事故结论，需调查正常维护、主备切换和初始化行为。

## 16. 日志源健康与监测盲区

必须监测：最后事件时间、事件速率、解析失败、队列、时间偏差、磁盘、证书、Agent 状态和配置版本。

攻击者可能关闭日志，也可能通过制造海量事件掩盖活动。设备故障和网络中断同样会造成沉默。日志源停止本身应告警，但响应要结合资产角色和过程状态。

## 17. 数字取证基本原则

数字取证目标是以可解释、可复核方式识别、采集、保存、分析和报告数字证据。核心原则：

- 合法授权与明确范围；
- 尽量减少对原始状态的改变；
- 记录每个动作、工具、时间和人员；
- 原件与分析副本分离；
- 使用哈希检测内容变化；
- 保管链记录证据流转；
- 结论与证据强度匹配。

哈希不自动证明采集过程正确，只能支持特定文件内容是否变化。工具、时间源和采集者仍需可信。

## 18. 易失性与采集顺序

内存、网络连接、进程、登录会话和临时文件比磁盘镜像更易失。传统“易失性顺序”只是指导，工控现场必须先考虑人员和过程安全。

```text
过程安全/运行状态
-> 时间与现场记录
-> 网络、会话、进程、内存等易失状态
-> 日志、配置、文件与磁盘
-> 备份、云/集中平台和外部系统
```

如果内存采集会使关键 HMI 卡顿，就不能机械执行。可改用冗余节点、厂商支持采集、网络侧证据或维护窗口。

## 19. 实时采集与停机采集

实时采集保持系统运行，可获得内存和连接，但会改变系统并使用资源；停机/离线采集更适合完整磁盘，但会丢失易失状态并影响生产。

选择依据：资产角色、过程状态、冗余、法律/合同要求、证据价值、工具兼容和风险。不要在未知工业主机直接运行大规模采集程序。

## 20. 磁盘、文件与内存证据

### 20.1 文件系统

文件证据包括内容、路径、元数据、ACL、签名、哈希、链接和删除状态。时间戳可被修改，也会被正常程序和取证操作影响。

### 20.2 磁盘镜像

物理镜像覆盖可寻址介质，逻辑采集只获取文件/对象。加密、RAID、虚拟磁盘、专用文件系统和数据库一致性会影响方法。

### 20.3 内存

内存可能包含进程、模块、连接、解密后数据和临时秘密，价值高且敏感。采集工具需要内核级访问并可能影响稳定性。分析结论受工具支持、符号和系统版本约束。

## 21. 证据保管链与清单

证据清单示例：

```text
Evidence ID: EV-2026-001
Source: historian-01 / System log
Collected at: 2026-07-14T09:10:00+08:00
Collector: analyst-a
Method/tool/version: ...
Original hash: SHA-256 ...
Storage: controlled-evidence-store/...
Actions: copied read-only, analysis copy created
Notes: source clock +2.4 s vs trusted NTP
```

每次转移、解封、复制和分析都记录。公开报告引用脱敏副本，不暴露客户和过程敏感数据。

## 22. 时间线分析

时间线可分为：

- 文件系统时间线；
- 登录/身份时间线；
- 进程与服务时间线；
- 网络会话时间线；
- 应用/数据库时间线；
- 工程操作与过程时间线。

先保留各源原始时间，再校正到共同基准。每个事件记录来源、置信度和误差。重复事件需去重但不能丢失多源验证关系。

## 23. 数据流、控制流与状态流

### 数据流

遥测从源产生，经采集、传输、解析和存储进入调查平台；证据副本再进入分析和报告。每次转换都要保留来源和完整性。

### 控制流

规则触发告警，分析者验证并启动响应；工程操作通过审批、应用和协议影响 PLC。日志应覆盖关键控制点。

### 状态流

```text
账户登录 -> 会话建立 -> 工程软件启动 -> 配置修改
-> 发布/下载 -> PLC 变量/程序变化 -> 输出 -> 现场反馈/SIS
```

日志通常只能证明部分状态。没有现场反馈时，不得直接宣布物理动作发生。

## 24. 工控场景案例：疑似设定值篡改

SOC 告警显示工程站在夜间向 PLC 发出写操作。VPN 与堡垒机日志显示一名值班工程师有批准会话；工程软件日志记录设定值从 60 改为 65；PLC 事件记录写入成功；趋势显示设定值变化，但过程值未越界，SIS 未动作。工单中批准范围是 60–62。

结论为真实身份执行了超出批准范围的变更，而非凭据必然失陷。证据链覆盖会话、应用、协议和 PLC 状态；物理后果有限但流程完整性失效。响应包括恢复批准值、核查原因、收紧范围校验、增加双人确认和实时偏差告警。

## 25. 安全风险与攻击链

```text
攻击者获得身份/主机权限 -> 执行工程操作
-> 同时关闭/篡改本地日志 -> 配置进入 PLC
-> 伪造 HMI/历史显示 -> 延迟操作员发现
-> 过程变化 -> 联锁/SIS 响应
```

纵深证据要求日志跨主机、网络和安全系统独立保存。若所有数据都来自被攻陷工程站，就不能高置信证明完整攻击链。

## 26. 防护原则

- 为每类关键操作定义必须记录的身份、对象、结果和关联 ID；
- 统一资产、身份、时间和字段规范，同时保留原始事件；
- 日志加密传输并存入独立、受控、可恢复平台；
- 监测采集链、解析失败、时间漂移和日志源沉默；
- 关键工程、发布、旁路和权限事件高优先级审计；
- 用工单、设备模式和过程状态丰富检测上下文；
- 预先制定低影响采集、隔离和恢复行动卡；
- 原始证据只读、哈希、清单和保管链完整；
- 结论区分事实、推断、置信度和待验证项；
- 定期用桌面演练和技术测试验证日志真的可用。

## 27. 隔离实验

### 实验一：统一时间与字段

在 Windows、Linux 和测试 Web 服务生成登录、进程和请求事件，配置本地 NTP，输出为结构化格式并关联同一请求 ID。人为制造一个时间偏差并校正。

### 实验二：日志采集链故障

模拟 Agent 停止、队列满、解析错误和网络中断，验证平台能否发现“没有日志”以及恢复后如何处理延迟与重复事件。

### 实验三：跨层时间线

在隔离 OpenPLC/协议环境执行一项批准的虚拟设定值变更，关联用户会话、应用日志、PCAP、PLC 变量和过程模型反馈。

### 实验四：证据保全

导出测试日志与 PCAP，计算哈希、创建证据清单和分析副本。修改副本后验证哈希差异，并记录所有操作。

## 28. 面试表达

### 60 分回答

可观测性主要由日志、指标和追踪组成，安全调查还需要 PCAP、主机和工业过程数据。我会统一时间、资产、身份和关联 ID，保留原始日志并监测采集链健康。数字证据采用只读原件、分析副本、哈希和保管链，结论只覆盖证据实际证明的层级。

### 90 分回答

我会建立从身份到物理过程的跨层时间线：VPN/堡垒机会话、主机登录和进程、工程软件操作、工业协议事务、PLC 事件、趋势反馈和 SIS 状态。先校准各源时间和误差，保留 Event/Receive/Ingest Time，再用会话、事务、工单和对象建立因果关联。采集设计还要监测日志源沉默、解析失败和队列，避免把没有事件误当没有攻击。工控取证优先安全生产，易失数据价值必须与采集对关键主机的影响权衡。

### 常见追问

**日志里没有记录是否证明事件没发生？** 不能。可能未启用、覆盖、丢失、解析失败、时钟错误或被清理。

**文件哈希一致能证明什么？** 在算法和采集可信前提下支持内容一致，不能单独证明来源和采集过程正确。

**时间接近是否说明两个事件有因果关系？** 不足。还需会话、事务、主体、对象和状态变化等证据。

**为什么工控取证不能照搬普通主机流程？** 采集、隔离和关机可能影响控制、监控、冗余和人员安全，必须先评估过程状态。

## 本章总结

日志与取证的价值来自覆盖、时间、上下文和完整性，而不是事件数量。成熟工控安全工程师会把身份、主机、网络、应用、数据库、PLC 和现场过程放入同一证据链，监测遥测系统本身，并在安全生产约束下选择采集与响应方式。最终报告明确事实、推断和证据边界，经得起他人复核。
