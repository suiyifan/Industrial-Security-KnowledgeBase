# C 语言、程序内存与内存安全

C 语言允许程序直接操作地址、内存布局和硬件接口，因此广泛存在于操作系统、驱动、嵌入式固件、实时运行时和工业协议栈中。这种可控性带来高性能与确定性，也把边界检查、对象生命周期和整数范围交给开发者。工控漏洞研究中的缓冲区溢出、释放后使用、整数溢出和格式化字符串问题，都建立在本章的程序模型之上。

## 学习目标

- 理解从源代码到目标文件、可执行文件和运行进程的过程；
- 掌握类型、指针、数组、结构体、函数和动态内存；
- 区分代码段、数据段、堆、栈和映射区；
- 解释 ABI、调用约定、对齐、字节序和链接；
- 识别越界、UAF、双重释放、整数和格式化字符串缺陷；
- 理解编译器、操作系统与硬件缓解机制的边界；
- 将内存缺陷映射到工业服务失效和物理过程影响。

## 1. 为什么工控安全工程师要学习 C

大量工业设备采用 C/C++ 开发：PLC 通信服务、HMI 组件、协议网关、RTU 固件、串口驱动和升级程序都可能包含原生代码。即使研究人员不负责开发控制程序，也需要阅读补丁、理解反汇编、定位崩溃并判断漏洞影响。

C 的核心特征是“抽象较薄”：整数位宽、结构体布局、指针算术、分配与释放会直接影响机器内存。它不会自动为数组访问做统一边界检查，也不会自动管理所有对象生命周期。

## 2. 编译链接原理：从源代码到运行进程

典型构建链包括预处理、编译、汇编和链接：

```text
.c 源文件
  -> 预处理：宏、条件编译、头文件
  -> 编译：语义分析、优化、生成汇编
  -> 汇编：生成目标文件
  -> 链接：解析符号与重定位，生成程序/库
  -> 装载：操作系统映射代码、数据和依赖
```

预处理宏只是文本替换，参数可能被重复求值；条件编译会让不同固件版本包含不同功能。优化会改变指令顺序、删除变量甚至移除被认为“不可能”的检查，逆向时不能假定源码结构与机器码一一对应。

### 2.1 静态库、动态库与符号

静态链接把所需代码合入最终程序，动态链接在运行时装载共享库。动态库便于修复和复用，但存在搜索路径、版本兼容和库劫持风险。分析工业软件时应记录程序哈希、库版本、导入符号和实际加载路径。

符号表把函数/变量名称与地址关联。发布固件常剥离符号，研究人员需通过字符串、调用关系、常量和行为恢复语义。

## 3. 类型、位宽与整数转换

C 标准没有规定 `int` 在所有平台上固定为 32 位。协议代码应使用 `uint8_t`、`uint16_t`、`uint32_t` 等明确位宽类型，并在转换前验证范围。

```c
#include <stdint.h>

uint16_t register_address;
int16_t signed_measurement;
```

有符号与无符号混合运算可能把负数转换为巨大正数；宽类型转窄类型可能截断；长度乘法可能溢出。这些问题常出现在“数量 × 单项长度”“头部长度 + 载荷长度”和地址计算中。

## 4. 指针、地址与数组

指针保存对象地址，指针类型告诉编译器如何解释目标对象及指针算术步长。

```c
uint16_t values[4] = {10, 20, 30, 40};
uint16_t *p = values;
uint16_t second = *(p + 1);
```

数组表达式在许多上下文会退化为首元素指针，但数组与指针不是同一类型。把数组传入函数后，函数通常无法自动得知元素数量，因此接口必须同时传递长度。

### 4.1 空指针、悬空指针与野指针

- 空指针：明确不指向对象；解引用会产生错误；
- 悬空指针：曾指向有效对象，但对象生命周期已结束；
- 未初始化指针：值不确定，可能指向任意地址；
- 越界指针：超出允许对象范围。

检查非空只能排除一种错误，不能证明地址有效、类型正确或对象仍存活。

## 5. 进程内存布局

典型原生进程可抽象为：

```text
高地址
栈：调用帧、返回状态、局部对象
映射区：共享库、映射文件
堆：动态分配对象
BSS：未显式初始化全局/静态对象
数据段：已初始化全局/静态对象
代码段：机器指令和只读常量
低地址
```

布局受架构、操作系统、编译器、链接器和随机化影响。不能把示意图当成固定地址表。

### 5.1 栈与调用帧

函数调用通常保存返回地址、旧栈帧信息、寄存器和局部对象。具体参数传递由 ABI/调用约定决定，可能优先使用寄存器，再使用栈。

栈缓冲区越界可能破坏相邻局部变量、保存的寄存器或控制数据。现代编译器会加入栈保护，但根因仍是写入超过对象边界。

### 5.2 堆与动态分配

`malloc` 分配未初始化内存，`calloc` 分配并清零，`realloc` 可能移动对象，`free` 结束对象生命周期。

```c
uint8_t *buffer = malloc(length);
if (buffer == NULL) {
    return ERROR_NO_MEMORY;
}
/* 使用 buffer */
free(buffer);
buffer = NULL;
```

把一个指针设为 `NULL` 不能清除其他别名。若其他指针仍引用已释放对象，仍可能发生释放后使用。

## 6. 结构体、联合体、对齐与 ABI

结构体按成员顺序存放，但编译器可能插入填充以满足对齐。不同平台或编译选项下 `sizeof(struct)` 可能不同。

```c
struct Record {
    uint8_t type;
    uint32_t timestamp;
    uint16_t value;
};
```

不能把未经验证的网络缓冲区直接强制转换为结构体指针：字节序、对齐、填充、长度和未对齐访问都可能出错。可靠解析器逐字段读取并转换。

联合体让多个成员共享同一内存，适合表达不同视图，但读取非当前有效成员存在可移植性和语义风险。位域布局也由实现定义，不适合直接映射跨平台协议。

## 7. 字符串与长度

C 字符串是以空字节终止的字符数组。若外部输入没有终止符，依赖 `strlen` 的代码可能继续读取边界之外。字符串长度、缓冲区容量和编码后字节数是不同概念。

安全接口设计应显式传递容量和实际长度。复制、拼接与格式化都要验证目标空间，避免只把一个危险函数替换成另一个而不修正长度模型。

```c
int parse_name(const uint8_t *input, size_t input_len,
               char *output, size_t output_capacity);
```

接口必须规定是否追加终止符、截断是否允许、失败后输出状态和编码规则。

## 8. 函数指针与控制流

函数指针用于回调、驱动接口和协议分派表：

```c
typedef int (*handler_fn)(const uint8_t *, size_t);
```

若内存破坏覆盖函数指针、返回地址或虚表等控制数据，程序可能偏离预期控制流。现代控制流完整性机制尝试限制间接跳转目标，但覆盖业务数据同样可能造成严重后果，例如修改权限标志、长度或控制参数。

## 9. 未定义行为

未定义行为意味着标准不约束结果，编译器可假定它不会发生。常见例子包括有符号整数溢出、越界访问、使用已结束生命周期对象、无效移位和数据竞争。

未定义行为可能在调试构建中“看似正常”，在优化、不同编译器或不同架构下崩溃。安全修复不能以“现场暂未复现”为依据，应消除违反语言规则的根因。

## 10. 常见内存安全缺陷

### 10.1 越界读写

长度、索引或终止条件错误会读写对象边界之外。越界读可泄露进程数据，越界写可破坏业务状态、分配器元数据或控制流。

### 10.2 释放后使用与双重释放

对象释放后仍被访问是 UAF；同一对象被重复释放可能破坏内存管理器状态。异步回调、错误路径和复杂所有权关系是高发位置。

### 10.3 未初始化内存

读取未初始化对象会得到不确定数据，可能泄露旧内存内容或造成不稳定分支。所有输出字段和结构体填充都应明确初始化。

### 10.4 格式化字符串

格式字符串控制参数解释方式。外部输入不能直接充当格式：

```c
printf("%s", untrusted_text);  /* 数据作为数据 */
```

日志接口同样要注意。格式与参数不匹配还可能造成崩溃或信息泄露。

### 10.5 整数与长度错误

即使最终内存复制函数接收 `size_t`，上游数量乘法也可能已经溢出。应在运算前检查边界，并保证协议允许范围、平台范围和业务范围同时满足。

## 11. 资源与错误路径

安全缺陷不只来自内存。文件描述符、锁、Socket 和临时文件也有生命周期。多处 `return` 容易遗漏清理，可以使用统一清理路径，但必须避免重复释放：

```c
int result = ERROR;
uint8_t *buffer = NULL;

buffer = malloc(length);
if (buffer == NULL) goto cleanup;

/* 处理 */
result = SUCCESS;

cleanup:
free(buffer);
return result;
```

错误码应保留层级信息，不能把解析失败、超时和权限拒绝全部变成同一个返回值。

## 12. 并发与数据竞争

多个线程在缺少同步时访问共享对象，且至少一个写入，会产生数据竞争。锁保护共享状态，原子操作适合有限状态，消息队列可减少共享内存。

错误的锁顺序可能死锁，锁范围过大可能拖慢实时任务。协议会话表、设备状态缓存和配置热更新都要明确定义所有权与同步策略。

## 13. 编译与运行时防护

常见缓解包括：

- 栈保护：检测部分栈数据破坏；
- ASLR：随机化代码、库、堆和栈位置；
- NX/DEP：阻止数据页直接执行；
- PIE：使主程序地址可随机化；
- RELRO：强化部分动态链接结构；
- FORTIFY：在可推断长度时增加检查；
- 控制流保护：限制间接分支；
- AddressSanitizer/UBSan：测试中发现内存和未定义行为。

缓解机制降低利用成功率或帮助发现缺陷，不替代边界检查、所有权设计和根因修复。嵌入式设备可能因资源或工具链限制缺少部分机制，风险评估需验证实际构建配置。

## 14. 协议解析器的安全设计

解析外部报文应遵循：

1. 先验证固定头是否完整；
2. 使用安全算术计算总长度；
3. 限制数量、递归深度和最大报文；
4. 每次读取前检查剩余长度；
5. 显式处理字节序和符号；
6. 将语法解析与业务执行分离；
7. 未知字段和状态采用安全失败；
8. 通过协议向量、边界测试和模糊测试验证。

```text
原始 bytes -> 长度边界 -> 字段解码 -> 语义校验
           -> 权限/状态校验 -> 业务对象 -> 控制或查询
```

解析成功只说明格式可接受，不代表当前会话有权限执行写命令。

## 15. 数据流、控制流与状态流

### 数据流

网络字节进入接收缓冲区，经长度计算、结构解析和类型转换形成业务变量。任何不安全转换都可能污染下游状态。

### 控制流

功能码或消息类型选择处理函数，权限和协议状态决定是否执行。内存破坏可能改变分支条件、函数指针或返回路径。

### 状态流

连接、认证、协议会话、设备模式、PLC 变量和现场反馈是不同状态。即使解析器被破坏，也要继续确认服务权限与下游控制路径。

## 16. 工控场景：网关解析服务崩溃

某工业网关处理畸形长度字段后服务崩溃并由看门狗重启。根因是数量字段乘以记录大小时发生整数截断，分配缓冲区小于后续复制长度，造成堆越界写。

软件后果是协议服务崩溃，系统后果是网关重启期间上下游通信中断。过程后果取决于 PLC 是否本地闭环、输出保持策略、SCADA 失联报警和操作员响应。若安全控制独立，可能只造成监控缺口；若控制依赖远程命令，影响会扩大。

## 17. 安全风险与攻击链

```text
攻击者可达协议端口 -> 发送边界报文 -> 长度整数错误
-> 堆越界写 -> 服务崩溃/潜在控制流破坏
-> 网关通信失联 -> 操作员态势下降或远程控制中断
-> PLC 本地逻辑与安全联锁决定物理结果
```

报告要区分已验证崩溃、潜在代码执行和条件性物理影响。没有控制流证据时不能把拒绝服务直接描述成远程代码执行。

## 18. 防护与工程实践

- 新组件优先评估内存安全语言；原生边界保持最小；
- 固定位宽类型配合安全算术和显式范围检查；
- 明确缓冲区所有权、生命周期、容量和实际长度；
- 解析、认证和业务执行分层；
- 启用工具链缓解、静态分析、编译告警和 Sanitizer；
- 使用真实协议样本、边界语料、回归测试与模糊测试；
- 崩溃时保留输入哈希、版本、调用栈和内存诊断证据；
- 工业服务设计安全失败、限速、隔离和可控恢复策略。

## 19. 隔离实验

### 实验一：内存布局

编写小程序输出基础类型大小、结构体成员偏移、全局变量、堆对象和栈对象地址。在不同优化级别下比较结果，不把地址规律当作安全假设。

### 实验二：边界检测

为一个长度前缀解析函数编写正常、零长度、最大长度、截断、数量溢出和多余数据测试，在隔离虚拟机中使用 AddressSanitizer/UBSan 运行。

### 实验三：补丁对比

构造一个存在长度检查缺失的教学解析器，再修复为逐字段边界检查。比较源代码、编译告警、测试结果和崩溃行为。禁止将实验程序部署到生产网关。

## 20. 面试表达

### 60 分回答

C 允许直接控制内存和硬件资源，因此广泛用于固件、驱动和协议栈。安全分析中我重点关注指针、数组长度、对象生命周期、整数转换和结构体布局。常见缺陷包括越界读写、释放后使用、双重释放和格式化字符串。防护需要边界检查、明确所有权、编译器缓解和测试共同完成。

### 90 分回答

分析 C 协议解析器时，我会从外部长度和数量字段建立数据流，检查固定位宽转换、乘加溢出、剩余缓冲区和结构体对齐，再沿功能码分派、权限和协议状态检查控制流。崩溃复现使用固定版本、最小输入、调用栈和 Sanitizer 证据；缓解机制只能降低利用难度，不能代替根因修复。工控影响还要继续验证服务账户、网关角色、失联策略、PLC 本地闭环、输出保持和 SIS，区分软件崩溃、系统通信中断和真实物理后果。

## 本章总结

C 的能力来自对内存和运行时的直接控制，风险也来自同一位置。成熟研究需要把源码类型、机器内存、协议字节、操作系统防护和工业过程连成一条证据链。识别一个越界只是起点，确认触发条件、根因、实际能力、系统角色和安全屏障，才构成完整漏洞分析。
