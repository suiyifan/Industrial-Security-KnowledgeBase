# 数据库与工业数据基础

工业系统不仅处理实时控制变量，还保存资产、点表、报警、趋势、批次、配方、工单、用户和审计记录。数据库错误或失陷未必立即停止 PLC，却可能破坏操作员判断、配置发布、质量追溯和事故调查。成熟工控安全工程师需要理解数据模型、事务、权限和恢复，而不是只会写几条 SQL。

## 学习目标

- 理解关系模型、表、键、约束、规范化与数据类型；
- 掌握查询、连接、聚合和参数化 SQL 的基本原理；
- 理解 ACID、并发控制、隔离级别和一致性异常；
- 理解索引、执行计划、锁与性能问题；
- 设计数据库身份、权限、审计和秘密管理；
- 区分实时库、关系库、时序库和工业历史数据库；
- 评估备份、恢复、复制、高可用和数据完整性；
- 将数据库状态映射到控制配置和物理过程。

## 1. 数据库为什么存在

普通文件可以保存数据，但当多个用户并发读写、需要按条件查询、保持关系一致并从故障恢复时，应用自行实现会非常复杂。数据库管理系统提供数据模型、查询语言、事务、并发控制、权限、日志和恢复。

```text
应用/API/采集服务
       -> 数据库驱动与连接池
       -> 数据库引擎：解析、优化、事务、缓存、存储、日志
       -> 数据文件、事务日志、备份与副本
```

数据库“成功执行”只是数据层状态。配置表更新是否进入 PLC，还取决于发布服务、审批、协议通信、控制器校验和现场反馈。

## 2. 关系模型与表

关系数据库用表表示具有相同属性的一组记录。列定义属性与类型，行表示实例。示例报警表：

```sql
CREATE TABLE alarms (
    alarm_id      BIGINT PRIMARY KEY,
    site_id       INTEGER NOT NULL,
    occurred_at   TIMESTAMP NOT NULL,
    tag_name      VARCHAR(128) NOT NULL,
    severity      INTEGER NOT NULL,
    message       VARCHAR(512),
    acknowledged  BOOLEAN NOT NULL DEFAULT FALSE
);
```

类型不仅影响存储，还限定运算语义。时间、整数、浮点、定点、字符串、二进制和布尔值不能随意互换。配方和计量数据常需要精确小数，不能默认使用浮点。

### 2.1 主键、候选键与外键

主键唯一标识行；候选键是可唯一标识行的其他字段组合；外键表达表之间引用。

```sql
FOREIGN KEY (site_id) REFERENCES sites(site_id)
```

外键可防止指向不存在站点的数据，但并不保证该用户有权访问该站点。数据完整性约束和对象级授权是不同边界。

### 2.2 NULL 与三值逻辑

NULL 表示未知或不适用，不等于空字符串、0 或 false。SQL 比较 NULL 使用 `IS NULL`，普通比较可能产生 UNKNOWN。将传感器坏质量值粗暴转换为 0，会把“无数据”误报成真实过程值。

## 3. 约束与数据完整性

常用约束包括 `NOT NULL`、`UNIQUE`、`PRIMARY KEY`、`FOREIGN KEY` 和 `CHECK`。约束应表达数据库能够可靠执行的通用规则：

```sql
CHECK (severity BETWEEN 0 AND 5)
```

复杂过程安全规则通常还需应用、发布服务和 PLC 共同验证。例如数据库可以限制设定值绝对范围，但变化率、设备模式、联锁和批次状态可能只在控制上下文中确定。

完整性分层：

- 实体完整性：每条记录可唯一识别；
- 引用完整性：关系指向有效对象；
- 域完整性：类型、范围和格式有效；
- 业务完整性：满足审批、版本和工艺规则；
- 过程完整性：现场反馈与命令/记录一致。

## 4. 规范化与工程权衡

规范化减少重复和更新异常。常见思想是：一个事实只在合适位置保存，非键字段依赖键而不是其他非键字段。过度重复会造成修改一个站点名称却遗漏多张表。

读取密集报表或时序系统可能为了性能反规范化，但必须明确数据来源、刷新方式和一致性窗口。缓存表、物化视图和历史宽表中的旧值不一定是篡改，也可能是同步延迟。

## 5. SQL 查询基础

查询通常经历过滤、连接、聚合、排序和限制：

```sql
SELECT site_id, severity, COUNT(*) AS alarm_count
FROM alarms
WHERE occurred_at >= :start_time
  AND occurred_at < :end_time
GROUP BY site_id, severity
ORDER BY site_id, severity;
```

逻辑查询顺序与书写顺序、物理执行顺序并不完全相同。优化器会根据统计信息和代价选择访问路径。

### 5.1 JOIN 与数据语义

`INNER JOIN` 只保留匹配行，`LEFT JOIN` 保留左表并为缺失匹配填 NULL。选错连接类型可能让无资产映射的报警静默消失，形成监测盲区。

```sql
SELECT a.occurred_at, a.tag_name, t.engineering_unit
FROM alarms AS a
LEFT JOIN tags AS t ON a.tag_name = t.tag_name;
```

连接条件不完整会产生重复或笛卡尔积，既可能导致错误报表，也可能消耗大量资源。

### 5.2 参数化查询

外部输入不能直接拼接到 SQL。参数化查询先固定结构，再绑定值：

```python
cursor.execute(
    "SELECT * FROM alarms WHERE site_id = ? AND occurred_at >= ?",
    (site_id, start_time),
)
```

动态表名、列名和排序字段不能作为普通值参数，应由服务端白名单映射。参数化防止值改变 SQL 结构，不替代身份鉴别、站点授权和业务校验。

## 6. 事务与 ACID

事务把多个操作作为一个逻辑工作单元。ACID 常解释为：

- Atomicity 原子性：全部完成或全部回滚；
- Consistency 一致性：事务把数据库从一个有效状态带到另一个；
- Isolation 隔离性：并发事务按规定方式互相可见；
- Durability 持久性：提交后的结果在规定故障模型下可恢复。

持久性不等于任何灾难都不会丢数据。它依赖事务日志、刷新策略、存储、复制和备份配置。

### 6.1 提交与回滚

修改配方版本、内容和审批状态可能需要同一事务，否则部分成功会留下不完整版本。外部动作如“向 PLC 下装”无法由普通数据库事务自动回滚，需使用任务状态、幂等键、补偿操作和最终校验。

```text
数据库事务提交 -> 发布任务入队 -> 发布服务执行
-> PLC 响应 -> 在线状态验证 -> 任务最终状态
```

这是一致性跨系统问题，不能用一条 `COMMIT` 代替全过程验证。

## 7. 并发控制与隔离异常

并发事务可能产生：

- 脏读：读取未提交数据；
- 不可重复读：同一行两次读取不同；
- 幻读：同一条件两次得到不同记录集合；
- 丢失更新：一个写入覆盖另一个；
- 写偏差：分别合法的并发更新破坏组合规则。

数据库通过锁、多版本并发控制（MVCC）或组合机制提供隔离。隔离级别名称相似，但具体行为随产品实现不同。

工程上要根据业务选择：报警查询可容忍短暂变化，配方审批与版本发布则需要更强一致性和冲突检测。提高隔离也可能增加等待或失败重试。

## 8. 锁、死锁与阻塞

读写行、页、表或元数据可能获取锁。长事务会扩大阻塞，多个事务以不同顺序持有资源可能死锁。数据库通常选择一个事务回滚以解除死锁。

排查要区分：

- 谁在等待；
- 谁持有阻塞资源；
- 事务何时开始；
- 正在执行什么语句；
- 是否有未提交的交互会话；
- 重试是否幂等。

直接终止会话可能回滚大量数据并影响生产报表，必须按变更与应急流程处理。

## 9. 索引与执行计划

索引用额外存储和写入成本换取更快查找。常见 B-tree 适合等值和范围查询，其他索引取决于数据库产品。

复合索引的列顺序影响可用性。索引过多会拖慢报警写入与维护；没有适当索引则会让趋势查询全表扫描。

```sql
EXPLAIN SELECT * FROM alarms
WHERE site_id = 1 AND occurred_at >= :start_time;
```

执行计划是优化器估计或实际执行信息。分析要比较估计行数与实际行数、扫描方式、连接顺序、排序和临时空间。统计信息过期会导致错误计划。

SQL 性能问题可能在网络层表现为响应超时或 TCP 零窗口，不能直接归因于网络。

## 10. 连接、驱动与连接池

应用通过数据库驱动连接，连接池复用会话以降低建立成本。关键配置包括最大连接、等待超时、空闲回收、事务状态重置和凭据。

连接池耗尽可能由慢查询、泄漏、数据库阻塞或流量突增造成。盲目增加连接数可能把压力推向数据库。

连接字符串不应硬编码在仓库、日志或业务表中。使用秘密管理、独立服务身份、轮换和 TLS。数据库只监听必要接口，限制源主机和网络区。

## 11. 身份、角色与最小权限

数据库主体可能是用户、登录、角色、服务账户或证书映射，产品模型不同。权限应按业务角色分离：

- Web 查询：只读指定视图和站点范围；
- 报警确认：只执行受控存储过程或更新范围；
- 采集写入：只写历史/事件入口；
- 配置发布：访问批准版本和发布状态；
- DBA：管理结构、备份和恢复，不用于日常应用。

应用使用管理员账户会把一个 Web 漏洞扩大为全库风险。数据库权限也不能代替应用对象授权：同一个查询账户服务多个站点时，应用必须限制当前用户可见站点。

## 12. 视图、存储过程与触发器

视图可隐藏列和封装查询，存储过程可提供受控操作接口，触发器在数据事件发生时执行逻辑。它们可增强边界，也可能隐藏控制流。

风险包括动态 SQL、权限上下文混乱、递归触发、触发器副作用和难以追踪的同步。数据库写入后“谁自动消费或触发什么”必须纳入攻击路径。

## 13. 日志、审计与时间

数据库可能产生错误日志、慢查询、登录审计、语句审计、事务日志和复制日志。每种日志用途不同。

审计至少关注：登录成功/失败、权限变更、结构变更、配置/配方写入、审计策略变化、备份恢复和高权限操作。全量记录所有查询可能造成性能与隐私问题，应按风险设计。

审计发送到独立平台并校准时间。应用账户共享会降低可归责性，因此重要变更还需携带最终用户、工单和关联 ID，但这些上下文字段也要防伪和校验。

## 14. 事务日志、检查点与恢复

许多关系数据库采用预写日志：数据页写盘前先记录日志。检查点缩短恢复扫描范围，但不是备份。数据库异常关闭后可利用事务日志重做已提交操作、撤销未提交操作。

恢复目标常用：

- RPO：可接受的数据丢失时间范围；
- RTO：可接受的恢复耗时。

报警历史、配方和用户数据的 RPO/RTO 不一定相同。恢复策略必须由业务与安全共同定义。

## 15. 备份与恢复验证

备份类型可能包括完整、差异/增量、事务日志和快照。快照不是自动离线备份，副本也可能同步逻辑删除或篡改。

可靠策略要求：

- 备份数据、日志、结构、账户/权限和必要密钥；
- 加密传输与存储，限制备份操作权限；
- 保留离线或不可变副本；
- 记录版本与依赖；
- 定期在隔离环境恢复；
- 验证记录数、关键约束、应用连接和业务功能；
- 测量实际 RPO/RTO。

“备份任务成功”只证明任务报告成功，恢复演练才证明可用性。

## 16. 复制、高可用与灾备

复制可同步主从/多节点数据，高可用可在节点故障时切换，灾备应对站点级事件。同步复制降低已提交数据丢失但增加延迟，异步复制可能存在滞后。

需要监测：复制延迟、断链、冲突、只读/读写角色、故障切换、脑裂和客户端连接目标。故障切换后应用连接到旧主或只读副本，会产生写失败或数据不一致。

高可用不是备份：攻击者删除或加密数据时，复制可能迅速传播破坏。

## 17. 实时数据库、时序数据库与历史库

工业系统中的“实时数据库”常指 SCADA/DCS 内部当前值与状态模型，并不等同于关系数据库。历史数据库则按时间归档过程数据，可能使用专用压缩、例外存储、插值和质量码。

时序数据通常包含：

```text
timestamp + tag_id + value + quality + source
```

只保存值而忽略质量码，会把通信中断、坏质量或替代值误当成真实过程。压缩和插值会影响取证：趋势线上显示的点不一定都是实际采样点。

### 17.1 点表与语义映射

点名、地址、数据类型、量程、单位、采样周期、死区、质量和报警阈值共同定义过程语义。数据库字段 `value=100` 本身不能说明是 100 °C、100% 阀位还是无缩放原始值。

## 18. 工业数据链

```text
传感器 -> PLC 输入/变量 -> 工业协议/OPC -> 采集服务
       -> 实时库 -> 历史归档 -> 报表/MES/分析平台

配方/配置库 -> 审批 -> 发布服务 -> SCADA/DCS/PLC -> 现场过程
```

第一条多为过程数据上行，第二条可能把数据下行转为控制行为。安全评估必须确认数据库是只读镜像还是控制链的一部分。

## 19. 数据流、控制流与状态流

### 数据流

数据从驱动、API 或文件进入数据库，经类型转换、约束、事务和索引持久化，再被查询、报表或发布服务读取。

### 控制流

SQL、存储过程、触发器、任务和消息队列决定谁在何时消费数据。配置表版本变化可能触发自动发布，形成隐藏控制路径。

### 状态流

数据库事务已提交、发布任务已创建、下发成功、PLC 变量改变和执行器反馈是不同状态。每层都需独立证据。

## 20. 工控场景案例：报警历史缺失

某厂报警页面出现两小时空白。初步怀疑 PLC 未产生报警，但实时 SCADA 事件缓存仍有记录。数据库监控显示历史写入账户因密码轮换后未更新而持续认证失败，采集服务本地队列随后达到上限并丢弃旧事件。

根因链为身份配置不同步和队列容量不足，而非 PLC 逻辑故障。影响是历史追溯和审计缺失，实时控制未受影响。整改包括受控秘密轮换、连接告警、队列容量与持久化、补录机制和端到端报警完整性监测。

## 21. 安全风险与攻击链

```text
Web/API 输入或凭据失陷 -> 获取数据库查询/写入能力
-> 读取资产、点表、趋势和维护信息
-> 修改配方/配置/审批状态 -> 发布服务消费
-> 数据进入 SCADA/PLC -> 控制逻辑与联锁决定物理后果
```

数据库风险包括保密性泄露、完整性破坏和可用性下降。读取点表可支持后续侦察；修改历史可能掩盖事件；慢查询和锁可能拖垮趋势/报警。是否影响物理过程取决于下行消费者和安全屏障。

## 22. 防护原则

- 数据分级，明确过程、配置、身份和审计数据责任人；
- 使用参数化 SQL、严格类型和对象级授权；
- 采集、查询、确认、发布与 DBA 身份分离；
- 数据库只监听必要网络，使用 TLS 和秘密管理；
- 配方/配置采用约束、版本、双人审批、签名和范围校验；
- 限制查询时间范围、行数、超时、连接与资源；
- 审计权限、结构和关键写操作并远程保留；
- 备份加密、离线/不可变保存并定期恢复；
- 监测复制、容量、锁、慢查询、队列和端到端数据缺口；
- PLC/DCS 保留限幅、联锁与独立安全功能。

## 23. 隔离实验

### 实验一：关系与约束

使用 SQLite/PostgreSQL 测试库建立站点、点表和报警表，配置主键、外键、唯一和范围约束，观察无效数据如何被拒绝。

### 实验二：事务与并发

使用两个测试会话模拟并发读取与更新，观察隔离、阻塞、回滚和冲突。只在本地测试库执行。

### 实验三：索引与执行计划

生成虚构历史数据，比较有无复合索引的执行计划和耗时，同时观察索引对批量写入的成本。

### 实验四：备份恢复与发布状态

备份测试数据库并在新实例恢复，验证约束、权限和关键记录；再模拟“配置提交—任务—设备确认”状态机，证明数据库提交不等于设备执行。

## 24. 面试表达

### 60 分回答

关系数据库用表、键和约束维护数据关系，事务通过 ACID 和隔离机制管理并发。安全上我会使用参数化查询、最小权限、身份分离、审计和备份恢复。工控场景还要区分实时库、历史库和配置库，因为历史数据泄露主要影响侦察与追溯，配置库写入则可能进入发布链。

### 90 分回答

我会沿数据库完整生命周期分析：驱动和连接池以服务身份连接，SQL 经参数绑定、授权、约束和事务写入，日志保证恢复，触发器/任务/发布服务继续消费。看到配置表修改时不会直接声称 PLC 被控制，而会验证事务提交、审批版本、发布任务、工业协议响应、PLC 在线变量和现场反馈。性能上也会把慢查询、锁、连接池和 TCP 现象关联；恢复方面以实际演练证明 RPO/RTO，高可用副本不能替代离线备份。

### 常见追问

**参数化查询是否解决所有数据库安全问题？** 不能。它主要防止值改变 SQL 结构，仍需身份、对象授权、最小权限和业务校验。

**事务提交是否代表 PLC 已执行配置？** 不代表。跨系统发布还需队列、服务、协议、控制器和反馈验证。

**高可用副本是否等于备份？** 不等于。逻辑删除和恶意篡改可能被复制，仍需离线/不可变备份与恢复演练。

**为什么历史趋势可能与真实采样不同？** 历史库可能使用压缩、死区、插值和质量处理，必须查看原始样本与配置。

## 本章总结

数据库把工业数据组织成可查询、并发、持久和可恢复的状态，但数据层状态不等于现场状态。成熟工控安全工程师要理解模型、事务、性能、权限与恢复，并沿“采集—存储—消费—发布—反馈”追踪数据，准确区分数据库异常、监控影响、控制影响和物理后果。
