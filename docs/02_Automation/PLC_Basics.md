# PLC 原理、扫描周期与程序组织

PLC 是面向工业现场的实时控制器。它不仅执行梯形图，还负责 I/O 更新、周期与事件任务、通信、诊断、保持数据和运行模式管理。工控安全工程师需要理解这些功能怎样争用时间和状态，才能判断一次变量写入、程序下载、任务超时或模式切换会在何时影响物理输出。

## 1. 原理解释：PLC 为什么适合工业控制

PLC 的核心设计目标包括：

- 在电磁干扰、振动、温湿度变化下可靠运行；
- 以可预测周期执行逻辑；
- 提供标准工业 I/O 和现场总线接口；
- 支持在线诊断、维护和模块化扩展；
- 掉电、重启或模块故障后按已定义策略恢复；
- 让自动化工程师用符合控制思维的语言组织程序。

与通用计算机相比，PLC 更强调确定性、长生命周期、故障行为和现场可维护性，而不是最大吞吐量。

## 2. 工程实现：PLC 的硬件组成

| 组件 | 职责 | 安全关注 |
| --- | --- | --- |
| 电源 | 为机架与模块供电 | 冗余、欠压、掉电顺序 |
| CPU | 执行任务、管理内存与通信 | 模式、固件、看门狗、访问控制 |
| 本地 I/O | 接入现场信号 | 通道组态、强制、故障值 |
| 通信模块 | 接入以太网、现场总线或串口 | 路由、会话、远程写入、固件 |
| 功能模块 | 运动、计数、称重、温控等 | 独立状态与参数、异步行为 |
| 背板/机架 | 模块供电与内部通信 | 槽位身份、模块替换、总线故障 |
| 存储介质 | 保存程序、参数和固件 | 离线篡改、复制、版本与恢复 |

紧凑型 PLC 把这些功能集成在一个设备中，模块化 PLC 则通过机架和远程 I/O 扩展。风险判断需以具体型号和组态为准。

## 3. 扫描周期的基本模型

经典模型为：

```text
输入采样 → 用户程序执行 → 输出刷新 → 通信与诊断 → 下一扫描
```

输入过程映像在扫描开始附近形成快照，程序读取相对一致的输入；程序计算输出映像，扫描末端统一写到物理输出。这种设计避免同一扫描中输入变化导致前后逻辑看到不同状态。

但真实 PLC 可能包含：

- 多个不同周期的任务；
- 事件、硬件中断和时间中断；
- 直接外设读写和高速模块；
- 独立运动控制或安全任务；
- 异步网络通信与后台诊断；
- 生产者/消费者或周期 I/O 连接。

因此“一个扫描周期”只是入门模型，不是所有变量的统一时间边界。

## 4. 周期任务、事件任务和优先级

### 4.1 周期任务

按固定周期触发，例如 10 ms 快速控制、100 ms 一般逻辑、1 s 统计。任务执行时间必须小于周期并保留余量。

### 4.2 连续/循环任务

任务完成后再次运行，周期随负荷变化。适合对固定时间要求不高的逻辑。

### 4.3 事件与中断任务

由高速输入、通信事件或硬件中断触发，可抢占低优先级任务。若事件频率异常升高，可能造成低优先级任务延迟。

### 4.4 看门狗

任务或扫描超过允许时间会触发看门狗。控制器可能进入故障、停止任务或按配置处理输出。攻击流量、死循环、异常中断或计算负荷都可能触发时间故障，但需要结合 CPU 负载和诊断缓冲区证明。

## 5. PLC 运行模式

常见模式语义因厂商而异，通常包括：

- **RUN**：执行用户程序并更新输出；
- **STOP/PROGRAM**：停止用户程序，允许工程操作，输出按配置处理；
- **REMOTE**：允许工程软件远程切换模式；
- **TEST/DEBUG**：用于调试，可能允许单步、断点或特殊输出行为；
- **FAULT**：CPU 或任务发生不可恢复错误；
- **安全模式**：安全程序和普通程序具有独立签名、权限或状态。

模式切换属于高风险控制动作。必须明确谁可以切换、切换后输出怎样处理、恢复 RUN 时保持变量和现场设备会怎样响应。

## 6. IEC 61131-3 程序语言

### 6.1 LD 梯形图

接近继电器逻辑，适合启停、联锁和布尔控制。触点表示条件，线圈表示结果，但不能把图形触点直接等同于物理常开/常闭接线。

### 6.2 FBD 功能块图

用块和连线表达 PID、逻辑、计算和信号流，适合过程控制。追踪时注意块的内部状态和执行顺序。

### 6.3 ST 结构化文本

适合算法、循环、数据处理和状态机。数组越界、除零、类型转换和复杂分支更接近传统软件缺陷。

### 6.4 SFC 顺序功能图

用步、转移和动作描述顺序过程。重点审阅跳步、并行分支、超时和恢复。

不同语言最终由运行时调度执行，安全审查要跨语言追踪同一变量的所有读写。

## 7. 程序组织单元

通用概念包括程序、函数、功能块和数据实例：

- **函数**通常无持久内部状态，相同输入产生相同结果；
- **功能块**保存实例状态，例如定时器、PID、设备控制块；
- **程序/组织块**由任务或系统事件调用；
- **数据块/标签库**保存配置、运行状态和过程数据。

厂商术语如 OB、FB、FC、DB、Routine、AOI、Program 不完全等价，分析时应回到“谁调用、是否有状态、数据存在哪里、何时执行”四个问题。

## 8. 调用顺序与执行语义

梯形图通常按网络从上到下、从左到右求值，但优化、并行任务和厂商编译器会增加复杂性。同一变量在一个扫描内被多次写入时，最后生效值通常取决于执行顺序。

典型危险模式：

- 多处写同一输出，维护人员只看到其中一处；
- 自动逻辑写值后，手动逻辑在后面覆盖；
- 高优先级任务与低优先级任务共享非原子数据；
- 通信异步写入被本地逻辑立即覆盖，或恰好在条件判断间生效；
- 保持位在重启后跳过初始化。

## 9. 通信服务与控制执行的关系

PLC 通信可能用于：

- HMI/SCADA 周期读写标签；
- 控制器间数据交换；
- 远程 I/O 与驱动器周期通信；
- 工程软件上传、下载和在线监控；
- 诊断、时间同步和资产管理。

通信服务可能在后台、系统任务或独立通信处理器中执行。网络报文到达时间与用户程序读取时间并不相同。判断写入是否影响输出，需要确认目标内存、通信更新语义、程序覆盖和任务时序。

## 10. 故障、重启与输出行为

PLC 发生故障或重启时需回答：

- 物理输出清零、保持还是进入替代值；
- 保持变量、累计量和步序是否保留；
- 首次扫描逻辑如何初始化；
- 远程 I/O 和驱动器何时重新建立连接；
- HMI 是否在控制器未完全就绪时允许操作；
- 故障确认和复位是否会自动恢复设备。

恢复 RUN 不等于过程恢复。设备可能仍处于就地、跳闸、机械运动或危险物料状态。

## 11. 安全视角：PLC 高风险能力

PLC 安全风险来自工程权限、运行模式、任务调度和物理输出的组合。相同的网络请求在 STOP、RUN、远程和安全锁定状态下可能得到不同结果。

### 11.1 程序上传与资产侦察

上传工程可暴露点表、控制策略、设备名称、网络参数和保护逻辑，为后续精确操纵提供语义。

### 11.2 变量写入与强制

写内部变量、SP、模式、步位、输出映像和直接外设的后果不同。强制可能优先于程序并长期保留，应单独审计。

### 11.3 程序下载与在线修改

下载可能停止 CPU、重启任务或改变部分逻辑；在线修改可能只替换少量块而不中断过程。隐蔽性和过程冲击不同。

### 11.4 模式切换与资源耗尽

切换 STOP 会立即影响控制；大量工程会话、异常通信或高频事件可能消耗 CPU 和连接资源，但风险必须由诊断和实验验证，不能仅凭端口开放推断。

### 11.5 PLC 攻击链

```text
进入工程网络或取得可信会话
 → 识别 CPU、工程版本和运行模式
 → 写变量、强制、在线修改或切换模式
 → 任务调度、程序状态或输出映像变化
 → I/O 和执行器动作
 → 过程变量偏移
 → 报警、设备保护与 SIS 响应
```

数据流证据包括工程请求、变量值和 I/O；控制流证据包括任务、调用和模式；状态流证据包括 CPU、设备状态机、强制和故障。三类证据必须与现场能量和过程响应关联。

## 12. 调度与性能可观测性

判断 PLC 是否健康，可长期采集：

- 当前、平均和最大扫描/任务执行时间；
- 周期超限、看门狗和任务丢失计数；
- CPU、工作内存和持久存储使用率；
- 通信连接数、错误、超时和队列；
- I/O 模块与现场总线诊断；
- 主备同步、时间同步和重启原因；
- 强制数量、在线差异和模式变化。

基线必须覆盖正常生产、启停、批次切换和检修，因为负荷随工况变化。瞬时扫描升高不等于攻击；持续偏离并与新连接、程序变更或事件频率关联才更有意义。

## 13. 确定性、实时性与安全边界

实时不等于“速度快”，而是关键任务能在规定截止时间内完成。安全分析需要确认：

- 控制周期是否小于对象允许的响应时间；
- 最坏执行时间是否包含中断和通信负载；
- 低优先级诊断是否会饿死；
- 安全任务是否与普通任务隔离；
- 时间超限后的输出与状态是什么；
- 网络安全设备或日志功能是否引入不可接受抖动。

新增监测和加固措施也必须验证实时影响，不能为了可见性破坏控制确定性。

## 14. 工控场景：高优先级事件任务阻塞常规控制

某高速输入事件任务包含耗时计算。异常脉冲导致事件频繁触发，低优先级温度控制任务无法按 100 ms 周期运行，PID 实际采样时间增大并出现输出波动。最终可能触发看门狗或使控制质量下降。

证据链包括事件计数、各任务执行时间与抖动、CPU 负载、看门狗诊断、SP/PV/MV 趋势和输出刷新时间。防护应限制事件频率、缩短中断逻辑并监测任务超时。

## 15. 防护与工程治理

- 禁止未授权的上传、下载、在线修改、强制和模式切换；
- 对控制器固件、硬件组态、程序、任务和通信连接建立基线；
- 监测任务执行时间、最大扫描周期、CPU/内存和通信资源；
- 将高风险工程操作与人员、工单、时间和版本差异关联；
- 配置并验证故障、重启和通信中断时的输出策略；
- 保留可恢复工程备份，并在实验设备上验证恢复流程；
- 安全 PLC 的程序、签名、钥匙位置和旁路状态单独治理。

## 16. 面试表达

### 60 分答案

> PLC 读取输入、执行程序、刷新输出，并循环运行。它支持梯形图、功能块图、结构化文本和顺序功能图，也有 RUN、STOP 和远程等模式。

### 90 分答案

> 我把 PLC 理解为带实时任务调度、I/O 映像、状态内存、通信服务和故障策略的工业运行时。经典扫描只是基础模型，实际还要考虑不同周期任务、事件中断、直接 I/O、后台通信和看门狗。分析一次远程写入时，我会确认目标内存、任务读取时间、是否被本地逻辑覆盖、输出何时刷新，以及模式、强制和现场反馈。程序下载或 CPU 重启后还要验证保持数据、首次扫描逻辑和真实设备状态，不能只看到 RUN 灯就认为生产恢复。

### 常见追问

**问：输入在扫描中途变化，程序会立即看到吗？**

答：通过过程映像读取时通常下一次更新才看到，但直接外设访问、高速模块和异步任务可能不同，要查具体平台和变量访问方式。

**问：网络写变量成功为何设备没动作？**

答：可能被程序覆盖、模式不允许、许可或联锁不满足、输出被强制、任务未执行，或执行器拒动，需要沿控制链验证。

## 17. 实操建议

### 实验一：扫描内覆盖

在仿真 PLC 中让两个程序段写同一变量，改变调用顺序，观察最终输出。记录在线监控为何可能误导。

### 实验二：任务抖动

建立快慢两个周期任务，在快任务加入受控负载，观察慢任务周期、看门狗和控制趋势。不得在生产 PLC 进行。

### 实验三：重启恢复

设计保持与非保持变量、首次扫描初始化和安全输出，模拟重启并核对程序状态与现场模型是否一致。

### 实验四：性能基线

记录空载、正常逻辑、在线监控和受控通信负荷下的任务周期与抖动，观察性能指标与过程趋势。实验仅在仿真器或专用 PLC 完成，并设置看门狗与停止条件。

## 18. 总结与检查清单

- [ ] 能解释经典扫描模型及其边界；
- [ ] 能区分周期、连续、事件和中断任务；
- [ ] 能解释不同 IEC 61131-3 语言和有状态功能块；
- [ ] 能追踪多处写入、通信写入和任务优先级；
- [ ] 能说明模式、看门狗、重启和故障输出行为；
- [ ] 能把工程操作映射到任务、内存、输出和物理过程。
