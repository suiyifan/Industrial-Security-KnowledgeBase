# 控制逻辑阅读与危险动作识别

安全人员不必独立设计整套控制程序，但必须能从工程中找出设备怎样启动、模式怎样切换、联锁如何阻断、远程写入在哪里生效，以及某个变量最终会不会改变物理输出。本章给出跨厂商、跨语言的逻辑阅读方法。

## 1. 原理解释：先理解行为，再阅读语句

直接从第一行程序向下读往往效率很低。应先建立设备行为模型：

```text
命令来源 → 模式仲裁 → 启动许可 → 状态机
 → 输出命令 → 现场反馈 → 报警/跳闸 → 复位与恢复
```

每个变量都放到“命令、条件、状态、动作、反馈、保护或诊断”之一。这样才能区分相似布尔位的不同安全意义。

## 2. 工程实现：阅读前准备

收集以下资料：

- 受控工程版本和控制器型号；
- I/O 清单、点表、P&ID、电气图和因果矩阵；
- 程序调用树、交叉引用和变量字典；
- HMI 画面、按钮动作和报警配置；
- 设备说明、操作规程与联锁测试记录；
- 当前模式、强制、旁路和在线差异。

没有资料时可以从代码推断，但必须标注推断和待验证项，不能把变量名当作事实。

## 3. 第一步：建立调用与任务地图

先确定：

1. 哪些任务运行该逻辑，周期和优先级是什么；
2. 主程序调用哪些功能块或子程序；
3. 哪些块有独立实例状态；
4. 同一变量在哪些任务和块中被读写；
5. 是否存在中断、首次扫描或故障处理逻辑。

交叉引用是入口，但动态索引、指针、数组、别名、间接寻址和生成代码可能不被完全捕获。

## 4. 第二步：从物理输出向上追踪

选择高后果输出，如进料阀、加热器、电机启动或联锁旁路，从最终输出反向追踪：

```text
物理输出地址
 ← 输出仲裁/安全门
 ← 自动或手动命令
 ← 状态机动作
 ← 许可与联锁
 ← HMI/通信/程序命令来源
```

记录每个写入点、执行顺序、优先级和覆盖条件。最后一个写点不一定总生效，高优先级任务、强制或安全系统可能再次覆盖。

## 5. 第三步：读设备控制模板

典型电机或阀门功能块包含：

- 自动、手动、就地、远程和维护模式；
- 启动/停止或开/关命令；
- 许可、联锁、跳闸和旁路；
- 运行、到位、故障和通信反馈；
- 启动/停止超时与动作次数；
- 状态枚举、故障码和复位；
- 输出命令与模拟状态。

先画状态图，再对应块接口。避免只看输出线圈而忽略块内部锁存和定时状态。

## 6. 第四步：区分许可、联锁、报警和跳闸

对每个保护条件记录：

| 字段 | 要回答的问题 |
| --- | --- |
| 触发源 | 来自哪个测点、计算或通信状态 |
| 判定 | 阈值、方向、延时、迟滞是什么 |
| 作用 | 禁止启动、停止设备、关闭阀还是只报警 |
| 锁存 | 条件恢复后是否仍保持 |
| 复位 | 谁可复位，是否检查原因已消除 |
| 旁路 | 谁可旁路、持续多久、是否报警 |
| 独立性 | 是否与正常控制共享测点、模块和程序 |

因果矩阵描述设计意图，PLC 程序描述实现，两者必须比较。

## 7. 不同语言的阅读重点

### 7.1 LD

关注自保持、置位/复位、多线圈、边沿触发、定时器和取反语义。画面上的常闭触点表示逻辑取反，不一定对应现场常闭接点。

### 7.2 FBD

沿连线追踪信号，确认块执行顺序、内部状态、选择器、限幅和质量传播。多个线交叉时依端口而非视觉位置判断。

### 7.3 ST

关注 `IF/CASE` 分支、提前返回、循环、数组边界、类型转换、位运算和异常输入。搜索赋值语句和函数副作用。

### 7.4 SFC

关注步激活、转移条件、并行分支、动作限定符、超时、跳步与恢复。一步完成位可能来自多个现场条件。

## 8. 模式、旁路、强制和仿真路径

危险逻辑往往不在正常自动路径，而在维护功能：

- 手动模式直接输出；
- 就地/远程切换改变命令权；
- 仿真位用固定值替代真实反馈；
- 旁路位屏蔽联锁；
- 强制表覆盖程序结果；
- 调试参数放宽限值或超时；
- 故障复位清除锁存状态。

审阅时单独建立“保护削弱状态表”，记录来源、权限、可见性、超时和恢复。

## 9. 通信写点的追踪

网络可写变量可分为：

- 请求位和参数；
- 设定值或配方；
- 模式、步号和维护命令；
- 联锁旁路与复位；
- 内部状态或直接输出。

从协议对象或 HMI 标签追到 PLC 变量，再检查所有读取点、边沿处理、许可和状态迁移。一次写入即使下一扫描被覆盖，也可能已触发锁存或命令队列。

## 10. 时序、边沿与竞态

阅读静态逻辑时还要问：

- 上升沿只在一个扫描有效，谁能捕获；
- 定时器基于任务周期还是系统时间；
- 多任务是否同时访问同一变量；
- 通信更新发生在判断前还是判断后；
- 复位和故障若同一扫描出现，谁优先；
- CPU 重启后首次扫描会置位什么。

关键条件可画扫描级时序表，避免仅靠截图判断。

## 11. 数据流、控制流、状态流与能量流

以加热器为例：

```text
数据流：温度 PV、SP、输出反馈、故障
控制流：模式 → PID/手动 → 限幅 → 联锁门 → DO/AO
状态流：Off → Starting → Heating → Holding → Faulted
能量流：电能/蒸汽 → 物料温度
```

逻辑阅读完成的标志，是能从任一外部输入解释到最终能量变化，并指出保护在哪一层阻断。

## 12. 安全视角：危险模式

逻辑安全风险的本质，是某条路径允许未经预期约束的状态迁移或能量释放。审阅不能只搜索危险函数名，还要证明路径可达、执行优先级和现场条件。

- 输出有多个写点且缺少明确仲裁；
- 关键许可只在启动瞬间检查，运行中不再验证；
- 旁路无超时、无权限或无明显报警；
- 坏质量被替换为“正常”值；
- 复位命令同时清除多个不同故障；
- 通信命令无序列号和防重放；
- 保持状态与现场位置在重启后不一致；
- HMI 隐藏关键模式、强制或联锁状态；
- 安全动作依赖与正常控制相同的单一变量。

### 12.1 逻辑攻击链

```text
取得 HMI、协议或工程写入能力
 → 影响命令、模式、步位、旁路或程序块
 → 满足/绕过某条控制流条件
 → 设备状态机发生迁移
 → 最终输出与能量流变化
 → 过程响应和保护动作
```

从输入到输出的逻辑切片应标出所有条件、写点、任务和覆盖关系。无法证明路径可达时，应把后果标为待验证。

## 13. 逻辑切片与最小审阅范围

大型项目不可能逐行阅读。可围绕高后果对象做切片：

1. 选择物理输出或安全动作；
2. 反向收集所有赋值和调用；
3. 展开命令、模式、许可和联锁依赖；
4. 加入通信映射、HMI 对象和保持状态；
5. 加入任务、初始化、故障和重启路径；
6. 用现场反馈和因果矩阵验证切片完整性。

切片边界和未解析的间接寻址必须记录，避免把局部审阅写成完整系统结论。

## 14. 测试用例设计

每个关键设备至少覆盖：

- 正常启动、运行、停止；
- 每项许可不满足；
- 运行中每项联锁触发；
- 反馈缺失、迟到和矛盾；
- 手动、自动、就地、远程切换；
- 旁路、强制、仿真和复位；
- 通信重复、超时和乱序；
- CPU 重启、保持状态和异常恢复；
- 多故障同时出现及优先级。

测试应检查内部状态、最终输出、报警、事件和过程响应，而不只检查某个线圈。

## 15. 工控场景：泵许可仅在启动沿检查

程序在收到启动上升沿时检查液位许可，随后锁存运行；运行期间液位许可下降并不停止泵。设计意图若要求防干转，这就是保护缺口。攻击者或故障只需在启动瞬间伪造高液位，之后即使真实液位低也可持续运行。

验证应比较因果矩阵、启动网络、运行锁存、低液位报警、跳闸逻辑和泵入口压力趋势。

## 16. 审阅记录与风险表达

每个发现至少记录：

```text
工程版本与控制器：
程序块/网络/行号：
变量和物理对象：
正常状态与预期保护：
触发前提和权限：
逻辑路径与执行时序：
命令、反馈和过程响应：
独立保护与可检测证据：
最坏可信后果：
修复与回归测试：
```

不要写“可修改 PLC 导致生产事故”这种跳跃结论，应明确修改对象、逻辑条件、现场能力和保护层。

## 17. 防护与验证

- 高后果输出建立单一、可审计的仲裁点；
- 关键保护同时检查启动前许可和运行中联锁；
- 旁路、仿真、强制和维护模式默认可见、限时并审计；
- 通信命令使用握手、序列号、许可和结果码；
- 逻辑变更同时更新因果矩阵、点表和测试用例；
- 对正常、边界、故障、重启和恢复路径做仿真；
- 用现场反馈和过程趋势验证程序动作。

## 18. 面试表达

### 60 分答案

> 阅读 PLC 逻辑时，我先看任务和调用关系，再从关键输出反向追踪命令、许可、联锁、反馈和报警，并检查手自动、旁路和强制。

### 90 分答案

> 我会先用 P&ID、点表和因果矩阵建立设备行为模型，再从高后果物理输出反向做交叉引用，记录所有写点、任务优先级和覆盖条件。随后分别审阅正常自动路径、手动维护路径、通信写点、旁路强制以及故障和重启恢复。对于发现，我会走完变量写入—状态迁移—输出命令—现场反馈—过程响应—保护层的链路，并用工程版本、逻辑位置和趋势证据限定结论。

### 常见追问

**问：怎么判断某个 HMI 按钮是否危险？**

答：从按钮动作追到标签，再到 PLC 读取点、模式与许可、状态机和最终输出，同时检查权限、确认、反馈和保护。

**问：交叉引用是否能找到所有写点？**

答：不一定，动态索引、间接寻址、别名、通信映射和生成代码可能遗漏，还要结合数据结构和在线监控。

## 19. 实操建议

### 实验一：泵逻辑审阅

实现含自动、手动、许可、联锁、反馈超时和复位的泵块，从 DO 反向画出完整依赖树。

### 实验二：通信写点

让外部脚本写请求位，比较电平触发、上升沿触发和请求—确认握手的重复执行风险。

### 实验三：异常路径

模拟反馈迟到、许可运行中消失、CPU 重启和旁路遗留，记录程序状态、输出与现场模型。

### 实验四：最小逻辑切片

从一个阀门 DO 开始，生成包含全部写点、任务、模式、联锁、通信标签和 HMI 按钮的切片图。让另一人根据图复核，记录遗漏的动态索引或间接调用。

### 回归测试证据

逻辑修复后要重跑正常路径和所有相关异常路径，并比较修改前后的状态、输出、报警与过程趋势。测试记录应注明工程版本、任务周期、初始状态、仿真输入、预期结果和实际结果。

对于无法在实验环境复现的现场逻辑，应通过桌面推演、厂商测试工具或维护窗口内的受控功能试验补充证据，并明确尚未验证的物理环节。

审阅结论应由程序作者以外的人员复核，高后果逻辑还应邀请工艺、仪表或电气专业确认设计意图和失效行为。

复核者应从原始工程独立重建关键控制链，而不是只阅读发现摘要，这样才能发现切片边界、变量别名和任务优先级遗漏。

## 20. 总结与检查清单

- [ ] 能建立任务、调用和变量交叉引用地图；
- [ ] 能从高后果输出追到所有命令与保护条件；
- [ ] 能跨 LD、FBD、ST、SFC 识别状态与副作用；
- [ ] 能审阅模式、旁路、强制、仿真和重启路径；
- [ ] 能分析通信写入、边沿和多任务时序；
- [ ] 能用工程和过程证据表达可信物理后果。
