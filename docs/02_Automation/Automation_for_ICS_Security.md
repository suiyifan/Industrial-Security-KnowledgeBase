# 自动化知识如何服务工控安全

## 1. 从漏洞到工控风险

普通漏洞报告常写：

```text
攻击者可远程执行命令。
```

工控安全报告要继续写：

```text
如果受影响主机是工程站，攻击者可能进一步获取工程文件、修改控制逻辑、下载恶意工程，影响 PLC 控制行为。
```

也就是说，要把技术影响翻译成工控影响。

## 2. 判断资产重要性

同样一台 Windows 主机，重要性可能完全不同：

| 主机类型 | 工控影响 |
|---|---|
| 办公网普通终端 | 入口或跳板 |
| 操作员站 | 可能下发控制命令 |
| 工程站 | 可能修改工程和控制逻辑 |
| 历史站 | 可能影响追溯和数据可信 |
| SCADA Server | 可能影响集中监控和采集 |

## 3. 判断漏洞影响

评估工控漏洞时，可以按这个顺序：

1. 漏洞在哪个资产上？
2. 资产处于哪个网络区域？
3. 攻击者需要什么前置条件？
4. 攻击者能读什么？
5. 攻击者能写什么？
6. 是否影响控制逻辑？
7. 是否影响报警、趋势、历史数据？
8. 是否影响联锁或安全功能？
9. 是否会造成停机、误操作、设备损坏或安全风险？

## 4. 协议分析时如何结合自动化知识

看到协议功能码时，不要只看“读写”。

要继续分析：

- 读的是过程值还是设备信息？
- 写的是设定值还是控制命令？
- 地址是否对应关键点位？
- 写操作是否需要权限？
- 设备是否有本地保护或联锁？
- 是否能通过流量判断操作员行为？

## 5. 逆向分析时如何结合自动化知识

逆向工控软件时要关注：

- 通信驱动
- 认证逻辑
- 工程文件格式
- 点表解析
- 脚本执行
- 下载工程流程
- 变量读写接口
- 报警和趋势处理

自动化知识能帮助你判断哪个函数更关键。

## 6. 安全评估时如何结合自动化知识

评估时不只列漏洞，还要写：

- 关键资产清单。
- 控制链路。
- 网络区域。
- 远程运维路径。
- 工程变更流程。
- 备份恢复能力。
- 关键联锁和安全功能。

## 7. 报告表达模板

可以使用：

```text
该漏洞位于【资产类型】上，该资产在控制链路中承担【功能】。
攻击者利用后可以【技术动作】。
在工业场景中，可能进一步导致【工控影响】。
如果该资产连接【关键设备 / 协议 / 点位】，风险会进一步扩大。
建议采取【整改措施】。
```

示例：

```text
该漏洞位于 HMI 操作员站上，该资产承担现场监控和设备操作功能。
攻击者利用后可获得系统权限，进一步读取组态画面、变量点表并下发异常操作。
在水处理场景中，可能影响泵站启停、阀门状态或加药参数。
建议隔离访问路径、修复漏洞、收敛操作权限，并审计关键操作日志。
```

## 8. 面试表达

可以这样说：

> 我学习自动化知识的目的，是在漏洞分析时判断资产在控制链路中的位置。比如同样是远程命令执行，如果发生在普通办公终端，主要是入口风险；如果发生在工程站或 HMI，就可能进一步影响工程文件、控制逻辑、操作命令和现场设备状态。
