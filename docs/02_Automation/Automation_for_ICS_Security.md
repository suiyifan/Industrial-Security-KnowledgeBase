# 自动化知识如何服务工控安全

自动化知识的价值，是把网络、软件和协议证据翻译为控制语义。工控安全工程师不需要替代工艺与控制工程师，但必须能提出正确问题、识别危险对象、限定物理后果，并设计不破坏生产的验证方法。

## 1. 从资产漏洞到控制链位置

发现漏洞后先定位资产角色：

- 它是工程、操作、控制、通信、历史还是安全保护节点；
- 它能读取、操作、维护、工程下载还是管理身份；
- 它连接哪些控制器、设备和生产单元；
- 它是否跨越办公、控制和安全系统边界；
- 它保存哪些点表、工程、配方、凭据和证据。

同样的远程代码执行发生在报表终端和工程站上，后续能力完全不同。

## 2. 协议分析中的自动化语义

抓包字段应映射为：

```text
协议对象/地址
 → PLC/RTU 标签与数据类型
 → 命令、状态、参数或诊断
 → 控制逻辑读取与写入位置
 → 现场设备、变量和过程后果
```

分析写请求时确认：

- 写的是 PV、SP、模式、步位、旁路还是直接输出；
- 是否有序列号、握手、许可和结果确认；
- 控制器模式和任务时序是否允许生效；
- 本地逻辑会不会立即覆盖；
- 命令、反馈和过程响应是否一致；
- 独立保护能否限制后果。

因此“写寄存器成功”只是传输证据，不是物理影响证据。

## 3. 漏洞研究中的危险功能识别

对厂商软件、固件或服务做逆向时，自动化语义帮助优先识别：

- 工程上传、下载、编译与固件升级；
- 控制器模式、强制、旁路和复位；
- 标签读写、配方导入和批量参数；
- 设备发现、身份校验和会话权限；
- 报警、历史、质量和时间处理；
- 安全程序签名、校验和密钥；
- 备份、恢复和在线比较。

一个解析缺陷的风险取决于它位于只读诊断路径，还是可到达下载、写变量或模式切换路径。

## 4. 流量监测中的过程基线

单纯记录 IP、端口和字节数无法判断控制异常。更有价值的基线包括：

- 哪些资产在什么模式下通信；
- 周期读、事件上送、工程会话和控制命令的时序；
- 正常可写对象与操作窗口；
- 设备启停、批次切换和检修时流量怎样变化；
- 命令后应出现什么反馈和过程响应；
- 通信中断后现场怎样退化。

监测规则应将异常报文与控制器状态、操作日志、趋势和工单关联。

## 5. 安全评估中的低扰动原则

生产系统验证优先使用：

1. 文档、工程和配置离线审阅；
2. 被动流量和已有日志；
3. 厂商诊断与受控查询；
4. 同型号实验环境复现；
5. 维护窗口内经审批的最小验证。

端口扫描、畸形报文、写变量、强制、下载和模式切换都可能影响确定性或生产，只能在明确授权和风险控制下实施。

## 6. 事件调查中的跨层证据

调查不能只看主机日志。应构建：

```text
身份与远程会话
 → 工程/操作命令
 → 控制器模式、程序和变量变化
 → I/O 与设备反馈
 → 报警、趋势、SOE 和历史
 → 现场物料、设备与保护动作
```

注意不同时间源的精度和漂移。HMI 截图只能证明某时某界面显示什么，不能单独证明真实过程状态。

## 7. 风险分级中的过程后果

除通用安全属性外，还要考虑：

- 人员安全与危险能量；
- 环境释放和法规影响；
- 设备损坏与维修周期；
- 产品质量和批次追溯；
- 生产停顿与安全停车复杂度；
- 操作员态势感知；
- 恢复所需工程、物料和现场验证。

风险等级应结合可利用前提、过程可达性、保护独立性和证据可信度。

## 8. 修复建议中的工程可实施性

“关闭端口”或“立即升级”可能破坏厂商支持和生产。建议应包含：

- 长期消除措施；
- 暂时补偿控制；
- 受影响资产和通信；
- 变更窗口、回退与验证；
- 对控制周期、冗余和操作的影响；
- 厂商、工艺、控制、电气和安全人员责任；
- 修复后命令、反馈、报警和保护测试。

## 9. 工控场景：工程站文件解析漏洞

恶意工程文件触发工程软件代码执行。完整分析不是“RCE 可控制 PLC”，而是：

1. 文件如何进入工程站及是否需用户打开；
2. 进程获得什么账号和网络权限；
3. 是否保存控制器凭据与受控工程；
4. 控制器是否允许远程下载或变量写入；
5. 可修改哪些逻辑、参数和画面；
6. 过程状态与保护是否允许形成后果；
7. 工程、主备、强制和现场状态如何恢复。

## 10. 面试表达

### 60 分答案

> 自动化知识帮助我理解漏洞所在资产的角色，并把协议地址和变量映射到 PLC、HMI、设备和过程影响。

### 90 分答案

> 我会把网络和软件证据沿控制链翻译。发现写请求后，不会止于“可写”，而会确认对象语义、任务时序、模式许可、命令握手、现场反馈和独立保护。做评估优先采用工程文件、被动流量、日志和隔离复现，避免生产主动测试。报告中区分理论可能、工程可信和实验验证，并给出包含窗口、回退和过程验收的可实施建议。

## 11. 实操建议

选择一个公开 PLC/HMI 漏洞，分别完成：资产角色图、能力清单、控制链、必要前提、保护层、证据等级、低扰动验证方案和恢复检查表。不得把厂商演示或仿真结论直接外推真实生产。

## 12. 总结

自动化知识让安全工程师能够回答三个关键问题：攻击者究竟改变了什么控制对象，这种变化如何到达物理过程，以及现有保护和证据是否支持所声明的后果。
