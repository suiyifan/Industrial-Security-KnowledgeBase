# 模拟量、数字量与 I/O 链路

I/O 是物理过程与控制程序的边界。现场温度、阀位和开关状态只有经过接线、信号调理、模数转换、通道组态和量程换算，才会成为 PLC 中的变量；程序输出也必须经过内存映像、模块、电气回路和执行器才能形成动作。工控安全分析需要逐层确认“程序看到什么”和“现场实际发生什么”是否一致。

## 1. 原理解释：为什么需要标准信号与 I/O

控制器不能直接理解压力或液位，也不能直接驱动大功率电机。I/O 系统承担四项职责：

- **适配**：把电流、电压、触点、脉冲和总线数据转换为控制器可用的数据；
- **隔离**：在现场高噪声、高电压环境与控制器电子电路之间建立边界；
- **确定性采样**：按照模块和任务周期稳定更新输入输出；
- **诊断**：识别断线、短路、超量程、模块故障和通信中断。

标准化让不同仪表和控制器能够组合，但每次转换也可能丢失语义或产生配置差异。

## 2. 工程实现：I/O 类型与现场回路

| 类型 | 含义 | 典型输入/输出 | 关键风险 |
| --- | --- | --- | --- |
| DI | 数字输入 | 限位、按钮、继电器反馈、故障触点 | 常开/常闭语义、抖动、断线不可区分 |
| DO | 数字输出 | 接触器、电磁阀、指示灯 | 输出容量、继电器粘连、失电状态 |
| AI | 模拟输入 | 4–20 mA、0–10 V、热电偶、热电阻 | 量程、分辨率、断线、共模干扰 |
| AO | 模拟输出 | 阀位、VFD 速度、功率指令 | 正反作用、保持值、故障方向 |
| PI/PTO | 脉冲输入/输出 | 流量累积、编码器、高速计数、步进控制 | 丢脉冲、溢出、频率限制 |
| 远程 I/O | 总线连接的分布式模块 | 上述各类通道 | 网络中断、设备替换、组态不一致 |

### 2.1 DI 回路

DI 不只是 0 和 1。现场常采用常闭回路表示健康状态，使断线趋向故障；但程序可能再次取反。阅读逻辑时必须区分：现场触点状态、模块输入位、程序变量语义和 HMI 显示文本。

按钮还可能存在机械抖动，PLC 会用硬件滤波或软件定时消抖。滤波过短可能重复触发，过长可能漏掉快速事件。

### 2.2 DO 回路

PLC DO 往往只驱动中间继电器，后者再驱动接触器或电磁阀。输出位为 0 不等于设备已停止：继电器可能粘连、接触器可能焊死、机械惯性仍在、现场回路可能被旁路。因此需要辅助触点、速度或位置反馈验证动作。

### 2.3 AI/AO 回路

电流信号抗长距离压降和干扰能力较强。4 mA 作为活零点，可为两线制仪表供电并与断线状态区分。常见约定会把低于约 3.6 mA 或高于约 21 mA 作为故障区间，但具体阈值必须以设备手册和项目组态为准。

## 3. 模拟量从电流到工程量

假设变送器量程为 0—10 m，对应 4—20 mA，线性换算为：

```text
PV = (I - 4 mA) / (20 mA - 4 mA) × (10 m - 0 m) + 0 m
```

12 mA 对应 50%，即 5 m。实际控制器通常先把 A/D 转换结果保存为整数原始码，再进行归一化和工程量缩放：

```text
现场电流 → ADC 原始码 → 0.0...1.0 → 工程量 → 滤波值 → 控制变量
```

安全审查至少核对：

- 模块通道类型与量程；
- 原始码上下限和有无符号；
- 工程量上下限、单位和小数精度；
- 越界值、坏质量与替代值处理；
- 滤波时间、变化率限制和报警限；
- PLC 与 HMI 是否重复缩放。

错误缩放可能产生“趋势方向正确但绝对值错误”的隐蔽问题。

## 4. 数字量语义与失效设计

同一个布尔值可能表示“命令开”“实际开到位”“允许打开”“故障时关闭”或“信号健康”。变量名和电气逻辑必须共同解释。

以紧急停车回路为例，现场常用失电触发设计：健康时回路带电，按钮按下、断线或失电都会使输入变为非健康。程序中可能命名为 `ESD_OK`，值 1 表示健康；如果只看到地址为 0 就把它解释成“未触发 ESD”，结论会完全相反。

## 5. 过程映像与更新时序

典型 PLC 周期如下：

```text
读取物理输入 → 更新输入过程映像
            → 执行用户任务
            → 写入输出过程映像
            → 刷新物理输出
            → 通信、诊断与后台任务
```

但不是所有 I/O 都严格遵循一次扫描：高速 I/O、事件任务、直接外设访问、运动控制和分布式总线可能异步更新。分析竞态或短脉冲时，需要确认模块更新周期、总线周期、任务周期和程序访问方式。

### 5.1 远程 I/O 的时序

远程 I/O 增加了网络和设备状态：控制器可能在通信丢失后保持最后值、切换替代值或将质量标为坏。不同通道的更新时间也可能不同，导致看似同一时刻的数据实际来自不同采样窗口。

### 5.2 输出故障策略

控制器停止、模块掉线或网络中断时，输出可能：

- 清零；
- 保持最后值；
- 切换预定义安全值；
- 由冗余控制器接管；
- 交给现场设备本地控制。

“清零”不天然安全。例如冷却阀的 0% 可能使温度继续上升。安全值必须由过程危害分析决定。

## 6. 数据流、控制流、状态流和能量流

以变频泵为例：

```text
AI 液位 → 原始码 → 工程量 PV → PID
PID 输出 → AO/总线速度给定 → VFD → 电机 → 泵流量
DI/总线反馈：就绪、运行、故障、远程模式
状态：Stopped → Starting → Running → Faulted
能量：电能 → 机械能 → 流体压力能
```

若 HMI 显示速度给定 50%，仍要确认 PLC 输出、VFD 接收值、实际频率、电机转速和流量响应。每一层都可能被故障或攻击割裂。

## 7. I/O 诊断与故障定位

定位异常时按从物理到逻辑的顺序检查：

1. 就地表计或设备实际状态；
2. 现场供电、保险、端子和接线；
3. 万用表/校验仪测得的电流、电压或触点；
4. I/O 模块指示灯、通道诊断和原始值；
5. 控制器过程映像和工程量；
6. 程序内部滤波、替代与仿真逻辑；
7. 协议传输的值和质量码；
8. HMI、报警与历史显示。

从 HMI 直接跳到“PLC 被攻击”会漏掉大量更常见的现场故障。

## 8. 安全视角：I/O 链路的攻击点

I/O 安全风险的核心是“控制器内部表象与现场真实状态分离”。攻击者不一定需要构造异常电流，也可能通过工程组态、远程 I/O 参数、强制表或坏质量处理改变上层解释。

### 8.1 量程与类型篡改

改变 AI 量程、热电偶类型或缩放参数，可在通信正常的情况下改变变量语义。改变 AO 正反作用可能让控制方向反转。

### 8.2 强制与仿真

工程软件常允许强制输入、输出或内部变量，用于调试。强制可能覆盖真实现场变化，并在程序下载或重启后继续存在，具体行为取决于平台。强制表必须作为高风险状态审计。

### 8.3 坏质量处理滥用

通信失败后长期保持最后值，会让控制和操作继续依赖过期数据；直接替换为 0 又可能触发错误动作。合理策略应根据变量用途区分控制、显示和保护，并明显标识质量。

### 8.4 远程 I/O 与设备替换

攻击或误维护可能让同型号模块以错误组态重新加入，导致通道顺序、类型或安全值变化。设备身份、固件、组态校验和接线复核缺一不可。

### 8.5 典型攻击链与必要证据

```text
取得工程或远程 I/O 管理权限
 → 修改通道类型、量程、滤波或故障替代值
 → 控制器获得格式合法但语义错误的数据
 → 控制逻辑、报警与 HMI 基于错误值动作
 → 输出和过程状态偏移
 → 设备保护或 SIS 决定最终后果
```

验证该攻击链至少需要工程差异、模块诊断、I/O 原始码、工程量、质量码、命令与现场响应。若只能证明配置接口可访问，就不能直接宣称已影响物理过程。

## 9. 工控场景：液位输入冻结与泵干转

某水池液位 AI 因远程 I/O 通信中断保持最后的高液位值，PLC 自动逻辑认为水量充足并继续启动出水泵。真实液位持续下降，若低液位联锁也依赖同一冻结值，泵可能干转。

风险链如下：

```text
I/O 通信中断
 → PV 保持最后值且质量未进入控制判断
 → 低液位许可持续成立
 → 泵保持运行
 → 水池抽空、汽蚀和设备损伤
```

防护应包括通信质量参与许可、独立低低液位开关、泵入口压力/电流监测、超时策略和操作员明确告警。

## 10. 防护与变更治理

- 对机架、槽位、模块、通道、量程、滤波和故障值建立基线；
- 限制 I/O 组态、强制、仿真和在线修改权限；
- 对强制项、旁路项和坏质量替代项设置持续告警和交接记录；
- 关键动作同时验证命令、反馈与过程响应；
- 对远程 I/O 网络监测设备掉线、重新加入和组态变化；
- 变更后执行点对点回路测试，确认方向、单位、报警和失效动作；
- 恢复生产前清除强制并现场核对输出实际状态。

## 11. 面试表达

### 60 分答案

> I/O 把传感器和执行器连接到 PLC。DI/DO 处理开关量，AI/AO 处理连续信号。模拟量要经过原始码和工程量换算，数字量要注意常开常闭和状态语义。

### 90 分答案

> 我会把 I/O 看成物理过程与程序的语义边界。一个现场量要经过变送器、接线、模块转换、过程映像、缩放、质量处理和标签映射，输出也要经过过程映像、电气回路和执行器才形成物理动作。评估时我会核对通道类型、原始码、量程、滤波、故障值、强制状态和更新时间，并用命令—反馈—过程响应做交叉验证。这样能区分现场故障、组态错误、通信异常和恶意操纵。

### 常见追问

**问：PLC 输出位为 0，设备为什么仍可能运行？**

答：可能存在继电器粘连、接触器焊死、现场手动回路、机械惯性、旁路接线或反馈错误，需要检查电气回路和真实状态。

**问：远程 I/O 掉线后应保持还是清零？**

答：没有统一答案。应按过程风险为每个输出定义保持、安全值或本地接管策略，并验证通信质量能够进入控制与报警逻辑。

## 12. 实操建议

### 实验一：缩放与故障区间

在仿真环境输入 3.6、4、12、20、21 mA 对应的原始码，观察工程量、质量和报警。随后制造 PLC 与 HMI 双重缩放，记录错误如何产生。

### 实验二：I/O 时序

创建一个短脉冲输入，改变模块滤波和任务周期，观察哪些脉冲被捕获。记录模块更新、任务执行和输出刷新之间的关系。

### 实验三：通信丢失策略

在远程 I/O 仿真中分别设置保持、清零和安全值，断开通信后观察控制器、HMI、报警和过程模型。实验前必须定义恢复步骤和停止条件。

### 实验四：端到端回路核对

为一个 AI 和一个 DO 建立证据表。AI 从校验电流、模块原始码、PLC 工程量一直核对到历史值；DO 从程序命令、输出映像、模块指示、继电器和现场反馈核对到过程响应。故意引入一次取反或双重缩放，记录哪一层首先暴露异常。

### 恢复验收

实验结束后不能只恢复工程文件。还要清除强制和仿真、恢复通道参数、重新建立远程 I/O 连接、确认质量为好、检查输出实际位置，并验证报警和历史重新连续。

## 13. 总结与检查清单

- [ ] 能解释 DI、DO、AI、AO 和脉冲 I/O 的工程实现；
- [ ] 能完成原始码、归一化值与工程量之间的转换；
- [ ] 能区分物理输入、过程映像、程序变量和 HMI 标签；
- [ ] 能解释扫描周期之外的异步 I/O 与远程 I/O 行为；
- [ ] 能分析保持值、替代值、强制和坏质量对控制的影响；
- [ ] 能沿命令—反馈—响应链证明真实物理动作。
