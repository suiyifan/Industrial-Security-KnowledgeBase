# 点表、标签与过程数据语义

点表是连接工艺、仪表、I/O、PLC、协议、HMI、报警和历史数据的核心字典。协议分析看到的地址或标签只有结合点表，才能判断它代表测量值、控制命令、模式、联锁还是诊断。点表错误会让多个系统在“通信正常”的情况下共同理解错误。

## 1. 原理解释：一个点不只是地址和值

完整过程点至少包含：

- 身份：位号、名称、描述、所属装置和设备；
- 来源：仪表、I/O 机架槽位通道或计算逻辑；
- 类型：DI、DO、AI、AO、累计量、字符串、结构体；
- 语义：测量、命令、反馈、许可、联锁、报警或诊断；
- 数值：原始范围、工程范围、单位、精度和有效边界；
- 时间：采样周期、源时间、更新时间和历史策略；
- 质量：好、坏、不确定、替代、强制、仿真；
- 访问：读写方向、角色、确认和审计；
- 映射：PLC 变量、协议对象、HMI 标签和历史点。

没有这些上下文，`40001 = 1` 不能直接解释为“启动泵”。

## 2. 工程实现：从 P&ID 位号到数字标签

过程工业常用位号表达变量和功能，例如 `FT` 可能表示流量变送器、`LIC` 可能表示液位指示控制器。实际命名依项目标准，不能机械套用。

```text
P&ID 位号
 → 仪表数据表/回路图
 → I/O 清单与端子
 → PLC 硬件地址和符号变量
 → 协议对象/寄存器
 → HMI 标签、报警和历史点
```

每次映射都应可追溯到版本和责任方。

## 3. 点表建议字段

| 字段组 | 典型字段 |
| --- | --- |
| 标识 | Tag、描述、区域、设备、回路 |
| 现场 | 仪表型号、量程、单位、失效方向 |
| I/O | 类型、机架、槽位、通道、隔离/安全栅 |
| PLC | 控制器、变量、数据类型、任务、保持性 |
| 通信 | 协议、节点、对象地址、字节/字序、刷新周期 |
| HMI | 标签、画面、显示范围、可写性、权限 |
| 报警 | 限值、优先级、延时、迟滞、抑制规则 |
| 历史 | 采样、死区、压缩、保留周期 |
| 安全 | 许可、联锁、SIS 关联、影响等级 |
| 治理 | 版本、变更单、负责人、验证日期 |

点表不应保存明文密码或不必要的敏感网络凭据。

## 4. 命名与层级

良好命名通常表达区域、设备、变量和用途，例如：

```text
AREA01_TANK02_LEVEL_PV
AREA01_PUMP03_START_REQ
AREA01_PUMP03_RUN_FB
AREA01_PUMP03_LOW_LEVEL_TRIP
```

命名只是辅助。`TRIP` 名称可能只是报警位，仍需检查逻辑。别名、缩写、历史遗留和不同厂商命名会导致歧义，应维护数据字典。

## 5. 数据类型、单位和缩放

点表需要明确：

- 原始整数与工程浮点如何转换；
- 单位是 Pa、kPa、MPa 还是 bar；
- 温度是 ℃ 还是 K；
- 流量是瞬时量还是累计量；
- 百分比代表阀位命令、反馈还是归一化 PV；
- 32 位值的寄存器顺序；
- 枚举值如 0/1/2 分别代表什么模式；
- 无效、未初始化和超量程如何编码。

单位不一致可造成数量级错误，尤其在上层优化、配方和跨系统接口中。

## 6. 质量码与时间语义

值、质量和时间应作为一个整体：

```text
Value = 5.2 MPa
Quality = Good / Bad / Uncertain / Substituted
SourceTime = 10:00:01.125
ServerTime = 10:00:01.480
```

只复制值而丢弃质量和源时间，会让陈旧或替代数据看似正常。跨系统映射需说明质量降级规则和时间戳来源。

## 7. 只读点、可写点与命令对象

可写性不能只按协议读写功能区判断：

- PV 和反馈通常只读；
- SP、模式和配方参数可受控写入；
- 请求位应由握手逻辑处理；
- 输出、步位、旁路和联锁参数属于高风险写点；
- 维护参数仅特定角色和窗口可写；
- 诊断确认与时间设置也可能影响证据和运行。

外部系统若直接写内部状态而不走命令接口，会绕过许可、审计和去重。

## 8. 跨层映射与通信矩阵

一条通信记录应同时描述：

- 源和目的资产；
- 协议、端口、方向与周期；
- 读取或写入的标签集合；
- 命令、状态、历史或工程用途；
- 允许模式与角色；
- 失联和超时行为；
- 物理对象与最大可信后果。

这样防火墙白名单才能从“允许端口”提升到“允许必要控制语义”。

## 9. 点表版本与变更

常见漂移包括：

- I/O 模块替换后通道变化；
- PLC 数据块重构导致地址改变；
- HMI 标签仍指向旧变量；
- 协议网关缩放或字序改变；
- 报警限修改但文档未更新；
- 历史点重命名造成趋势断裂；
- 新增设备复用旧标签导致语义冲突。

点表应与工程版本一起发布，并通过自动比较和现场点对点测试验证。

## 10. 点对点与回路验证

验证流程通常从现场激励开始：

1. 确认设备与安全条件；
2. 在现场施加受控输入或模拟信号；
3. 检查 I/O 原始值和诊断；
4. 检查 PLC 工程量、方向和质量；
5. 检查 HMI 数值、单位、颜色和报警；
6. 检查历史采样和时间；
7. 对输出执行受控动作并确认反馈；
8. 记录结果、异常和恢复状态。

生产输出测试需经过许可，可能使用隔离、模拟或检修窗口。

## 11. 安全视角：点表泄露与篡改

点表安全风险既包括机密性泄露，也包括完整性和语义一致性破坏。一个地址仍可正常通信，但若单位、类型、质量或物理对象被改，控制系统会在错误语义下正常运行。

### 11.1 泄露

点表暴露设备、工艺、可写对象、联锁和地址，可显著降低攻击者理解控制语义的成本。应按敏感工程资料保护。

### 11.2 映射篡改

把 HMI 按钮绑定到错误设备、交换两个阀门地址或改变字序，会造成合法命令作用于错误对象。

### 11.3 语义降级

网关只传值、不传质量和时间，会让上层无法判断数据陈旧。安全性在接口转换中悄然下降。

### 11.4 权限错误

将只读 PV 暴露为可写，或允许上层批量写配方和模式，会扩大攻击面。

### 11.5 点表攻击链

```text
取得工程、网关或组态权限
 → 修改地址、类型、缩放、可写性、质量或标签绑定
 → 数据流指向错误对象或丢失语义
 → 控制流读取错误值或向错误对象发命令
 → 设备状态流和过程响应偏离
 → 操作员、报警与保护可能被同一错误映射误导
```

验证该攻击链需要比较工程版本、协议字节、PLC 变量、HMI 对象、历史和现场反馈。若只看到点表文件被改，还需证明修改已发布并生效。

## 12. 点表的关系模型

复杂项目不宜只用一张超宽表。可拆分为关联实体：

- `PhysicalAsset`：设备、管线、单元和位置；
- `Instrument`：位号、量程、单位和校准；
- `IOChannel`：机架、槽位、通道和电气类型；
- `ControllerTag`：变量、类型、任务和保持性；
- `ProtocolObject`：节点、地址、编码和读写方向；
- `HMITag`：画面、权限、显示和命令；
- `Alarm`：限值、优先级和状态规则；
- `HistoryPoint`：采样、压缩和保留；
- `SafetyRelation`：许可、联锁、SIF 和后果。

每个实体使用稳定唯一 ID，名称可变但关系可追踪。导出给不同团队时只提供所需字段，减少敏感信息暴露。

## 13. 自动语义校验

可以用脚本或数据库规则发现：

- 同一 I/O 通道映射多个不兼容变量；
- 标签类型与协议长度不一致；
- AI 缺少单位或量程；
- 只读测量被配置为可写；
- HMI、报警或历史引用不存在的标签；
- 高报警不高于低报警；
- 质量和源时间在网关映射中丢失；
- 安全相关点未经批准发生变化；
- 同一位号在两个工程版本中指向不同设备。

自动检查只能发现结构错误，不能判断工艺限值是否合理，仍需专业复核。

## 14. 点表与监测规则生成

版本化点表可生成通信白名单、标签写入分级、异常检测上下文和证据报告。生成流程需保留源版本和人工批准，避免错误点表自动传播到防火墙和监测系统。

监测结果应显示“谁在何时写了哪个过程对象、当时模式与许可是什么、反馈和过程怎样响应”，而不只是寄存器地址。

## 15. 工控场景：两个阀门标签交换

检修后 HMI 中进料阀 A 与排放阀 B 的命令标签被交换。画面上点击“关闭进料”实际关闭排放，进料仍持续。反馈标签也被同样交换，界面看似动作成功。

单纯检查 HMI 命令和反馈无法发现，需要结合 PLC 地址、现场阀位、流量变化和物料平衡。点对点验证若只在软件层完成，同样会漏检。

## 16. 防护与治理

- 点表按敏感工程资产实施访问控制、备份和版本管理；
- 变量、单位、质量、时间和权限作为整体设计；
- 外部写入只暴露受控命令接口，不直接暴露内部状态；
- 地址或类型变更自动检查 HMI、报警、历史和网关引用；
- 关键点执行端到端点对点和回路测试；
- 通信监测使用点表解释命令语义和物理对象；
- 对导出点表脱敏，移除不必要的真实地址与凭据。

## 17. 面试表达

### 60 分答案

> 点表记录仪表位号、I/O 地址、PLC 变量、量程、单位、报警和 HMI 映射，是现场设备与控制系统之间的数据字典。

### 90 分答案

> 我会把点表作为跨层语义模型，不只记录地址，还记录数据类型、单位、缩放、质量、源时间、读写方向、保持性、报警、联锁和物理后果。协议抓到的寄存器必须通过版本化点表映射到 PLC 变量、HMI 标签和现场设备，再判断写入是否经过命令握手与许可。验证时要从现场激励一直检查到历史归档，并从输出命令回到真实阀位和过程响应。

### 常见追问

**问：为什么地址正确仍可能显示错误？**

答：数据类型、字节序、缩放、单位、质量或 HMI 二次转换可能错误。

**问：点表为什么是敏感资料？**

答：它直接暴露控制对象、可写变量、联锁和协议地址，为精确操纵提供语义地图。

## 18. 实操建议

### 实验一：构建跨层点表

为水箱建立液位、进出阀和泵点表，覆盖现场、I/O、PLC、协议、HMI、报警和历史字段。

### 实验二：端到端验证

改变仿真液位，记录原始值、工程量、协议字节、HMI 和历史；故意制造单位或字序错误并定位。

### 实验三：写点分级

把全部变量分为只读、普通参数、受控命令和高后果维护对象，分别设计权限、握手和审计要求。

### 实验四：自动校验

用 CSV 和简单脚本检查重复地址、缺失单位、非法报警范围和可写 PV。保留输入版本、规则版本和结果，人工复核误报与漏报。

### 实验五：关系追踪

选择一个阀门，从物理设备 ID 追踪到 I/O、PLC、协议、HMI、报警和历史，再从任一层反向查询。故意重命名标签，验证稳定 ID 是否保持关系。

## 19. 总结与检查清单

- [ ] 能建立从 P&ID 到历史点的完整映射；
- [ ] 能解释类型、单位、缩放、质量和时间；
- [ ] 能区分状态、参数、命令和高风险写点；
- [ ] 能把点表与通信矩阵和物理后果关联；
- [ ] 能管理地址、标签和报警配置的版本漂移；
- [ ] 能执行现场到应用的端到端验证。
