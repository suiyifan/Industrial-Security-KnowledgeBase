# PLC 工程工作流深挖

这一节模拟工程师日常如何使用 PLC 工程软件。理解这些动作，才能判断哪些安全事件真正危险。

## 1. PLC 工程生命周期

典型流程：

```text
新建项目
-> 配置硬件
-> 配置通信
-> 编写变量表
-> 编写控制逻辑
-> 编译
-> 下载到 PLC
-> 在线监控
-> 调试
-> 备份归档
```

安全分析要关注每一步是否有权限、审计和完整性保护。

## 2. 新建项目

工程项目通常包含：

- PLC 型号
- CPU 版本
- I/O 模块
- 通信模块
- 网络地址
- 程序块
- 数据块
- 变量表
- 硬件组态

如果工程文件泄露，攻击者可能知道现场设备型号、地址、变量和逻辑结构。

## 3. 硬件组态

硬件组态记录：

- CPU 型号
- 机架
- 槽位
- DI/DO/AI/AO 模块
- 通信模块
- 电源模块
- 模块地址

安全价值：

- 判断 PLC 能控制哪些现场设备。
- 判断是否有以太网通信模块。
- 判断哪些模块可能对应关键控制。

## 4. 通信配置

通信配置可能包含：

- PLC IP 地址
- 子网掩码
- 网关
- 远程站点地址
- 通信协议
- 连接对象
- OPC Server 配置
- HMI 连接配置

安全关注：

- 是否暴露到办公网。
- 是否有跨网段通信。
- 是否存在未使用但开启的服务。
- 是否存在默认站号或默认端口。

## 5. 程序块

PLC 工程常按块组织。

常见块：

- 主程序
- 功能块
- 数据块
- 中断程序
- 通信处理块
- 设备控制块
- 报警处理块

阅读工程时，优先看：

- 设备启停逻辑。
- 联锁条件。
- 报警条件。
- 模式切换。
- 通信读写。
- 安全相关输出。

## 6. 下载到 PLC

下载是高风险动作。

下载可能影响：

- 控制逻辑
- 数据块初值
- 硬件配置
- 通信配置
- 运行状态

安全问题：

- 下载是否需要密码？
- 下载前是否有审批？
- 下载后是否有日志？
- 下载是否要求停机？
- 是否能比对工程版本？

## 7. 上传工程

上传是从 PLC 读取当前工程。

安全风险：

- 攻击者获取控制逻辑。
- 攻击者获取变量和点表。
- 攻击者理解联锁条件。
- 攻击者为后续修改做准备。

保护措施：

- 上传保护。
- 工程密码。
- 工程文件加密。
- 访问控制。

## 8. 在线监控

在线监控可以看到变量实时值。

用途：

- 调试逻辑。
- 观察输入输出。
- 判断设备状态。

安全风险：

- 泄露现场状态。
- 帮助攻击者定位关键变量。
- 与 Force 功能结合会非常危险。

## 9. 变量强制

Force 常用于调试，但在生产中风险很高。

例子：

```text
正常逻辑要求液位达到条件才启动泵。
Force 可以绕过输入条件，直接强制输出启动。
```

安全分析要记录：

- 是否允许 Force。
- Force 是否有权限控制。
- Force 是否在 HMI 或工程站有提示。
- Force 是否记录日志。
- Force 是否能被联锁覆盖。

## 10. 版本管理

真实项目里常见问题：

- 工程文件散落在工程师电脑。
- 文件名用“最终版”“最终最终版”。
- 现场 PLC 程序和备份工程不一致。
- 修改没有审批。
- 回滚版本不可用。

安全建议：

- 建立工程版本库。
- 每次下载前后备份。
- 工程文件加密存储。
- 控制变更审批。
- 定期校验 PLC 程序与基线一致性。

## 11. 面试可讲版本

> 我理解 PLC 工程不是只有一段程序，而是包含硬件组态、通信配置、变量表、程序块和数据块。工程站可以进行上传、下载、在线监控和变量强制，这些动作在安全上非常敏感。评估时我会重点关注工程下载权限、工程文件保护、Force 使用记录、版本管理和控制逻辑完整性。
