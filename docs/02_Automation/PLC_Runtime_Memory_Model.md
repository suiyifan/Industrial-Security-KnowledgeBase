# PLC 运行时与内存模型

PLC 程序最终操作的是运行时中的变量、过程映像、实例状态和系统对象。网络协议中的“读寄存器”或“写标签”只有映射到具体内存语义后，才能判断是否会改变设定值、模式、联锁、步序或物理输出。不同厂商地址体系差异很大，本章建立跨平台分析方法。

## 1. 原理解释：为什么 PLC 需要多类内存

控制程序既要保存当前输入和输出，也要保存定时器、计数器、PID 积分、设备状态、配方、累计量和诊断信息。不同数据对更新时机、掉电保持、访问权限和一致性要求不同，因此不能都放在一个无差别的“寄存器区”。

分析任一变量时回答六个问题：

1. 谁创建和定义它；
2. 谁在何时读取它；
3. 谁可以写它；
4. 写入能维持多久；
5. 重启或下载后是否保留；
6. 它怎样映射到现场设备和上层系统。

## 2. 工程实现：典型内存类别

| 类别 | 典型内容 | 更新来源 | 关键风险 |
| --- | --- | --- | --- |
| 输入过程映像 | DI、AI、远程 I/O 输入 | I/O 系统 | 冻结、替代、异步更新 |
| 输出过程映像 | DO、AO 目标值 | 用户程序/强制 | 多处写入、刷新时机 |
| 内部变量 | 中间结果、模式、命令、状态 | 用户程序/通信 | 语义不透明、权限过宽 |
| 实例数据 | 定时器、PID、设备块内部状态 | 功能块 | 隐藏状态、跨扫描记忆 |
| 保持数据 | 累计量、配方、步号、校准参数 | 程序/工程操作 | 重启持久化、污染恢复 |
| 系统区 | 时钟、诊断、任务和硬件状态 | 运行时 | 非法访问、平台依赖 |
| 通信缓冲 | 接收/发送、连接状态、协议对象 | 通信任务 | 并发、长度、陈旧数据 |
| 工程/装载存储 | 编译程序、组态、初始值 | 工程下载 | 离线篡改、版本不一致 |

## 3. 装载存储、工作存储和保持存储

不同 PLC 会把工程的持久副本、CPU 执行时的工作数据和掉电保持数据分开：

- **装载存储**保存可重新装入的程序与初始数据；
- **工作存储**保存正在执行的代码和当前变量；
- **保持存储**在掉电或特定重启后保留选定状态。

在线修改可能只改变工作存储，随后需显式复制到持久存储；也可能自动持久化。若只检查离线工程，可能看不到在线差异；若只读取在线值，也不能证明重启后仍会存在。

## 4. 过程映像与直接 I/O

输入映像提供扫描快照，输出映像保存待刷新的目标值。直接 I/O 则绕过映像访问模块当前值，适合高速或特殊需求，但会破坏“同一扫描一致”的直觉。

```text
物理输入 → 模块缓冲 → 输入映像 → 程序变量
程序结果 → 输出映像 → 模块缓冲 → 物理输出
```

远程 I/O 还会附带连接状态、更新时间和质量。变量值不变可能是过程稳定，也可能是连接中断后保持最后值。

## 5. 符号变量与绝对地址

老式项目常用绝对地址，如输入字、内部位和数据寄存器；现代平台更多使用符号标签和结构体。

- 绝对地址便于协议映射，但重构后含义容易漂移；
- 符号变量可表达语义和类型，但协议客户端可能通过外部访问表映射；
- 优化数据块可能没有固定偏移，不能凭旧地址解释；
- 别名和引用可能让多个名称指向同一对象；
- HMI 标签可能再次做缩放、取反或脚本转换。

点表必须记录版本，地址相同不保证不同工程版本中的语义相同。

## 6. 数据类型、对齐与字节序

PLC 常用 BOOL、整数、浮点、时间、日期、字符串、数组、结构体和枚举。解析需注意：

- BOOL 可能按位打包；
- 16/32/64 位值可能跨多个寄存器；
- 字节序与字序可能分别变化；
- 有符号整数与无符号整数范围不同；
- 浮点异常值、精度和缩放影响比较；
- 字符串可能包含最大长度、当前长度和字符区；
- 结构体可能因对齐和优化出现填充。

协议工具读出 `0x3F800000` 只是字节证据，只有结合类型、顺序和单位才能解释为浮点 1.0。

## 7. 功能块实例与隐藏状态

定时器、计数器、PID 和设备控制功能块具有跨扫描状态。例如定时器不仅有到达位，还保存累计时间和运行条件；PID 保存积分、上次误差、模式和限幅状态。

直接修改块输出而不修改内部状态，下一扫描可能被重新计算；修改实例状态则可能持续改变行为。恢复也必须考虑整个实例，而非单一输出变量。

## 8. 保持变量、初始值与启动类型

典型启动语义包括冷启动、暖启动和热重启，但名称与保留范围依厂商而异：

- **初始值**来自工程定义或装载存储；
- **当前值**是运行中的实际状态；
- **保持值**在特定掉电或重启后继续使用；
- **首次扫描逻辑**可重新初始化部分对象。

保持累计量通常合理，保持直接输出或危险步号则可能不安全。一次程序下载还可能覆盖当前值、保留当前值或要求重新初始化，需要在变更前确认。

## 9. 通信变量与一致性

HMI、SCADA、控制器间通信和工程软件可能并发访问数据。风险包括：

- 32 位值分两次写入，程序读到“撕裂”中间态；
- 结构体部分更新，版本号和数据不一致；
- 通信写入后被周期逻辑覆盖；
- 握手位未按顺序更新，命令被重复执行；
- 生产者数据过期但消费者仍使用；
- 时间戳或质量未与值一起更新。

常见工程方法包括双缓冲、有效标志、序列号、校验、写完成握手和原子复制。协议分析应识别这些状态，而不是把每个寄存器独立解释。

## 10. 命令与状态的握手机制

可靠远程命令通常不只写一个位：

```text
客户端写入命令参数和序列号
 → 置 Command_Request
 → PLC 校验模式、权限语义和许可
 → 执行动作
 → 写 Command_Ack、结果码和相同序列号
 → 客户端清除请求
```

若只重放请求位，可能因序列号或状态不匹配而无效；若设计没有去重，重放则可能重复投料或重复动作。

## 11. 安全视角：内存写入如何分类

内存安全风险不能只按“可读/可写”二分。还要判断变量更新时序、保持性、是否参与边沿或握手、是否映射物理输出，以及写入能否绕过正常控制流。

### 11.1 瞬态写入

写入被程序下一扫描覆盖，可能无物理效果，也可能在一个周期内触发边沿逻辑。不能用“值马上恢复”证明无风险。

### 11.2 持久状态写入

修改配方、累计量、保持步位、校准或控制参数，可能跨重启存在，并污染后续恢复。

### 11.3 直接输出与强制

直接 I/O、输出映像或强制通常更接近物理动作，但仍受 CPU 模式、模块状态、电气回路和执行器反馈影响。

### 11.4 系统与诊断对象

修改时钟、通信连接、任务参数或诊断确认可能影响证据、调度和可用性。其风险依具体平台能力验证。

### 11.5 内存攻击链

```text
识别协议对象或符号标签
 → 推断类型、字节序、保持性和写入者
 → 写入命令、参数、状态或实例数据
 → 任务读取并触发控制流/状态流变化
 → 输出映像、通信对象或保持区改变
 → 现场设备与过程响应
```

若攻击者只知道绝对地址而不知道工程版本，写入可能无效、破坏无关数据或触发意外故障。工程语义既提升攻击精度，也是防守侧判断风险的必要条件。

## 12. 快照、一致性与取证

读取在线 PLC 数据时需要考虑一致性：逐个读取变量可能跨越多个扫描，所得组合从未在同一时刻真实存在。结构体、状态机和批次数据应尽量使用平台快照、原子复制或带序列号的诊断缓冲。

事件取证建议同时保存：

取证过程本身也是数据流：控制器内存经工程协议和采集工具进入证据文件，每一步都要记录时间、账号、工具版本和完整性校验。

- 离线工程与哈希；
- 在线程序和硬件比较结果；
- 当前值、初始值和保持值；
- 强制、旁路、模式和任务诊断；
- 控制器时间、时区、启动原因和诊断缓冲；
- I/O 质量、远程连接和现场反馈；
- 采集方法、账号、时间和是否改变状态。

禁止为了取证随意上传、停止 CPU 或清除诊断。先确认操作是否会改变控制器状态和证据。

## 13. 数据生命周期与清理

敏感数据不仅存在在线 CPU：

- 工程备份保留符号、注释和初始凭据；
- 存储卡可能保存程序与固件；
- 更换下来的 CPU 和模块仍有保持数据；
- 工程站缓存在线值、下载包和日志；
- 厂商支持包可能包含完整诊断和网络信息。

设备退役或送修前应按厂商支持的方法清理，并保留必要的合规和事件证据。简单删除工程文件不一定擦除持久介质。

## 14. 工控场景：保持步号导致重复投料

批次控制器掉电前处于“投料完成、准备加热”。步号被保持，但流量累计量在重启时清零。恢复逻辑误认为尚未达到目标量，再次打开进料阀，造成重复投料。

风险并非单一变量错误，而是保持策略不一致：步序、累计量、阀门位置和真实物料状态没有共同恢复。正确恢复需要人工或独立测量确认釜内物料，并使用事务化批次记录。

## 15. 防护与工程治理

- 建立变量字典，记录类型、地址、单位、读写者、保持性和物理语义；
- 最小化外部可写标签，分离命令、参数、状态和诊断；
- 高风险命令使用许可、序列号、结果码和超时握手；
- 对在线值、初始值、保持值和离线工程做一致性比较；
- 监测强制表、保持区变化和重启后的异常初始状态；
- 对多字和结构化数据使用原子更新或版本校验；
- 恢复时验证程序状态、现场反馈和物料状态一致。

## 16. 面试表达

### 60 分答案

> PLC 内存包括输入输出映像、内部变量、数据块、保持区和系统诊断。输入映像来自现场，程序写输出映像，保持变量在特定重启后继续存在。

### 90 分答案

> 我分析 PLC 内存时不会把所有对象都叫寄存器，而会确认变量的类型、地址、更新任务、写入者、保持性和物理映射。还要区分装载存储、运行工作值、初始值和保持值，以及过程映像与直接 I/O。网络写入是否有效取决于是否被程序覆盖、是否参与边沿或握手、是否跨重启保留。恢复时要同时比较离线工程、在线实例状态、强制表和真实现场状态，避免数字状态与物料状态不一致。

### 常见追问

**问：写入一个位后立刻恢复，是否说明攻击失败？**

答：不一定，该位可能在恢复前触发边沿、锁存或状态迁移；需要看任务时序和下游状态。

**问：地址相同为什么语义可能变化？**

答：工程版本、数据块布局、优化方式、别名或 HMI 映射可能改变，地址必须与版本化点表关联。

## 17. 实操建议

### 实验一：保持与非保持变量

在仿真 PLC 中建立步号、累计量和输出三类变量，分别配置保持策略，模拟重启并记录一致性问题。

### 实验二：通信覆盖

让 HMI/脚本写入一个由 PLC 周期计算的变量，观察写入窗口和覆盖；再改为请求—确认握手，比较可靠性。

### 实验三：字节与类型解析

构造 BOOL、INT、DINT、REAL 和结构体样本，用抓包或脚本按不同字节序解析，记录错误解释会怎样影响工程量。

### 实验四：一致性快照

让状态机在多个变量间更新状态、序列号和累计量，使用逐变量读取与一次原子快照两种方式采集。统计逐变量读取出现不一致组合的条件，并设计版本号重读机制。

## 18. 总结与检查清单

- [ ] 能区分过程映像、内部变量、实例、保持区和系统区；
- [ ] 能解释装载值、当前值、初始值和保持值；
- [ ] 能处理数据类型、对齐、字节序和结构体；
- [ ] 能分析通信并发、握手和多字一致性；
- [ ] 能判断瞬态写入、持久写入和直接输出的差异；
- [ ] 能设计重启后数字状态与物理状态的一致性验证。
