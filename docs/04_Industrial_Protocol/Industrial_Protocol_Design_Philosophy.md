# 工业协议设计思想

## 1. 原理解释：工业协议和互联网协议的差异

互联网协议通常面向开放网络、兼容性、吞吐量和尽力交付。

工业协议面向控制网络，更关注：

- 确定性
- 低时延
- 简单实现
- 可靠控制
- 设备资源有限
- 与现场变量绑定

工业协议传输的不是普通网页数据，而是控制变量、设备状态、设定值、报警和工程操作。

## 2. 工业协议解决的四件事

1. 设备识别：谁和谁通信。
2. 数据表示：寄存器、变量、节点、数据块如何编码。
3. 通信时序：谁先说、谁后说、是否允许冲突。
4. 错误恢复：CRC、异常码、超时、重试。

## 3. 为什么工业协议喜欢“变量访问模型”

工业控制系统的核心是变量：

- 温度 PV
- 压力 SP
- 阀门开度
- 电机运行状态
- 报警位
- 设备模式

不同协议用不同抽象访问变量：

| 协议 | 抽象模型 |
|---|---|
| Modbus | Coil / Register |
| S7Comm | I/Q/M/DB 内存区 |
| OPC UA | Node 节点 |
| EtherNet/IP | Object / Attribute |
| Profinet | 设备、模块、周期数据 |

本质上都是远程访问工业状态。

## 4. 数据流：一次工业变量读取

```text
HMI / SCADA
  -> 通信驱动
  -> 工业协议请求
  -> PLC / RTU / 控制器
  -> 读取变量 / 寄存器 / 节点
  -> 返回响应
  -> HMI 画面刷新
```

写操作则可能进一步影响控制逻辑：

```text
HMI 写 SP
  -> 协议写请求
  -> PLC DB / Register 改变
  -> PID 使用新 SP
  -> 控制输出变化
  -> 物理过程变化
```

## 5. 安全视角

早期工业协议常默认控制网络可信，因此普遍存在：

- 缺少认证
- 缺少加密
- 缺少完整性保护
- 缺少细粒度权限
- 读写功能过强
- 工程操作暴露

攻击本质不是“发了一个包”，而是远程改变控制变量和设备状态。

## 6. 工业案例：读写权限差异

同样访问 PLC：

- 读 PV：泄露生产状态。
- 写 SP：改变控制目标。
- 写 DO：改变设备启停。
- 下载程序：改变控制逻辑。

风险等级完全不同。

## 7. 面试表达

### 60 分答案

> 工业协议的核心是让上位机、控制器和现场设备交换工业变量。不同协议模型不同，但本质都是访问设备状态、过程值、设定值和控制命令。

### 90 分答案

> 我会从变量访问模型理解工业协议。Modbus 是寄存器模型，S7 是内存区和 DB 模型，OPC UA 是节点模型。安全分析时，我不会只看端口开放，而会判断协议操作是读还是写，写的是设定值、命令、报警屏蔽还是工程数据，因为这些操作可能直接影响控制闭环和物理过程。

### 常见追问

问：工业协议为什么很多没有认证？

答：

> 很多早期工业协议设计时默认运行在封闭可信的控制网络中，优先考虑简单、实时和可靠，没有按互联网威胁模型设计认证和加密。现在接入以太网和远程运维后，这些假设被打破。

## 8. 实操建议

后续练习：

1. 选择 Modbus TCP 作为入门协议。
2. 抓取读寄存器和写寄存器报文。
3. 对比读操作和写操作的功能码。
4. 将寄存器地址映射到点表。
5. 写报告说明“协议操作 -> 控制变量 -> 物理影响”。
