# 限制与验证

## 1. AI 的典型风险

- 幻觉：编造不存在的事实。
- 过期：不知道最新版本或补丁。
- 上下文不足：没有完整代码或环境。
- 误判：把正常逻辑当漏洞。
- 过度自信：不给限制条件。
- 安全风险：生成危险操作建议。

## 2. 验证原则

所有安全结论至少需要一种证据：

- 官方公告。
- 源码或反编译证据。
- 抓包证据。
- 日志证据。
- 调试证据。
- 复现截图。
- 版本信息。

高风险结论最好有两类证据交叉验证。

## 3. 事实、推测、待验证

写笔记时建议分三栏：

| 类型 | 含义 |
| --- | --- |
| 事实 | 已由证据确认 |
| 推测 | 根据样本或代码推断 |
| 待验证 | 需要实验或资料确认 |

## 4. 面试表达模板

> 我使用 AI 时最重视验证。AI 的输出会被我拆成事实、推测和待验证三类，只有有证据支撑的内容才进入报告。比如漏洞影响版本必须查公告或实际环境，协议字段必须通过多组样本或代码验证，PoC 安全性必须人工审计。
