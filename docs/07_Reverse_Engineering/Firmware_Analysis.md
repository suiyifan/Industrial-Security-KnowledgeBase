# 固件分析

## 1. 固件在工控安全中的意义

PLC、RTU、工业网关、交换机、摄像头、边缘计算盒子、协议转换器通常运行嵌入式固件。固件里可能包含：

- Web 管理后台。
- Telnet/SSH/FTP 服务。
- 私有协议服务。
- 默认账号和硬编码口令。
- 升级校验逻辑。
- 设备控制脚本。
- 证书、密钥和云平台地址。

固件分析可以帮助判断设备真实攻击面，而不是只依赖说明书或端口扫描。

## 2. 固件分析流程

### 2.1 获取和记录样本

记录：

- 厂商、型号、硬件版本。
- 固件版本。
- 下载来源。
- 文件哈希。
- 文件大小。
- 发布时间。

这些信息对报告和 CVE 复现很重要。

### 2.2 识别文件格式

常见格式：

- 压缩包：zip、tar、gz、xz。
- 文件系统：SquashFS、CramFS、JFFS2、UBIFS。
- Bootloader 镜像：uImage、FIT image。
- 裸二进制：需要手动找偏移。
- 加密/签名升级包：需要分析解包器或更新程序。

常用工具：

- `file`
- `binwalk`
- `strings`
- `hexdump`
- `7z`
- `firmware-mod-kit`

### 2.3 解包后看什么

解包出文件系统后，优先看：

- `/etc/passwd`、`/etc/shadow`
- `/etc/init.d/`
- `/etc/rc.d/`
- `/etc/config/`
- `/www/`、`/var/www/`
- `/bin/`、`/sbin/`
- `/usr/bin/`、`/usr/sbin/`
- Web CGI、Lua、PHP、Go/Rust/C 二进制。
- 证书、私钥、配置模板。

启动脚本可以告诉你设备开机后启动了哪些服务，监听哪些端口，使用哪些配置。

## 3. Web 后台分析

固件中的 Web 后台常见风险：

- 默认账号。
- 硬编码密码。
- 登录校验只在前端完成。
- API 缺少鉴权。
- 命令注入。
- 任意文件读取。
- 固件上传校验弱。
- 会话 Cookie 可预测。

分析路径：

1. 找 Web 根目录。
2. 找路由、CGI、API 处理器。
3. 找登录函数和会话校验函数。
4. 找危险调用：`system`、`popen`、`execve`。
5. 判断参数是否来自 HTTP 请求。
6. 构造最小请求验证。

## 4. 服务二进制分析

固件中的协议服务通常是 ELF 文件。分析时关注：

- 架构：ARM、MIPS、PowerPC。
- 是否 stripped。
- 动态链接还是静态链接。
- 启动参数来自哪里。
- 监听端口。
- 协议字符串和错误码。
- 调用了哪些系统函数。

如果不能直接运行，可以用 QEMU 用户态或系统态模拟，也可以只做静态分析加真实设备验证。

## 5. 升级机制分析

升级机制是高价值入口，因为它关系到持久化和供应链风险。

重点问题：

- 固件包是否签名。
- 签名校验在哪里完成。
- 是否只做 CRC 或 MD5。
- 是否允许降级。
- 是否校验设备型号。
- 解包路径是否可能目录穿越。
- 升级脚本是否拼接命令。

弱升级机制可能导致攻击者上传恶意固件、植入后门或破坏设备启动。

## 6. 工控设备固件的安全影响

固件漏洞要落到设备能力：

- 能否读取或修改控制配置。
- 能否进入维护模式。
- 能否中断协议转发。
- 能否篡改遥测数据。
- 能否改变网关路由或白名单策略。
- 能否作为跳板进入生产网。

## 7. 面试表达模板

> 我做固件分析时会先记录设备型号、固件版本和哈希，然后用 binwalk、file、strings 判断固件结构。解包后我会先看启动脚本、Web 目录、配置文件和服务二进制，确认真实启动的服务和端口。对 Web 后台我重点看认证、命令执行、文件操作和升级接口；对协议服务我会结合字符串、导入函数和监听端口定位解析逻辑。最后我会把风险和设备能力关联起来，比如是否能修改配置、影响协议转发或作为内网跳板。

## 8. 实操验证

建议练习：

- 下载一个公开路由器或工业网关固件。
- 用 binwalk 解包。
- 找到启动脚本和 Web 根目录。
- 列出启动服务。
- 搜索 `password`、`system(`、`upgrade`、`telnet`。
- 写一页攻击面分析。
