# 面试问答

## Q1：你如何开展一个未知二进制的逆向分析？

我会先做样本画像，确认文件类型、架构、编译器、是否加壳、导入表、字符串和运行环境。然后根据目标找入口，比如网络输入、文件解析、登录认证、升级接口或协议处理。静态上用 IDA/Ghidra 从字符串和 API 交叉引用恢复调用链、命令分发和状态机；动态上用 x64dbg/WinDbg 在输入读取、危险函数和关键分支处下断点。最后用最小输入、抓包或日志验证结论，并输出安全影响和限制条件。

## Q2：IDA/Ghidra 中你最常用哪些分析入口？

我最常用字符串交叉引用、导入函数交叉引用、导出函数、网络收发函数、文件读取函数和错误提示字符串。工控软件里我会特别关注 `download`、`upload`、`force`、`run`、`stop`、`Modbus`、`PLC`、`OPC` 这类业务字符串，因为它们往往能连接到真实控制动作。

## Q3：看到危险函数就能判断漏洞吗？

不能。危险函数只是风险线索。需要继续确认参数是否外部可控、是否经过长度校验、目标缓冲区大小、是否能稳定触发异常、异常是否造成可利用影响。比如看到 `memcpy`，我会在动态调试中确认目标地址、源地址、长度分别来自哪里。

## Q4：你如何做协议逆向？

我会先抓完整正常流程，包括连接、登录、读写、下载等动作，然后对比多组流量恢复固定头、长度、序列号、命令码和校验字段。接着在客户端或服务端程序里找 `recv`、功能码分发、CRC、错误字符串，用交叉引用恢复解析逻辑。最后把流量字段和程序逻辑互相验证，输出字段表、命令码表、状态机和安全风险。

## Q5：工控逆向和普通软件逆向有什么区别？

工控逆向更强调工程语义和物理影响。普通软件逆向可能关注权限、数据、执行流，工控还要进一步判断这个漏洞是否能影响控制回路、设备启停、设定值、联锁、报警或生产连续性。所以报告中不能只写技术漏洞，还要写攻击条件、工艺影响、恢复方式和安全边界。

## Q6：如何分析固件中的 Web 漏洞？

我会先解包固件，找启动脚本确认真实启动的 Web 服务，再定位 Web 根目录、路由或 CGI 程序。之后重点看认证流程、会话校验、文件上传、升级接口和危险函数调用，例如 `system`、`popen`、`execve`。如果发现参数进入命令执行函数，会继续确认是否有白名单校验、接口是否需要登录、进程权限和设备网络位置。

## Q7：如何判断一个崩溃是否有安全价值？

我会先记录异常类型、崩溃地址、调用栈和触发输入，判断是读越界、写越界、执行异常、空指针还是主动断言。然后看输入是否外部可控、是否稳定复现、是否影响服务可用性。如果是写越界或控制流异常，会进一步分析保护机制和可控程度；如果只是空指针，也可能定性为拒绝服务。

## Q8：你如何分析补丁前后差异？

先确认两个版本的样本信息和函数匹配关系，然后用 BinDiff、Diaphora 或手动比对定位变化函数。重点看新增的长度检查、权限检查、状态判断、过滤逻辑和错误处理。之后回到旧版本确认缺失逻辑是否可触发，再用 PoC 或最小样本复现，最后总结根因和修复方式。

## Q9：PLC 程序逆向关注什么？

我会关注工程文件、下载流量和控制逻辑。工程文件里看变量表、I/O 映射、程序块、通信配置和权限；下载流量里看认证、程序块传输和校验；控制逻辑里按输入、处理、联锁、输出还原工艺链路。安全上重点关注未授权上传/下载、明文传输、重放和维护旁路。

## Q10：你如何把逆向经历写进项目？

我会写成 “目标、方法、证据、结论” 四段。目标说明分析对象和问题；方法说明用了静态分析、动态调试、抓包、差分等手段；证据包括关键函数、字段表、调用链、调试记录；结论说明风险、触发条件、影响范围和修复建议。这样面试官能看到不是只跑了工具，而是完成了可复盘的研究过程。
