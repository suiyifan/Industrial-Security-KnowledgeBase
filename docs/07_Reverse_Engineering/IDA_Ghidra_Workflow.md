# IDA/Ghidra 工作流

## 1. 工具定位

IDA 和 Ghidra 都是静态逆向分析工具。它们的核心价值是把二进制恢复成可阅读的函数、控制流、交叉引用和伪代码。

在工控安全中，常见用途包括：

- 分析工程软件的登录、授权、下载、上传逻辑。
- 分析协议库的功能码解析、长度校验、状态机。
- 分析固件中的 Web 服务、命令执行点、硬编码凭据。
- 对 CVE 补丁前后版本做差异分析。

## 2. 第一次打开样本做什么

### 2.1 确认加载是否正确

检查：

- 架构是否识别正确。
- 基址是否合理。
- 函数数量是否异常。
- 字符串是否可读。
- 导入表是否正常。
- 是否可能加壳或压缩。

如果一个 Windows 程序只有很少函数、字符串很少、节区熵很高，要怀疑加壳。

### 2.2 从高价值入口开始

不要从入口点一路 F5 到天荒地老。优先入口：

- 字符串交叉引用。
- 导入函数交叉引用。
- 导出函数。
- 网络收发函数。
- 文件读取函数。
- 错误提示字符串。
- 协议关键字。

示例：看到字符串 `Invalid function code`，可以查看交叉引用，通常会定位到协议解析或错误处理函数。

## 3. 函数重命名方法

函数名是逆向项目的 “知识库”。重命名要体现证据强度：

- `parse_modbus_header`：证据较强，确认是在解析 Modbus 头。
- `maybe_check_login_state`：证据中等，疑似登录状态检查。
- `sub_recv_loop`：暂时只知道是接收循环。

推荐命名层次：

| 前缀 | 含义 |
| --- | --- |
| `parse_` | 解析输入 |
| `build_` | 构造输出 |
| `check_` | 校验条件 |
| `handle_` | 处理请求或事件 |
| `send_` | 发送数据 |
| `recv_` | 接收数据 |
| `dispatch_` | 分发命令 |
| `maybe_` | 证据不足的推测 |

## 4. 交叉引用分析

交叉引用分两类：

- 谁引用了这个字符串、函数、全局变量。
- 这个函数又调用了谁、读取了什么、写了什么。

典型路径：

1. 字符串 `download project`。
2. 找到引用函数 `handle_download_button`。
3. 继续找到调用的 `build_download_packet`。
4. 找到 `send_packet`。
5. 结合抓包确认协议字段。

这条路径能把 UI 操作、业务函数、协议数据包串起来。

## 5. 控制流与状态机

工控协议和工程软件常见大量 `switch/case`：

- 功能码分发。
- 命令码分发。
- 状态分发。
- 错误码处理。
- 设备类型适配。

分析时建议把 `switch` 表整理成表格：

| 命令码 | 函数 | 推测语义 | 权限检查 | 风险 |
| --- | --- | --- | --- | --- |
| `0x01` | `handle_read` | 读变量 | 无 | 信息泄露 |
| `0x05` | `handle_write` | 写变量 | 有 | 参数篡改 |
| `0x10` | `handle_download` | 下载工程 | 疑似缺失 | 高风险 |

状态机要整理为：

`INIT -> CONNECTED -> AUTHED -> ENGINEERING_MODE -> DOWNLOAD_READY`

重点看危险命令是否必须经过认证状态和工程模式。

## 6. 数据结构恢复

很多协议解析函数会使用结构体。恢复结构体可以显著提升理解速度。

例如一个请求上下文可能包含：

- socket 句柄。
- 客户端 IP。
- 会话 ID。
- 用户权限。
- 接收缓冲区。
- 当前状态。
- 设备对象指针。

如果看到同一个基址加不同偏移：

- `[rcx+0x10]`
- `[rcx+0x18]`
- `[rcx+0x20]`

可以推测 `rcx` 是上下文对象，逐步给字段命名。

## 7. 伪代码不要完全相信

反编译伪代码是辅助，不是事实本身。遇到以下情况要回到汇编：

- 指针类型看起来奇怪。
- 循环边界不清晰。
- switch 表被反编译错。
- 优化导致变量复用。
- 调用约定识别错误。
- 结构体字段类型不确定。

安全结论必须能被汇编、动态调试或输入输出行为支撑。

## 8. 面试表达模板

> 我用 IDA/Ghidra 时会先从字符串、导入表、导出函数和网络/文件 API 的交叉引用开始，而不是从入口点硬读。定位到关键函数后，我会按证据强度重命名函数，恢复命令分发表、状态机和关键结构体。对于反编译伪代码，我会在长度判断、危险函数、权限判断这些关键位置回到汇编确认，并用动态调试或抓包验证。

## 9. 实操验证

一次完整静态分析至少输出：

- 目标样本信息。
- 关键字符串列表。
- 关键导入函数列表。
- 5 到 10 个重命名函数。
- 一个命令码或功能码分发表。
- 一条从外部输入到处理函数的调用链。
- 一个安全关注点及验证计划。
