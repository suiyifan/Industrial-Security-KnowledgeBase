# 逆向工程方法论

## 1. 逆向工程到底在解决什么

逆向工程是从已有二进制、固件、脚本、配置、协议流量中恢复设计意图的过程。安全岗位里的逆向目标通常不是完整重写软件，而是定位关键路径：

- 输入从哪里进入程序。
- 程序如何解析输入。
- 哪些字段影响控制流、内存访问、文件操作、命令执行或设备控制。
- 错误输入会不会导致崩溃、越界、认证绕过或危险动作。
- 修复补丁改变了哪一段逻辑。

工控场景的逆向还要多考虑 “物理影响”。同样是一个认证绕过，在普通 Web 系统中可能是数据泄露，在工控系统中可能意味着可以下发启停、修改设定值、强制输出或影响联锁逻辑。

## 2. 通用分析流程

### 2.1 建立样本画像

拿到一个目标后，先不要急着进 IDA。先建立画像：

- 文件类型：PE、ELF、脚本、固件镜像、PLC 工程文件、配置文件。
- 运行平台：Windows、Linux、嵌入式 Linux、RTOS、PLC 专用运行时。
- 架构：x86、x64、ARM、MIPS、PowerPC。
- 是否加壳、压缩、签名、加密。
- 是否有符号、日志、配置、帮助文档、协议说明。
- 是否有网络端口、服务名、默认账号、Web 页面、升级包。

画像的价值是缩小搜索范围。例如一个 Windows 上位机如果导入 `WS2_32.dll`、`WinHTTP.dll`、`sqlite3.dll`，就可以初步判断它可能存在网络通信、HTTP 调用和本地数据库操作。

### 2.2 找入口

入口不是只有程序入口点。安全研究更关心 “外部可控输入入口”：

- 网络入口：socket、HTTP API、RPC、OPC UA、Modbus TCP。
- 文件入口：工程文件、配置文件、日志导入、历史数据导入。
- 设备入口：串口、USB、现场总线、驱动 IOCTL。
- 用户入口：命令行参数、界面输入、脚本接口。
- 更新入口：固件升级包、插件、授权文件、补丁包。

找入口的常用线索：

- 字符串：`login`、`password`、`firmware`、`upload`、`Modbus`、`S7`、`OPC`。
- 导入函数：`recv`、`send`、`CreateFile`、`ReadFile`、`RegOpenKey`、`CreateProcess`。
- 配置键：端口号、IP、设备型号、协议名。
- 交叉引用：从字符串或导入函数向上追溯调用链。

### 2.3 找解析逻辑

解析逻辑通常具备明显特征：

- 对长度字段、类型字段、功能码、命令码做判断。
- 大量 `switch/case` 或连续比较。
- 有循环读取字节、解析 TLV、计算 CRC、解析 JSON/XML/二进制结构。
- 会将外部数据拷贝到缓冲区、结构体或对象字段。

工控协议逆向中，最关键的是还原字段语义：

| 字段 | 关注点 |
| --- | --- |
| 长度 | 是否校验真实长度，是否可能越界 |
| 功能码/命令码 | 是否存在危险命令、隐藏命令 |
| 地址 | 是否能访问线圈、寄存器、变量、内存区 |
| 数据值 | 是否能影响设定值、输出、模式 |
| 会话/认证 | 是否真的绑定用户身份和权限 |
| 校验 | CRC、签名、摘要是否可绕过 |

### 2.4 做静态和动态交叉验证

静态分析可以告诉你 “可能发生什么”，动态分析用于确认 “实际是否发生”。

静态分析常做：

- 函数重命名、结构体恢复、调用图梳理。
- 字符串交叉引用。
- 危险 API 交叉引用。
- 协议字段和状态机恢复。
- 补丁前后 diff。

动态分析常做：

- 在输入读取处下断点。
- 在关键判断处观察寄存器和内存。
- 在危险函数处观察参数。
- 构造最小输入验证路径。
- 用抓包确认网络交互。

如果静态看到 `strcpy(dst, user_input)`，动态要继续确认 `user_input` 是否真的外部可控、是否有长度限制、目标缓冲区大小是多少、崩溃能否稳定复现。

## 3. 工控逆向中的特殊关注点

### 3.1 状态机比单个函数更重要

很多工控协议不是一个请求就能触发关键动作，而是有状态流程：

1. 建立连接。
2. 握手或设备识别。
3. 登录或会话建立。
4. 进入工程模式、维护模式或下载模式。
5. 下发读写、启停、下载、强制输出等命令。

逆向时要画出状态转换图，而不是只盯着某个函数。很多漏洞的本质是 “状态检查缺失”，例如未登录状态也能调用下载命令，普通用户也能进入维护命令。

### 3.2 安全影响要落到工艺动作

写报告时不要只写 “存在认证绕过”。应该进一步说：

- 绕过后能访问哪些接口。
- 是否能读取工程配置、变量名、工艺参数。
- 是否能修改设定值、输出点、控制策略。
- 是否需要内网访问、同网段访问、工程站权限。
- 是否会导致停机、误动作、拒绝服务或数据篡改。

### 3.3 版本差异很关键

工控设备生命周期长，同一产品可能存在多个固件、多个硬件版本、多个定制版本。逆向结论必须写清版本：

- 样本哈希。
- 软件/固件版本。
- 编译时间。
- 设备型号。
- 补丁版本。
- 配置条件。

## 4. 面试表达模板

可以这样表达自己的逆向方法：

> 我做逆向时不会一上来就陷入汇编细节，而是先建立样本画像，确认文件类型、架构、导入表、字符串和运行环境。然后根据研究目标找外部输入入口，比如网络协议入口、工程文件解析入口或固件升级入口。静态上我会用 IDA/Ghidra 从字符串、危险 API 和协议关键字做交叉引用，恢复调用链和状态机；动态上用 x64dbg/WinDbg 在输入读取、长度判断、危险函数处下断点，再结合 Wireshark 或最小 PoC 验证。最后把结论落到工控影响，比如是否能越权下载程序、修改寄存器、触发拒绝服务或影响控制回路。

## 5. 实操建议

建议练习顺序：

1. 对普通 Windows 小程序做字符串、导入表、函数交叉引用练习。
2. 对开源网络服务做协议解析函数定位。
3. 对简单固件镜像做 binwalk 解包、文件系统分析、启动脚本分析。
4. 对 Modbus/TCP 客户端或服务端程序做功能码解析路径分析。
5. 对公开 CVE 做补丁对比，写一页根因分析。

每次练习都保留四类证据：样本信息、定位过程、验证截图或日志、风险结论。
