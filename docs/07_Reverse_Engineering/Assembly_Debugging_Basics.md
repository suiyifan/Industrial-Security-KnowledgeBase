# 汇编与调试基础

## 1. 为什么安全分析需要汇编

逆向不是要求一开始就手写汇编，而是要能读懂关键位置发生了什么。尤其在漏洞分析中，源码经常不可得，只能通过反汇编判断：

- 函数参数从哪里来。
- 长度字段有没有校验。
- 条件跳转是否能绕过。
- 缓冲区大小和拷贝长度是否匹配。
- 崩溃点是空指针、越界、非法指令还是栈破坏。

## 2. x86/x64 基础概念

### 2.1 寄存器

常见寄存器：

- `RAX/EAX`：返回值、临时值。
- `RBX/EBX`：通用寄存器。
- `RCX/ECX`、`RDX/EDX`、`R8`、`R9`：x64 Windows 下常用于前四个函数参数。
- `RSP/ESP`：栈指针。
- `RBP/EBP`：栈帧基址。
- `RIP/EIP`：当前执行地址。

调试时要重点看：

- 当前指令地址。
- 函数调用前参数寄存器。
- 函数返回后 `RAX`。
- 栈上的返回地址和局部变量。

### 2.2 常见指令

| 指令 | 含义 | 分析关注 |
| --- | --- | --- |
| `mov` | 数据移动 | 参数、变量赋值 |
| `lea` | 取地址 | 缓冲区地址、结构体字段 |
| `cmp` | 比较 | 条件判断前置 |
| `test` | 位测试/零判断 | 是否为空、是否成功 |
| `jmp` | 无条件跳转 | 控制流转移 |
| `jz/jnz` | 条件跳转 | 认证、长度、状态判断 |
| `call` | 函数调用 | 调用链核心 |
| `ret` | 函数返回 | 栈平衡、返回路径 |
| `push/pop` | 栈操作 | 参数、保存寄存器 |

### 2.3 调用约定

Windows x64 常见约定：

- 第 1 到第 4 个参数放在 `RCX`、`RDX`、`R8`、`R9`。
- 更多参数放在栈上。
- 返回值放在 `RAX`。

这对观察 API 参数非常重要。例如断在 `memcpy` 时：

- `RCX` 通常是目标地址。
- `RDX` 通常是源地址。
- `R8` 通常是长度。

如果 `R8` 来自网络包长度字段，而目标缓冲区固定，可能存在越界风险。

## 3. 调试器工作流

### 3.1 断点类型

- 软件断点：修改指令为 `int3`，最常用。
- 硬件断点：利用 CPU 调试寄存器，适合监控内存读写。
- 内存访问断点：观察某段内存何时被读写执行。
- 条件断点：满足某个条件才中断，适合循环和高频函数。

### 3.2 常见调试目标

| 目标 | 下断点位置 |
| --- | --- |
| 找网络输入 | `recv`、`WSARecv`、协议解析函数 |
| 找文件解析 | `CreateFile`、`ReadFile`、解析函数 |
| 找认证逻辑 | `strcmp`、`memcmp`、登录字符串引用 |
| 找命令执行 | `CreateProcess`、`ShellExecute` |
| 找崩溃根因 | 异常地址、调用栈、最后一次内存写 |
| 找协议发送 | `send`、`WSASend` |

### 3.3 一次动态分析怎么做

假设你要分析一个工控网关程序是否存在协议解析问题：

1. 启动程序并确认监听端口。
2. 用 Wireshark 抓包，记录正常请求和响应。
3. 在 `recv` 或解析函数下断点。
4. 发送正常请求，观察缓冲区内容和长度。
5. 单步进入解析逻辑，记录功能码、长度、分支判断。
6. 修改某个字段，重新发送，观察是否进入错误处理。
7. 对异常请求做崩溃记录：寄存器、调用栈、异常类型、输入样本。

## 4. WinDbg 与 x64dbg 的定位差异

x64dbg 更适合交互式用户态调试：

- 界面友好。
- 下断点、看内存、看字符串方便。
- 适合小程序、协议客户端、PoC 复现。

WinDbg 更适合深层定位：

- 崩溃 dump 分析。
- 服务进程、驱动、系统组件。
- 调用栈、异常、符号、内存结构分析。
- 和 Windows 内部机制结合更好。

面试中可以说：日常用户态逆向我会先用 x64dbg 快速验证路径，遇到崩溃 dump、服务进程或复杂异常时用 WinDbg 深入分析。

## 5. 崩溃分析基本判断

崩溃时先不要急着说 “可利用”。先分类：

- 读访问违规：程序读取非法地址，可能是空指针或越界读。
- 写访问违规：程序写非法地址，风险通常更高。
- 执行访问违规：程序跳到不可执行地址，可能涉及控制流破坏。
- 栈 cookie 检测失败：可能存在栈溢出，但被保护机制拦截。
- 断言或主动退出：可能只是错误处理。

需要记录：

- 崩溃地址。
- 异常类型。
- 调用栈。
- 触发输入。
- 是否稳定复现。
- 是否影响服务可用性。

## 6. 面试表达模板

> 我做动态分析时通常会先从外部输入入口下断点，比如 `recv`、`ReadFile` 或协议解析函数。Windows x64 下我会重点看 `RCX/RDX/R8/R9` 这几个参数寄存器，比如在 `memcpy` 处观察目标地址、源地址和长度。对崩溃样本，我会先判断异常类型和调用栈，再反推最后一次外部输入如何影响了长度、指针或状态，而不是直接把所有崩溃都当成可利用漏洞。

## 7. 实操验证

建议建立调试笔记模板：

- 程序名、版本、哈希。
- 调试器和系统版本。
- 断点列表。
- 输入样本。
- 关键寄存器截图或记录。
- 调用栈。
- 结论：路径确认、风险判断、下一步。
