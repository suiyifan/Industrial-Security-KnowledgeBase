# 协议逆向

## 1. 协议逆向的目标

协议逆向是从流量、程序、固件或客户端行为中恢复协议结构和语义。工控安全中常见目标：

- 私有协议没有公开文档。
- 文档和实际实现不一致。
- 需要分析未知功能码、工程下载命令、认证流程。
- 需要为检测系统、扫描器、审计工具编写解析器。
- 需要判断协议是否存在明文认证、重放、越权或解析漏洞。

## 2. 输入材料

协议逆向通常结合四类材料：

- PCAP 流量：真实请求和响应。
- 客户端程序：上位机、工程软件、配置工具。
- 服务端程序：网关、设备固件、仿真服务。
- 文档和日志：说明书、调试日志、错误码。

单靠抓包容易只看到现象，单靠程序容易误判语义。最好交叉验证。

## 3. 从流量恢复字段

### 3.1 固定头部

观察多个数据包的共同前缀：

- 魔数。
- 版本号。
- 长度。
- 会话 ID。
- 序列号。
- 命令码。
- 校验字段。

如果某两个字节随 payload 长度变化，很可能是长度字段。如果某个字段每次请求递增，可能是序列号。

### 3.2 请求响应对应

整理请求和响应：

| 请求序号 | 请求字段 | 响应字段 | 推测 |
| --- | --- | --- | --- |
| 1 | 命令 `0x01` | 返回设备信息 | 设备识别 |
| 2 | 命令 `0x02` + 用户名 | 返回 token | 登录 |
| 3 | 命令 `0x10` + 地址 | 返回变量值 | 读变量 |
| 4 | 命令 `0x11` + 地址和值 | 返回成功 | 写变量 |

### 3.3 长度和大小端

判断大小端：

- 长度 16 如果表示为 `00 10`，可能是大端。
- 长度 16 如果表示为 `10 00`，可能是小端。

工控协议中大端、小端、混合端序都可能存在，不能凭经验武断。

## 4. 从程序恢复协议

程序里常见线索：

- `switch(command)`。
- `case 0x10`、`case 0x11`。
- `CRC16` 表。
- `send_packet`、`parse_packet`。
- 错误字符串：`invalid length`、`bad checksum`。
- 结构体偏移。

恢复过程：

1. 从 `recv` 或抓包定位接收缓冲区。
2. 找第一个长度校验。
3. 找功能码或命令码分发。
4. 给每个处理函数命名。
5. 对照流量验证字段语义。

## 5. 认证与会话逆向

认证流程是高价值点：

- 是否明文口令。
- 是否固定 challenge。
- token 是否可预测。
- token 是否绑定 IP、会话、权限。
- 退出后 token 是否失效。
- 危险命令是否检查权限。

很多协议表面有登录，但危险函数内部没有再次检查权限。逆向时要看命令处理函数本身是否使用会话权限字段。

## 6. 安全风险模式

| 风险 | 逆向证据 | 影响 |
| --- | --- | --- |
| 明文认证 | 流量中直接出现账号口令 | 凭据泄露 |
| 可重放 | 请求无 nonce/时间戳/序列校验 | 命令重放 |
| 越权命令 | 危险命令无权限检查 | 未授权控制 |
| 长度校验缺失 | 使用外部长度直接拷贝 | 崩溃或内存破坏 |
| 隐藏命令 | 客户端未暴露但服务端支持 | 后门或维护风险 |
| 弱校验 | CRC 当作安全校验 | 可伪造请求 |

## 7. 输出协议说明

逆向完成后，应输出可复用文档：

- 连接方式。
- 握手流程。
- 认证流程。
- 报文头格式。
- 字段说明。
- 命令码表。
- 错误码表。
- 示例请求响应。
- 安全风险。
- 解析器或 PoC。

## 8. 面试表达模板

> 我做协议逆向时会先用 Wireshark 把正常业务流程抓完整，比如连接、登录、读变量、写变量、下载。然后用多组样本对比字段变化，先恢复固定头、长度、序列号、命令码和校验。之后我会在客户端或服务端二进制里找 `recv`、功能码分发、CRC 和错误字符串，给处理函数重命名。最后把流量和程序逻辑互相验证，输出协议字段表、状态机和安全风险，比如明文认证、重放、越权命令或长度校验缺失。

## 9. 实操验证

可以用 Modbus/TCP 做协议逆向训练：

- 抓取读保持寄存器请求。
- 标出事务 ID、协议 ID、长度、单元 ID、功能码、起始地址、数量。
- 修改起始地址和数量，观察字段变化。
- 编写一个简单解析脚本打印字段。
- 总结哪些字段如果不校验会产生风险。
