# 逆向风险案例

## 案例 1：工程软件下载命令缺少权限检查

### 场景

某工程软件连接控制器后，可以执行上传、下载、运行、停止等操作。界面要求用户登录后才能点击下载按钮。

### 逆向发现

静态分析中，从字符串 `Download project` 找到按钮处理函数，再向下追踪到 `build_download_packet` 和 `send_packet`。继续分析服务端处理函数，发现命令分发中：

- `0x01`：设备信息。
- `0x02`：登录。
- `0x10`：上传。
- `0x11`：下载。
- `0x12`：运行/停止。

登录函数会设置会话对象中的权限字段，但 `handle_download` 内部没有检查该字段，只检查包格式和 CRC。

### 安全结论

如果攻击者能直接构造下载命令，就可能绕过客户端界面限制。风险不是 “客户端按钮可绕过”，而是 “服务端危险命令缺少服务端权限校验”。

### 面试表达

> 这个案例里我没有把客户端界面限制当作安全边界，而是继续逆向服务端命令处理函数，确认下载命令内部是否检查会话权限。最后发现登录状态只影响客户端按钮和部分会话字段，但危险命令处理函数没有强制校验，所以风险成立。

## 案例 2：固件 Web 后台命令注入

### 场景

工业网关 Web 后台提供网络诊断功能，可以输入 IP 做 ping 测试。

### 逆向发现

在固件 Web 目录中发现 CGI 程序，字符串包含 `ping -c 4 %s`。交叉引用显示 HTTP 参数 `host` 经过简单拼接后传入 `system`。

伪代码类似：

```c
snprintf(cmd, 128, "ping -c 4 %s", host);
system(cmd);
```

如果没有过滤 `;`、`&&`、反引号等 shell 元字符，就可能命令注入。

### 安全结论

风险取决于接口是否需要登录、登录是否可绕过、Web 服务运行权限、设备所在网络位置。工控影响可能是攻击者获得网关 shell 后修改转发规则、抓取工控流量或作为跳板。

### 面试表达

> 我会先确认参数来源是否真的是 HTTP 请求，再看是否经过白名单校验。对命令注入类问题，我还会关注 Web 进程权限和设备角色，因为工业网关一旦被拿下，影响可能不只是设备本身，还包括它连接的生产网络。

## 案例 3：私有协议长度字段导致崩溃

### 场景

某协议服务接收二进制报文，报文头中包含长度字段。

### 逆向发现

通过抓包恢复出报文头：

| 偏移 | 字段 |
| --- | --- |
| 0x00 | magic |
| 0x02 | version |
| 0x03 | command |
| 0x04 | length |
| 0x08 | payload |

IDA 中发现解析函数直接使用 `length` 调用 `memcpy`，但目标缓冲区是固定大小。动态调试时在 `memcpy` 下断点，确认长度来自外部报文字段。

### 安全结论

如果超长长度导致服务崩溃，可判定为拒绝服务风险。是否能进一步利用，需要看写入目标、保护机制、可控数据和崩溃类型。

### 面试表达

> 我不会因为看到 memcpy 就直接说有漏洞，而是确认 length 是否外部可控、目标缓冲区大小、是否有上限检查，并用动态调试记录 memcpy 参数和崩溃调用栈。最后按可复现拒绝服务或潜在内存破坏来定性。

## 案例 4：硬编码密钥导致升级包可伪造

### 场景

设备固件升级包声称有加密和校验。

### 逆向发现

解包升级程序后，发现校验逻辑使用固定密钥计算摘要，密钥硬编码在二进制字符串中。升级包没有非对称签名，只有对称校验。

### 安全结论

一旦密钥泄露，攻击者可以伪造升级包。对工控设备而言，这可能导致持久化后门或设备不可用。

### 面试表达

> 我会区分完整性校验和安全签名。CRC、MD5 或硬编码对称密钥只能防误传，不能抵抗攻击者伪造。固件升级更适合使用厂商私钥签名、设备内置公钥验证，并防止降级。

## 复盘清单

每个逆向案例都建议整理：

- 样本信息。
- 入口点。
- 关键函数。
- 数据流或状态机。
- 触发条件。
- 安全影响。
- 限制条件。
- 修复建议。
