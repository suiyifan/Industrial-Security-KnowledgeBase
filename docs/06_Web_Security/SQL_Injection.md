# SQL 注入与数据层风险

SQL 注入发生在应用把用户输入直接拼接到 SQL 语句中，导致攻击者可以改变查询逻辑。

工控场景中的 Web 系统可能包括运维平台、资产管理平台、报警查询系统、报表系统、历史数据查询和工业网关 Web 管理界面。

```text
浏览器参数 -> Web 后端 -> SQL 查询 -> 数据库 -> 返回账号、资产、报警、趋势或配置
```

安全风险包括读取账号、资产清单、点表、工程资料，篡改报警、报表或配置。

> 面试表达：我分析 SQL 注入时会继续看数据库里有什么工控信息。普通系统里可能是用户数据泄露；工控平台里可能泄露资产清单、点表、报警记录和工程配置，从而帮助攻击者理解控制系统。

## 为什么这个知识点存在

工业 Web 系统需要按设备、站点、时间和报警等级查询数据。HTTP 参数来自不可信客户端，SQL 却是一种可执行语言；应用必须保证外部输入始终只是“数据”。当开发者把输入直接拼接进 SQL 文本时，数据库无法区分哪部分是业务值、哪部分是语法，攻击者便可能改变原有查询逻辑。

```text
不可信输入 -> Web 业务逻辑 -> SQL 解释器
    数据         应保持为数据       被错误解释为指令
```

参数化查询正是为重建这条边界：应用先固定 SQL 结构，再通过数据库驱动绑定参数值。数据库把参数作为值处理，不会把其中字符重新解释成查询结构。输入校验仍用于限制站点编号、时间范围和页码，但不能替代参数化。

## 工业现场真实场景

SQL 注入点常见于 SCADA Web 发布模块、历史趋势与报警平台、MES、能源管理系统、资产管理平台、远程运维门户，以及工业网关和安全设备的管理界面。这些系统未必直接参与毫秒级闭环控制，却可能保存：

- PLC、RTU、网关的型号、固件、IP、机架槽位和协议；
- 点表、变量地址、工程单位、量程和报警阈值；
- 报警、趋势、批次、停机原因和维护窗口；
- 配方、控制参数、联锁旁路记录和工程文件路径；
- 用户角色、远程访问记录、API 密钥或数据库连接信息。

以水处理厂报警查询平台为例：PLC 采集液位、流量和泵状态，SCADA 生成报警，历史服务把事件写入数据库，Web 平台再按站点和时间查询。

```text
液位变送器 -> PLC 输入映像区 -> 报警逻辑 -> SCADA/历史服务
                                             |
                                             v
浏览器 <- Web 报警页面 <- 应用后端 <- 报警数据库
```

若数据库只是只读副本，直接物理影响有限；若平台还支持报警确认、配方发布或配置同步，数据篡改便可能进入控制链路。分析时不能只凭“存在注入”就声称可控制 PLC，而要确认数据内容、账户权限、同步方向和安全联锁。

## 底层原理

### 字符串拼接与参数绑定

危险实现把测点名直接放入 SQL 文本：

```python
# 反例：仅用于解释根因，不要用于生产代码
sql = "SELECT tag_name, value FROM tag_history WHERE tag_name = '" + tag + "'"
cursor.execute(sql)
```

此时 `tag` 既可能是测点名称，也可能改变语句结构。安全实现使用数据库驱动提供的占位符：

```python
# sqlite3 示例
sql = "SELECT tag_name, value FROM tag_history WHERE tag_name = ?"
cursor.execute(sql, (tag,))
```

参数化适合绑定“值”，不能把表名、列名或排序关键字当普通参数。动态标识符应由服务端白名单映射，例如客户端选择“时间排序”，服务端再映射为固定的 `occurred_at`，而不是原样拼接。

### 常见类型与能力边界

从观察方式看，注入可分为回显型、错误型、布尔盲注型、时间侧信道型和二次注入型。二次注入中的危险输入先被正常保存，之后在报表或同步任务中再次拼接执行，更容易跨越审计边界。

漏洞结果可能是读取、修改或删除数据，也可能绕过身份验证。是否能写数据、访问文件或进一步影响主机，取决于数据库类型、语句上下文、驱动配置和账户权限。专业报告应基于证据描述实际能力，而不是把所有注入等同于“拿下服务器”。

### 为什么手工转义不可靠

手工替换引号会受数据库方言、字符集、编码转换和上下文影响。字符串、数字、模糊查询、排序语句和存储过程的边界并不相同。参数化把语法和值的分离交给成熟驱动，可靠性高于自行维护转义规则。

## 数据流、控制流与状态流

### 数据流

```text
请求数据流：浏览器 -> 反向代理 -> Web 路由 -> 查询服务 -> 数据库驱动
           -> 结果集 -> JSON/HTML -> 浏览器

过程数据流：传感器 -> I/O -> PLC 变量 -> OPC/采集服务 -> 历史数据库
           -> 报表与趋势平台
```

两条数据流在数据库交汇。点表可暴露变量含义、单位和量程，历史趋势可暴露班次、生产节奏与设备启停规律。多个表关联后，攻击者可能形成完整过程画像，因此“数据泄露”还需说明其对控制系统侦察的价值。

### 控制流

Web 请求依次经历身份鉴别、对象授权、参数处理、查询执行和结果渲染。注入改变数据库分支的控制逻辑，使原本限定于某站点、某时间段的查询扩展到非预期范围。

如果触发器、定时任务或配置发布服务监听相关表，写入还可能触发后续流程。例如配方表版本变化后，发布服务读取新记录并同步到控制站。评估的关键问题是：“数据库写入之后，谁会消费这些数据？”

### 状态流

工业系统同时存在真实设备状态、PLC 内部状态、SCADA 显示状态、数据库历史状态和业务审批状态，它们不一定一致。

```text
真实阀位 -> PLC 反馈 -> SCADA 实时状态 -> 历史记录
                            |
工单审批 -> 配置发布状态 ---+
```

篡改历史表可能只造成追溯失真；篡改报警确认表可能让值班员误以为事件已处置；篡改待发布配方可能在下次批次切换时才影响过程。风险评估必须说明受影响的状态层、传播时机和独立反馈机制。

## PLC、DCS、SCADA 与协议中的实现

PLC 一般不直接执行 SQL。过程变量经 OPC UA、Modbus TCP、厂商驱动或消息中间件进入上位系统。DCS/SCADA 常用实时库保存当前值，用关系型数据库保存报警、审计、用户、批次和报表索引。

```text
PLC/RTU --工业协议--> 采集驱动 --> SCADA 实时库
                                      |
                                      +--> 历史服务 --> SQL/时序数据库
                                                            |
浏览器 --> Web/API --> 查询服务 -----------------------------+
```

工业协议流量与 SQL 查询处于不同层次。抓到 Modbus 功能码不能证明数据库字段语义，发现名为 `pump_start` 的字段也不能证明它会下发 PLC。研究人员应通过时间戳、点名、采集配置、发布日志和授权实验建立映射。

工业软件中容易遗漏的入口包括高级筛选、报表模板、批量导出、日志搜索、设备发现结果、固件版本检索和动态排序。新旧模块混合部署时，部分接口可能已参数化，遗留接口仍采用字符串拼接。

## 工控安全风险

### 保密性：过程知识泄露

资产表可暴露 PLC 型号和网络位置，点表可暴露寄存器语义，趋势数据可暴露正常波动和启停规律，维护记录可暴露无人值守时段。这些信息降低后续攻击的不确定性，并帮助攻击者选择更隐蔽的过程操纵目标。

### 完整性：错误数据进入工程决策

报警记录被改写会影响事件研判；检修工单被伪造会诱导人员错误操作；资产状态被标记为“已修复”会造成治理遗漏；配方或阈值被篡改可能改变产品质量或运行边界。完整性后果取决于写入路径：只读报表库主要影响分析可信度，双向业务库则可能影响审批、工单和配置发布。

### 可用性：查询拖慢监控服务

复杂查询、全表扫描或大量并发请求会占用连接池、CPU、内存和磁盘 I/O。历史数据库阻塞可能造成趋势缺口、报表延迟或报警归档失败。即使 PLC 闭环仍运行，操作员也可能失去态势感知。因此不得用自动化工具直接扫描生产平台。

### 物理后果的条件链

```text
可控输入 -> 改变 SQL 语义 -> 账户具备写权限
         -> 数据进入配置/命令发布链路 -> 缺少审批、签名与范围校验
         -> PLC/DCS 接受执行 -> 独立联锁或 SIS 未阻断
         -> 过程变量越过质量或安全边界
```

报告应逐项标记“已验证、合理推断、理论可能”。只验证越权读取时，不应声称已经能够操纵物理过程。

## 攻击链案例：污水站运维平台

以下为用于威胁建模的虚构案例，不提供针对真实系统的利用步骤。

1. 承包商运维门户的历史报警接口直接拼接查询参数。
2. 攻击者读取资产、点表和维护记录，识别提升泵、液位测点和无人值守时段。
3. 数据库中又错误保存了配置发布服务的连接信息，使攻击者获得接触内部应用的条件；这依赖额外的凭据管理缺陷。
4. 攻击者尝试修改待发布的泵组轮换参数和版本号，使异常配置进入发布流程。
5. 双人审批阻止了发布，PLC 的单泵连续运行限幅构成第二道屏障。
6. SOC 根据非维护时段写入、配置版本突变和异常数据库访问发现事件。

案例说明 SQL 注入只是入口，数据库权限、秘密管理、发布架构和控制器边界共同决定后果。修复一个参数之外，还应消除共享账户、明文秘密和单人发布等放大因素。

## 防护措施

### 开发与代码层

1. 所有值查询使用参数化语句或正确绑定参数的 ORM。
2. 动态表名、列名和排序字段使用服务端固定映射。
3. 对站点编号、时间范围、页码和测点名实施类型、长度及业务范围校验。
4. 不把数据库错误、SQL 文本和堆栈返回浏览器。
5. 为报表和导出设置最大时间范围、记录数、超时及并发限制。
6. 在代码评审和 CI 中检查 SQL 字符串拼接及缺失的对象授权。

```python
def query_alarm(cursor, site_id, start_time, end_time):
    sql = """
        SELECT occurred_at, tag_name, severity, message
        FROM alarms
        WHERE site_id = ? AND occurred_at BETWEEN ? AND ?
        ORDER BY occurred_at DESC LIMIT 1000
    """
    return cursor.execute(sql, (site_id, start_time, end_time)).fetchall()
```

参数化只解决 SQL 结构边界，函数外仍需完成身份鉴别、站点授权、时间解析和审计。

### 数据库、网络与架构层

- Web 查询账户默认只读，只授权必要的表、视图或存储过程；
- 查询、报警确认和配置发布使用不同账户，禁止共用管理员账户；
- 关闭不需要的文件访问、命令执行和扩展能力；
- 凭据进入秘密管理系统，定期轮换，不写入业务表或仓库；
- 配置和配方启用版本、双人审批、签名与不可抵赖审计；
- 拆分实时采集、历史归档、Web 查询和配置发布，通过工业 DMZ 与访问控制降低耦合；
- PLC/DCS 保留限幅、联锁和安全状态，不能把上层数据库作为唯一可信来源。

### 监测、响应与恢复

Web 层监测异常参数、响应码和导出量；应用层监测跨站点查询与超大时间范围；数据库层监测非维护时段写入、权限错误、结构变更和慢查询；发布层监测版本跳变与参数越界；过程侧比对设定值变化、工单和真实反馈。

发生事件时，先保护现场安全与生产连续性，再隔离受影响服务、吊销应用凭据、保存访问日志和数据库审计、确认数据完整性、检查下游同步，并从可信备份恢复。不能只修补页面参数后立即上线，因为数据库内容和凭据可能已失陷。

## 实验设计

### 目标、环境与边界

目标是在本机隔离环境观察字符串拼接与参数绑定的差异，并对应 HTTP 请求、应用日志和数据库结果。可使用 Python、Flask、SQLite 与虚构的 `alarms` 表，也可使用专门教学靶场。禁止导入真实资产、连接生产数据库或真实 PLC。

### 实验步骤

1. 绘制浏览器到数据库的信任边界，标记输入、身份信息和 SQL 执行点。
2. 实现仅供本机靶场使用的不安全查询，记录最终 SQL，不部署到共享环境。
3. 使用正常设备名与边界测试字符，对比错误、返回行数和日志；只证明输入能否改变结构，不扩大读取数据。
4. 改为参数化语句后重复测试，确认输入仅作为普通测点名处理。
5. 加入站点级授权、时间范围、分页上限和统一错误处理，再验证越权及资源消耗是否受控。
6. 将测试数据库权限调整为只读，验证纵使应用缺陷存在，写操作仍被权限拒绝。
7. 保存修复前后代码差异、测试用例、应用日志和截图，形成可复现报告。

### 抓包点与指标

```text
[浏览器] --抓包点 A--> [Web 服务] --日志点 B--> [数据库驱动]
                             |
                             +--审计点 C--> [测试数据库]
```

A 点观察参数位于 URL、表单还是 JSON；B 点观察授权结果、查询耗时和统一错误码；C 点观察语句类型、返回行数、被拒绝写入和慢查询。核心指标是正常功能保持、异常输入仅作为值、跨站点结果为零、查询上限生效。

修复前，边界输入可能导致语法或查询逻辑异常；修复后，同一输入只按测点名匹配。报告应包含根因、接口、账户权限、数据分类、下游消费者、物理影响条件链、修复代码和回归结果，而非只附工具截图。

实验不得执行批量枚举、数据导出、破坏性写入或压力测试。若在授权评估中意外发现真实漏洞，应停止扩大验证，按约定渠道报告并保全最小证据。

## 面试 60 分回答

SQL 注入的根因是应用把用户输入直接拼接进 SQL，导致数据被数据库当作语法解释。修复应优先使用参数化查询，并结合输入校验、最小权限和统一错误处理。在工控 Web 平台中，我还会确认数据库保存的是普通用户数据，还是资产、点表、报警、趋势、配方和配置；如果数据进入配置发布链路，完整性破坏可能间接影响生产过程。测试只在隔离授权环境中进行。

## 面试 90 分回答

我会从信任边界和数据链路分析 SQL 注入。外部参数经 Web 路由和业务逻辑进入数据库驱动，安全实现应先固定 SQL 结构再绑定参数，动态列名和排序字段使用白名单映射。发现漏洞后，我不会直接说“能控制 PLC”，而会继续验证数据库账户权限、表中数据语义、读写方向、同步服务、审批签名、PLC 限幅和 SIS 联锁。

历史报警库的只读注入主要造成资产与过程信息泄露；如果漏洞位于配方管理库，且写入会被发布服务自动同步，完整性风险才可能沿“数据库—发布服务—SCADA/DCS—控制器”传播。防护除参数化和最小权限外，还要拆分查询与发布账户、限制资源、监测非维护时段写入和版本突变，并保留过程侧独立反馈。验证时应把 HTTP 请求、应用日志、数据库审计和过程状态对应起来，区分事实与条件性推断。

## 面试追问

### 参数化查询后是否绝对安全？

不是。仍需处理身份鉴别、对象级授权、动态标识符白名单、数据暴露、资源限制和最小权限。业务越权不会因参数化自动消失。

### ORM 是否天然没有注入？

不是。标准查询接口通常会绑定参数，但原生 SQL、字符串拼接和动态排序仍可能引入漏洞，应检查实际生成的查询。

### SQL 注入一定能写数据或执行系统命令吗？

不一定。能力受数据库类型、语句上下文、账户权限、驱动配置和网络架构限制，报告必须基于证据。

### 如何判断篡改数据库会不会影响 PLC？

追踪谁读取该表、何时同步、使用什么协议、是否审批、控制器是否校验范围，并在授权靶场比对数据库版本、发布日志、SCADA 变量和 PLC 状态。不能凭字段名推断。

### WAF 能替代代码修复吗？

不能。WAF 可临时缓解和辅助监测，但存在编码、方言和业务上下文差异。根因修复仍是参数化、安全的动态 SQL 设计和权限收敛。

### 为什么不能直接扫描生产系统？

自动化探测可能制造复杂查询和大量错误，导致连接池耗尽、锁等待或历史服务延迟。工业环境重视可用性，应在副本或隔离靶场复现，生产验证遵循书面授权和最小请求。

## 本节总结

SQL 注入的本质是数据与指令边界失效。工业场景的重点不止是“能否改变查询”，还包括数据库承载何种过程信息、账户具有什么权限、数据如何流向 SCADA/DCS/PLC，以及独立联锁能否阻止异常进入物理过程。

完整研究主线是：定位输入到 SQL 的路径，验证参数是否改变语义，确认数据和权限边界，追踪下游数据流、控制流与状态流，评估对保密性、完整性、可用性和物理过程的条件性影响，最后通过参数化、最小权限、架构隔离、发布审批、监测响应和过程侧安全屏障形成闭环。
