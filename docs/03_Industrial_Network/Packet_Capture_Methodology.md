# 工业网络抓包与证据方法论

## 1. 原理解释：抓包是一个观察窗口，不是完整事实

网络抓包记录某个接口或监测点在特定时间看到的帧。它适合回答谁与谁通信、何时建立会话、应用请求与响应如何变化，却不能自动证明另一段路径也收到报文，更不能单独证明 PLC 执行了命令或执行器产生物理动作。

同一事件在客户端、边界、交换核心和控制器附近可能呈现不同源地址、VLAN 标签、时序与丢包。NAT、代理、冗余、非对称路由和镜像配置都会改变证据。分析前必须记录抓包位置、接入方式、方向、过滤器、时钟、接口负载和已知盲区。

## 2. 抓包前的工程问题

先定义问题，再选择位置和时长：

1. 是定位通信中断、性能抖动、未知资产、异常写入还是工程下载？
2. 目标业务的数据流、控制流和状态流是什么？
3. 源、目的、VLAN、NAT、代理和冗余路径在哪里？
4. 正常周期、故障窗口和过程事件时间是什么？
5. 哪个点最接近需要证明的事实？
6. 抓包会否改变拓扑、增加负载或引入新攻击面？
7. 需要哪些设备配置、日志、工程文件与过程趋势配合？

例如要判断 HMI 写命令是否到达 PLC，抓 HMI 侧可证明发送意图，抓 PLC 上联更接近接收事实；若中间存在代理，两侧都需要捕获并用事务字段关联。要判断物理动作，还要查看控制器响应、输出反馈、阀位/电流和过程 PV。

## 3. 捕获方式与能力边界

### 3.1 交换机 SPAN / Port Mirroring

SPAN 把指定源端口或 VLAN 的收发流量复制到目的端口，部署方便，但在源流量大于目的口能力、交换芯片拥塞或配置不完整时会丢包。部分设备不保留 FCS、前导码、原始错误帧或 VLAN 标签；入/出双向镜像可能产生重复帧。

配置时记录源端口/VLAN、RX/TX 方向、目的端口速率、会话优先级和设备型号。镜像目的端口应禁止普通交换，分析主机不配置生产路由，防止误发流量。

### 3.2 TAP

网络 TAP 在链路中复制流量，通常比 SPAN 更稳定并能分别输出两个方向。无源光 TAP 会引入光功率损耗，有源 TAP 需要供电并定义失电行为；聚合 TAP 也可能在输出带宽不足时丢包。安装涉及生产链路，应经过变更、光预算和回退评审。

### 3.3 主机捕获

在服务器或工程站本机捕获可观察进程附近流量，但受到网卡卸载、虚拟交换、容器、主机防火墙和恶意软件影响。Checksum Offload 可能让发送包在抓包中显示校验和错误，Large Send/Receive Offload 会改变分段形态。应记录接口与卸载设置。

### 3.4 防火墙、传感器和设备内置捕获

边界设备捕获便于关联规则、NAT 和会话，但可能只保留匹配接口或经处理后的报文。PLC、交换机或 RTU 内置诊断捕获资源有限，启用可能影响性能。生产设备上使用前必须确认厂商支持和风险。

## 4. 抓包点设计

单点捕获适合建立通信基线，多点同时捕获适合定位报文在哪一段丢失或改变。典型位置包括企业—DMZ、DMZ—控制区、SCADA 服务器上联、工程站上联、控制器单元、远程站 WAN 和冗余 A/B 网。

多点关联依赖时间与不变量。可使用应用事务 ID、TCP 序号、IP ID、报文长度、载荷摘要和相对间隔匹配；经过代理后 TCP 序号和五元组会改变，只能用应用字段或业务事件关联。若时钟无法精确同步，报告使用时间区间并注明误差。

抓包位置应贴近结论。例如“防火墙允许”由防火墙日志证明更直接；“报文离开交换端口”需要出口附近证据；“PLC 接受写入”需要应用响应或控制器审计；“阀门动作”需要现场反馈。

## 5. 捕获质量与丢包判断

捕获文件没有明显异常，不等于没有丢包。应检查：

- 捕获工具 dropped packets / interface drops；
- 接口速率、峰值和缓冲区；
- TCP 序号缺口、重复 ACK 与重传；
- UDP/工业协议应用序号缺口；
- 周期报文间隔异常；
- SPAN 源流量与目的口容量；
- 文件轮转间隙、磁盘写入和 CPU 负载；
- 多点捕获的报文数量差异。

TCP 重传可能是网络真实丢包，也可能是监测点漏掉原包或 ACK。若只在单点看到 `Previous segment not captured`，不能直接认定生产链路丢包。需结合端点重传行为、接口错误、队列丢弃和另一捕获点验证。

长时间捕获使用合理 SnapLen 和环形文件。截断载荷能减少存储和敏感数据暴露，但可能无法重组工业协议；完整捕获需更严格访问控制、容量和保留策略。

## 6. 分层分析流程

### 6.1 二层与拓扑

检查源/目的 MAC、VLAN 标签、ARP、LLDP、STP、组播和端口冗余。MAC 变化可能来自 HA、设备更换、环路或欺骗。ARP Reply 异常需结合邻居表和交换 MAC 表。

### 6.2 三层与路径

检查 IP、TTL、分片、ICMP、路由变化、NAT 前后地址和非对称。TTL 变化不能单独证明攻击；路径设备、操作系统初始 TTL 和隧道都会影响值。

### 6.3 传输层

TCP 关注握手、RTT、窗口、重传、RST/FIN、Keepalive 和连接并发。TCP 是字节流，必须重组后解析应用 PDU；单个 TCP Segment 不等于一个 Modbus、S7 或 OPC 消息。UDP 关注周期、序号、重复、乱序和应用确认。

### 6.4 工业应用与控制语义

解析请求/响应、事务 ID、功能、对象、地址、值、异常码、会话与订阅状态。然后映射到点表、PLC 内存、控制器模式和设备对象。协议成功响应可能只表示请求被服务接受，不一定证明程序使用该值或输出已改变。

### 6.5 过程与保护

把命令时间与 SP、PV、MV、模式、阀位、电流、报警和联锁关联。网络写入后 PV 未变化，可能被逻辑覆盖、设备未获许可、反馈故障或写入无效地址。物理后果结论必须跨越网络和自动化证据。

## 7. 时间、时区与事件关联

保存捕获主机时区、时钟源、同步状态、偏差估计和工具显示设置。PCAP 时间戳可能来自内核软件、网卡硬件或远程传感器，精度不同。Wireshark 显示的绝对时间、UTC 和相对时间可切换，报告中应统一。

事件时间线区分：用户操作、应用发送、网络经过监测点、PLC 响应、输出刷新、设备反馈、PV 变化、报警生成和历史入库。不同系统的记录粒度和延迟不同，无法精确校正时使用区间，而非制造毫秒级假精度。

## 8. 过滤器、会话与可复现分析

捕获过滤器在采集时减少数据，显示过滤器在分析时筛选已有数据。生产调查中捕获过滤器过窄可能永久丢失上下文，优先按抓包点和容量设计，再用显示过滤器分析。

常用显示过滤示例：

```text
eth.addr == 00:11:22:33:44:55
vlan.id == 20
arp
ip.addr == 192.168.10.20
tcp.flags.syn == 1
tcp.analysis.retransmission
tcp.stream eq 12
icmp
```

具体工业协议过滤器在 04 篇展开。每次分析保存工具版本、Profile、Name Resolution 设置、过滤表达式、统计导出和脚本。名称解析可能泄露目标或误导显示，证据分析通常保留原始 IP/MAC。

## 9. 证据保全与敏感信息

PCAP 可能包含账号、会话令牌、工程文件、配方和生产数据。采集前确认授权、范围、最小必要内容和保留期；采集后计算哈希、记录来源、操作者、时间和传递链，原始文件只读保存，分析使用副本。

导出对象、跟踪流或修改时间戳会生成派生证据，应记录步骤和哈希。报告中的截图只是说明材料，不能替代原始 PCAP 与可复现过滤。对外共享前脱敏，但脱敏副本与原始证据分开管理。

## 10. 数据流、控制流与状态流

```text
数据流：网络帧 → 捕获接口 → 内核缓冲 → PCAP → 重组/解析 → 结论
控制流：调查问题 → 抓包点/范围 → 采集授权 → 过滤分析 → 交叉验证
状态流：Prepared → Capturing → Rotating → Preserved → Analyzed → Archived
业务流：Disconnected → Connecting → Established → Operating → Faulted → Recovering
```

捕获链本身也有状态和故障。传感器重启、磁盘满、时间失步或镜像配置变化会制造证据缺口，监测平台必须监控自身健康。

## 11. 安全风险与攻击链

抓包基础设施可能成为攻击面：镜像口接入未授权笔记本泄露控制流量，传感器双网卡形成旁路，集中 PCAP 平台存放明文凭据，远程管理接口被控制后可篡改过滤或删除证据。攻击者也可能制造高流量使传感器丢包，在盲区执行工程操作。

典型攻击链是：攻击者取得交换机管理权限，临时把工程站与 PLC 流量镜像到维护端口；收集工程会话和设备地址后关闭镜像掩盖痕迹，再利用窃取凭据登录工程站修改逻辑。防护包括管理面隔离、配置变更审计、镜像会话基线、目的端口物理控制、传感器单向接入和 PCAP 加密/访问审计。

## 12. 工控场景：单点抓包误判 PLC 未响应

调查人员在核心交换机 SPAN 口抓包，看到 HMI 请求但看不到 PLC 响应，判断 PLC 无响应。实际 SPAN 只配置了 HMI 入口方向，PLC 返回帧未被镜像；控制器和 HMI 运行正常。根据错误结论重启 PLC 反而造成生产中断。

正确做法是先核对 SPAN 源与方向，检查捕获丢包，并在 PLC 上联或 HMI 主机增加授权捕获；结合 TCP 状态和 HMI 日志确认响应。任何主动处置前都要评估抓包证据边界和过程风险。

## 13. 面试表达

### 60 分答案

> 工业抓包应先明确问题和拓扑，再选择 SPAN、TAP、主机或边界设备。分析 Ethernet、IP、TCP/UDP 和工业协议，并注意时间同步、镜像丢包和证据保全。

### 90 分答案

> 我把抓包视为特定观察点的证据，不把“没看到”直接当成“没发生”。采集前记录路径、方向、NAT/代理、冗余和时钟，验证 SPAN/TAP 容量与丢包；分析时从二层表项、路由、传输状态重组到工业事务，再用点表、控制器模式、输出反馈和过程趋势验证。多点捕获用事务字段和误差区间关联，最终保留原始 PCAP、哈希、过滤与工具版本，确保结论可复现。

### 常见追问

**问：Wireshark 显示 TCP Retransmission 是否说明网络丢包？**

答：不一定，可能是真实丢包，也可能是捕获点漏包、乱序或只看到单向。要结合序号、ACK、端口计数和多点证据。

**问：看到 PLC 返回成功是否证明阀门动作？**

答：不能。还需确认写入对象、逻辑使用、输出、设备反馈和过程响应。

## 14. 实操建议

### 实验一：SPAN 方向与丢包

在实验交换机分别配置 RX、TX、双向和 VLAN 镜像，比较捕获差异；提高测试流量观察目的口容量和丢包计数。

### 实验二：多点路径关联

在客户端、路由边界和服务端同时捕获一次会话，比较 MAC、TTL、NAT、TCP 序号和时间偏差，重建逐段路径。

### 实验三：工业事务到过程

使用仿真 PLC/HMI 捕获一次读取和写入，将事务映射到变量、模式、反馈和趋势，明确网络响应与物理动作的证据差异。

### 实验四：证据保全

为无害 PCAP 建立采集记录、哈希、只读原件、分析副本、过滤器和结论表，让另一人按记录复现结果。

## 15. 总结与检查清单

- [ ] 能根据调查问题选择抓包点和方式；
- [ ] 能识别 SPAN、TAP、主机捕获与设备捕获局限；
- [ ] 能验证捕获丢包、时间和方向；
- [ ] 能重组 TCP 并分析工业应用事务；
- [ ] 能把网络证据映射到控制器和物理过程；
- [ ] 能维护原始 PCAP、哈希和可复现分析链；
- [ ] 能保护抓包基础设施和敏感数据。
