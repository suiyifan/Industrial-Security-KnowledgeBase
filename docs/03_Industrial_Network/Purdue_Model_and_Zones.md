# Purdue 模型、区域与通道

## 1. 原理解释：为什么需要层级和安全域

工业系统由物理过程、控制器、操作监控、生产管理和企业业务共同构成。若所有资产位于同一广播域或彼此任意路由，一个办公终端的恶意代码、错误组播或工程误操作就可能扩散到控制系统。Purdue Enterprise Reference Architecture 用 Level 0—Level 5 描述制造系统功能层级，帮助工程人员讨论不同功能之间的数据关系。

Purdue 模型首先是功能参考模型，不是可以直接复制的防火墙拓扑。安全设计还需要 Zone 与 Conduit：Zone 把具有相近功能、安全要求和风险的资产归为一组；Conduit 描述区域之间受控的通信通道。层级回答“资产做什么”，区域回答“哪些资产应受到相近保护”，通道回答“跨区业务如何流动并被控制”。

## 2. Purdue 层级及工程语义

| 层级 | 典型功能 | 常见资产 | 主要状态与能力 |
| --- | --- | --- | --- |
| Level 0 | 物理过程 | 管道、储罐、电机、阀门 | 温压流液、机械位置、能量状态 |
| Level 1 | 基本控制与感知 | PLC、RTU、远程 I/O、驱动器、仪表 | I/O、控制逻辑、设备模式和输出 |
| Level 2 | 区域监控操作 | HMI、操作员站、控制服务器 | 监视、报警、设定、启停和模式操作 |
| Level 3 | 现场运营管理 | 工程站、历史库、资产管理、批次/MES 接口 | 工程变更、维护、生产协调与长期记录 |
| Level 3.5 | 工业 DMZ | 跳板、代理、文件交换、历史镜像 | 终止、缓存、复制、审计跨区业务 |
| Level 4 | 企业业务 | ERP、办公终端、企业 AD、分析系统 | 计划、财务、办公身份和企业数据 |
| Level 5 | 外部服务 | 云、互联网、供应商平台 | 外部连接、更新、远程服务 |

不同组织对 Level 3、4、5 的名称可能不同，Level 3.5 也不是原始 Purdue 层级中的正式整数层。项目中应记录采用的定义，不要为了匹配图示强行归类。例如工程站通常服务 Level 1 控制器，但它可能部署在 Level 2 工程网络或 Level 3 维护区；其归域应依据实际通信、权限与失效后果。

## 3. Zone：按共同风险而不是按网段分组

一个安全区域应有明确的成员、功能、责任人、风险等级、允许通信、身份来源、补丁策略、监测要求和恢复目标。Zone 可以与 VLAN 或子网对应，但两者不是同义词：VLAN 是二层广播域，Zone 是安全与治理概念；一个 Zone 可包含多个 VLAN，同一 VLAN 中也可能错误混入多个安全要求完全不同的资产。

常见区域包括：企业区、工业 DMZ、现场运营区、操作监控区、基本控制区、安全仪表区、设备单元区、远程站点区、实验测试区、供应商临时接入区和网络管理区。SIS 不应只因物理上连接到同一交换机就与 BPCS 视为同一安全域；即使网络隔离，也需检查工程站、身份、时间源、供电和最终元件的共同依赖。

### 3.1 资产归域的判断问题

1. 资产承担什么生产或安全功能？
2. 它能观察、操作、维护还是工程修改哪些对象？
3. 被攻陷或故障后最大可信后果是什么？
4. 它依赖哪些身份、时间、名称、更新和管理服务？
5. 需要与哪些区域通信，方向、频率和应用操作是什么？
6. 谁负责变更、事件响应和恢复？

双网卡服务器、虚拟化集群和多功能工程站尤其不能只按某个 IP 归域。应逐接口记录所连接网络、路由能力、主机防火墙、服务监听、转发设置、虚拟交换和管理面，识别其是否成为绕过边界的隐式通道。

## 4. Conduit：受控的数据与信任通道

Conduit 不只是两区之间一根线，而是由网络路径、安全设备、协议、身份、方向和运行规则组成的受控关系。一个完整定义至少包含：

| 字段 | 示例 |
| --- | --- |
| 源/目的 Zone 与资产 | 控制区历史采集器 → DMZ 历史镜像 |
| 发起方向 | 控制区主动上送 |
| 协议与应用操作 | 厂商复制协议，只允许增量数据 |
| 身份与认证 | 专用服务账号、证书绑定 |
| 频率和性能 | 5 秒批量，允许 30 秒缓存 |
| 安全控制 | 双边界白名单、代理终止、审计 |
| 失联策略 | 本地缓存，不接受反向命令 |
| 恢复策略 | 从确认点补传，防止重复记录 |

允许 TCP 443 不代表定义了 Conduit。443 上可能是历史复制、Web 管理、远程桌面网关或任意隧道，它们具有不同能力。规则必须从业务和应用语义出发，再落实为地址、端口、证书、账号、代理策略和检测规则。

## 5. 数据流、控制流与状态流

以企业报表获取历史数据为例：

```text
数据流：控制器 → 历史采集 → 控制区历史库 → DMZ 镜像 → 企业报表
控制流：采集配置 → 复制任务 → 代理/防火墙策略 → 查询授权
状态流：Healthy → Link Lost → Buffering → Reconnected → Backfilling → Synchronized
```

跨区设计要解释值、质量、源时间和补传状态如何传递。若 DMZ 只复制数值而丢失质量码，企业分析可能把陈旧值当作当前值；若重连补传没有幂等或序号，可能产生重复数据。安全监测也需要区分稳定复制、缓存补传和异常批量导出。

控制流关注谁能改变通道。防火墙管理员、代理管理员、证书签发者和应用配置员可能分别控制不同环节。只做网络分区但共享企业域管理员账号，会让身份通道绕过网络层级；只隔离数据面但从企业网直接管理工业交换机，也会保留高权限路径。

## 6. Purdue 模型的适用边界

### 6.1 云、边缘与 IIoT

边缘网关可能同时采集 Level 1 数据并连接云平台，无法简单放在某一层。应将其拆成南向采集、处理、缓存、北向发布和管理面，分别建模信任边界。云连接不应使外部平台获得到控制器的透明路由。

### 6.2 分布式基础设施

电力、水务、油气管线拥有大量远程站点和中心调度，物理拓扑不是单厂垂直金字塔。每个站点应视为自治 Zone，通过受控广域 Conduit 与中心连接，并定义链路中断、本地控制、命令有效期和重连策略。

### 6.3 虚拟化与共享平台

多个层级的服务器可能运行在同一虚拟化集群。VLAN 分离不能消除 Hypervisor、存储、备份和管理面的共同风险。需要明确虚拟交换、宿主管理、迁移、快照、备份和恢复的跨域权限。

### 6.4 安全系统独立性

Purdue 层级不等于功能安全独立性。SIS 与 BPCS 即使位于不同 VLAN，若共享工程站、时间源、交换机管理、机柜或变送器，仍可能共因失效。安全评估需结合 SIF 和保护层分析。

## 7. 安全风险与攻击链

常见安全风险包括：把所有 Level 2/3 资产置于扁平 VLAN；企业与控制区建立透明双向路由；DMZ 仅有名称而无双边界；双网卡历史库绕过防火墙；控制器默认网关指向企业可达路由；管理面跨越所有区域；共享 AD、备份或杀毒服务器形成高权限通道；临时维护规则长期保留。

一条典型攻击链是：攻击者控制企业域账号，经“生产报表需要”开放的宽泛规则登录 Level 3 MES；利用共享服务账号访问历史库，再通过历史服务器上残留的工程软件连接 Level 1 PLC；下载修改后的逻辑并抑制 Level 2 报警。Purdue 图中的每层看似都有设备，但身份复用和多宿主服务器让攻击绕开预期边界。

防护不能停在“部署防火墙”。应建立资产归域和通信矩阵；分离企业与工业身份；在 DMZ 终止代理和远程会话；禁止透明跨区管理；对工程下载、控制器模式、远程桌面和文件交换做语义审计；验证边界故障时业务进入可接受状态；用独立保护层限制最终物理后果。

## 8. 工控场景：错误归域导致配方路径失控

某食品工厂将配方服务器归为“普通数据库”，部署在企业生产管理区，并允许操作站通过数据库端口直接访问。实际配方服务不仅提供查询，还能修改批次参数并向 PLC 下发目标值。企业账号被盗后，攻击者修改温度和保持时间；操作站仍从同一数据库读取被篡改配方，导致产品质量偏离。

调查需要关联配方版本、数据库审计、批次状态、PLC 目标值、操作确认和质量检测。整改时把配方服务按控制能力归入受保护的现场运营 Zone，在 DMZ 通过审批接口交换订单，不允许企业数据库直写；配方变更双人复核、签名并绑定适用设备和版本；控制器侧限制范围并保留独立质量验证。

## 9. 工程实施步骤

1. 从资产清单、交换配置、路由、防火墙和现场访谈建立实际拓扑，而不是先画理想 Purdue 图。
2. 为每项资产标记功能、接口、身份、关键通信、控制能力、失效后果和恢复责任。
3. 依据共同风险形成 Zone，记录例外与多宿主资产。
4. 从业务流定义 Conduit，包含方向、应用操作、状态、性能和失联策略。
5. 将逻辑设计落实到 VLAN、VRF、路由、防火墙、代理、账号、证书和监测点。
6. 在隔离或维护窗口验证正常、阻断、失联、切换、补传和恢复场景。
7. 用配置基线、变更审批、日志和定期复核维持设计，而不是一次性交付拓扑图。

## 10. 面试表达

### 60 分答案

> Purdue 模型把工业系统按 Level 0 到 Level 5 分层，Zone 按功能和风险对资产分组，Conduit 控制区域间通信。企业区与控制区之间通常设置工业 DMZ，减少直接访问。

### 90 分答案

> 我把 Purdue 当成功能参考而不是固定拓扑。项目中先从实际资产、接口和业务流出发，再按控制能力、失效后果与治理要求划分 Zone，并用 Conduit 描述源目的、发起方向、应用操作、身份、性能、失联和恢复策略。VLAN 只提供二层分段，不能替代安全域；DMZ、防火墙、代理、身份隔离和监测需要共同落实。对云边缘、多站点、虚拟化和 SIS，我会明确模型局限与共同依赖，不为了套层级隐藏真实攻击路径。

### 常见追问

**问：Level 3.5 是不是 Purdue 的正式层级？**

答：它是安全工程中常用的工业 DMZ 表达，不应纠结编号，而应验证是否真正终止跨区会话、实施双边界和最小业务流。

**问：一个 VLAN 是否就是一个 Zone？**

答：不一定。VLAN 是技术实现，Zone 是风险和治理边界。两者可映射，但还需路由、访问控制、身份、管理面和恢复要求支持。

## 11. 实操建议

### 实验一：从实际拓扑反推 Zone

在实验网络部署企业客户端、DMZ 代理、HMI 和仿真 PLC，导出交换机 VLAN、路由和主机接口信息。先画实际连通图，再划分 Zone，找出双网卡、共享账号和直接路由等偏差。

### 实验二：Conduit 状态验证

模拟历史上送，记录正常复制、链路中断、本地缓存、重连补传和同步完成。抓取序号、时间、质量和重复行为，验证防火墙“允许会话”与应用“数据一致”之间的差异。

### 实验三：攻击路径与屏障

对“企业账号—MES—历史库—工程站—PLC”路径逐步列出网络、身份、应用、工程语义和工况前提，并为每步匹配日志与防护。只在隔离环境验证，不向生产控制区主动探测。

## 12. 总结与检查清单

- [ ] 能区分 Purdue 层级、Zone、Conduit、VLAN 和子网；
- [ ] 能按功能、能力与后果进行资产归域；
- [ ] 能为跨区业务描述数据流、控制流和状态流；
- [ ] 能解释云边缘、多站点、虚拟化和 SIS 的模型局限；
- [ ] 能识别身份、管理面和多宿主资产形成的隐式通道；
- [ ] 能把攻击链映射到控制动作、过程后果与独立保护；
- [ ] 能通过配置、流量、日志和恢复测试验证设计。
