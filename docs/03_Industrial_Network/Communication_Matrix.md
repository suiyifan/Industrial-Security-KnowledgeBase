# 通信矩阵与数据流分析

## 1. 通信矩阵是什么

通信矩阵记录资产之间的通信关系。

它回答：

- 谁访问谁？
- 用什么协议？
- 哪个端口？
- 通信方向是什么？
- 是否需要读写？
- 是否经过边界设备？

## 2. 为什么通信矩阵重要

工业网络安全不是“全部不通”，而是只允许必要通信。

没有通信矩阵，就不知道：

- 哪些访问是正常业务。
- 哪些访问是异常横向移动。
- 防火墙策略应该怎么写。
- 工业 IDS 告警是否合理。

## 3. 通信矩阵字段

| 字段 | 示例 |
|---|---|
| 源区域 | 监控层 |
| 源资产 | SCADA Server |
| 源 IP | 192.168.10.20 |
| 目的区域 | 控制层 |
| 目的资产 | PLC-01 |
| 目的 IP | 192.168.20.10 |
| 协议 | Modbus TCP |
| 端口 | 502 |
| 方向 | SCADA -> PLC |
| 权限 | 读 / 写 |
| 业务说明 | 采集液位和泵状态 |
| 风险等级 | 中 |
| 控制措施 | 防火墙白名单、IDS 监测写功能码 |

## 4. 正常通信示例

```text
SCADA Server -> PLC: TCP 502
HMI -> SCADA Server: 应用协议
历史库 -> SCADA Server: 数据采集接口
工程站 -> PLC: 工程协议，仅维护窗口
```

## 5. 可疑通信示例

```text
办公终端 -> PLC: TCP 502
互联网 VPN 用户 -> 工程站: RDP
历史库 -> PLC: 工程协议
HMI -> 多个无关网段: SMB
PLC -> 办公网服务器: 任意 TCP
```

这些不一定都是攻击，但都值得解释。

## 6. 读写方向的重要性

同样是 SCADA 到 PLC：

- 只读过程值：风险较低。
- 写设定值：风险升高。
- 写启停命令：高风险。
- 下载工程：极高风险。

所以通信矩阵不能只写端口，还要写功能和权限。

## 7. 如何从抓包生成通信矩阵

步骤：

1. 收集一段稳定运行期间的流量。
2. 提取源 IP、目的 IP、端口、协议。
3. 合并重复连接。
4. 标注资产角色。
5. 识别工业协议。
6. 统计通信频率。
7. 标记写操作、工程操作、远程桌面、文件共享。
8. 与业务人员确认必要性。

## 8. 报告表达模板

```text
发现【源资产】可通过【协议/端口】访问【目的资产】。
该通信路径跨越【区域】到【区域】。
根据资产角色，该路径可能允许【读数据/写变量/远程运维/工程下载】。
若被攻击者滥用，可能导致【工控影响】。
建议【限制源地址、最小开放端口、增加审计、仅维护窗口开放】。
```

## 9. 面试表达

> 我会用通信矩阵把工业网络从“拓扑图”变成“数据流”。不只记录 IP 和端口，还要记录资产角色、区域、通信方向、读写权限和业务必要性。比如 SCADA 到 PLC 的 502 端口如果只读采集是一种风险，如果允许写寄存器或工程操作，风险级别就完全不同。
