# 通信矩阵与数据流分析

## 1. 通信矩阵是什么

通信矩阵记录资产之间的通信关系。

它回答：

- 谁访问谁？
- 用什么协议？
- 哪个端口？
- 通信方向是什么？
- 是否需要读写？
- 是否经过边界设备？

## 2. 为什么通信矩阵重要

工业网络安全不是“全部不通”，而是只允许必要通信。

没有通信矩阵，就不知道：

- 哪些访问是正常业务。
- 哪些访问是异常横向移动。
- 防火墙策略应该怎么写。
- 工业 IDS 告警是否合理。

## 3. 通信矩阵字段

| 字段 | 示例 |
|---|---|
| 源区域 | 监控层 |
| 源资产 | SCADA Server |
| 源 IP | 192.168.10.20 |
| 目的区域 | 控制层 |
| 目的资产 | PLC-01 |
| 目的 IP | 192.168.20.10 |
| 协议 | Modbus TCP |
| 端口 | 502 |
| 方向 | SCADA -> PLC |
| 权限 | 读 / 写 |
| 业务说明 | 采集液位和泵状态 |
| 风险等级 | 中 |
| 控制措施 | 防火墙白名单、IDS 监测写功能码 |

## 4. 正常通信示例

```text
SCADA Server -> PLC: TCP 502
HMI -> SCADA Server: 应用协议
历史库 -> SCADA Server: 数据采集接口
工程站 -> PLC: 工程协议，仅维护窗口
```

## 5. 可疑通信示例

```text
办公终端 -> PLC: TCP 502
互联网 VPN 用户 -> 工程站: RDP
历史库 -> PLC: 工程协议
HMI -> 多个无关网段: SMB
PLC -> 办公网服务器: 任意 TCP
```

这些不一定都是攻击，但都值得解释。

## 6. 读写方向的重要性

同样是 SCADA 到 PLC：

- 只读过程值：风险较低。
- 写设定值：风险升高。
- 写启停命令：高风险。
- 下载工程：极高风险。

所以通信矩阵不能只写端口，还要写功能和权限。

## 7. 如何从抓包生成通信矩阵

步骤：

1. 收集一段稳定运行期间的流量。
2. 提取源 IP、目的 IP、端口、协议。
3. 合并重复连接。
4. 标注资产角色。
5. 识别工业协议。
6. 统计通信频率。
7. 标记写操作、工程操作、远程桌面、文件共享。
8. 与业务人员确认必要性。

## 8. 报告表达模板

```text
发现【源资产】可通过【协议/端口】访问【目的资产】。
该通信路径跨越【区域】到【区域】。
根据资产角色，该路径可能允许【读数据/写变量/远程运维/工程下载】。
若被攻击者滥用，可能导致【工控影响】。
建议【限制源地址、最小开放端口、增加审计、仅维护窗口开放】。
```

## 9. 面试表达

> 我会用通信矩阵把工业网络从“拓扑图”变成“数据流”。不只记录 IP 和端口，还要记录资产角色、区域、通信方向、读写权限和业务必要性。比如 SCADA 到 PLC 的 502 端口如果只读采集是一种风险，如果允许写寄存器或工程操作，风险级别就完全不同。

## 10. 增强版通信矩阵模板

| ID | 业务/责任人 | 源进程 | 目的服务 | 发起 | 协议操作 | 身份 | 周期/窗口 | 状态与恢复 | 控制/证据 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| F-001 | HMI 监控/自动化 | SCADA 服务 | PLC-A | SCADA | 周期读、批准写 | 服务账号 | 500 ms/持续 | 超时 Bad，重连不补命令 | FW+DPI；PLC/趋势 |
| F-002 | 历史上送/数据组 | 控制历史 | DMZ 镜像 | 控制侧 | 增量复制 | 证书 | 5 s | 缓存、幂等补传 | 双边界；质量/源时间 |

矩阵中的“协议操作”应写读取、写入、订阅、下载、文件、远程桌面或管理，不只写端口。状态与恢复记录心跳、超时、缓存、补传和主备源地址；证据列指出配置、抓包、日志和工程资料。

## 11. 从流量到矩阵的校正流程

1. 从被动流量提取对端、发起方、周期、会话和应用操作；
2. 用 NAT、代理和主备配置还原真实资产；
3. 与现有规则、工程项目和责任人访谈交叉确认；
4. 把稳态、启动、维护、故障、回切和补传分别建模；
5. 对未解释流量建立偏差工单，不自动吸收为正常；
6. 将批准矩阵生成防火墙、监测和测试用例；
7. 变更后复核实际流量和业务状态。

## 12. 通信矩阵质量检查

- 是否使用真实资产/进程，而非整个网段？
- 是否区分发起方向和状态返回？
- 同一端口内的读、写、工程能力是否区分？
- 是否包含 DNS、时间、身份、备份和管理等依赖？
- 是否覆盖备用服务器、虚拟 IP、NAT 和远程站？
- 失联后是缓存、保持、过期、重试还是安全停车？
- 是否有责任人、批准依据、到期和最近验证时间？
- 每条流是否能对应到边界规则和检测策略？

删除“零命中”项前要覆盖完整生产周期及故障场景。新增观察流量时先判断是合法变更、误配置、资产冒用还是攻击，不能为消除告警直接放宽矩阵。
