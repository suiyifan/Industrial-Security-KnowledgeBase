# 工业 DMZ 与跨区数据交换

## 1. 原理解释：DMZ 为什么不是“中间放一台服务器”

企业网络需要生产报表、历史数据、远程支持和文件交换，控制网络又不能接受来自低信任区域的透明访问。工业 DMZ（IDMZ）通过两个边界之间的独立区域终止、代理、缓存或复制跨区业务，使企业端与控制端不建立直接端到端会话。

DMZ 的价值是缩短信任链和限制攻击路径，而不是增加一个网络层级名称。如果企业客户端仍能穿过 DMZ 直接登录控制区服务器，或一台双网卡主机同时信任两侧域，DMZ 只是路由中转。成熟设计要回答：谁主动发起、会话在哪里终止、传输什么数据与状态、用什么身份、失联后怎样缓存、恢复后怎样补传、谁能管理两侧边界。

## 2. 工程架构：双边界与独立性

```text
企业区
  ↓ 企业侧防火墙
工业 DMZ：代理 / 镜像 / 跳板 / 文件交换 / 更新中继
  ↓ 控制侧防火墙
现场运营区 / 控制区
```

双边界不一定要求两个品牌，但应避免共同失效：独立管理凭据、最小管理来源、受控配置发布、明确故障模式和分别审阅规则。若两台防火墙由同一被攻陷平台一键下发策略，或两侧都使用同一域管理员，物理双设备无法提供足够独立性。

DMZ 不应成为新的“全能服务器区”。远程访问、文件交换、历史镜像、补丁、杀毒和日志服务应按功能分段，限制横向通信。关键资产不能因为都在 DMZ 就默认互通；DMZ 内同样需要主机加固、身份分离、应用白名单、日志和备份。

## 3. 跨区数据交换模式

### 3.1 代理终止

企业客户端只连接 DMZ 代理，代理再以独立会话和身份访问控制侧服务。代理能够限制 URL、方法、对象、文件类型或应用操作，并让两侧协议、证书和账号解耦。透明代理或简单端口转发若不终止身份与会话，安全收益有限。

### 3.2 历史镜像与数据复制

控制区历史库向 DMZ 镜像上送数据，企业报表只查询镜像。设计要定义值、质量、源时间、删除/更正、压缩、断点和补传语义。只复制数值而丢失质量码会让企业系统把陈旧或替代值当作真实测量；补传无序号或幂等控制会产生重复记录。

### 3.3 文件摆渡

工程包、配方、补丁、签名文件和报表通过受控落地区交换。流程包括来源验证、恶意代码检查、类型/大小限制、内容解析、审批、签名/哈希、暂存、单向投递、接收确认、隔离保留和审计。杀毒通过不等于文件可信，宏、脚本、工程对象和合法签名供应链仍可能危险。

### 3.4 远程会话代理

外部用户先进入企业远程访问区，再经 DMZ 堡垒或远程桌面网关，在审批窗口访问指定维护目标。会话应终止于代理并重新认证，限制剪贴板、磁盘、端口转发和文件传输。禁止外部 VPN 给客户端分配可直接路由到 PLC 的地址。

### 3.5 单向传输

数据二极管或单向网闸在物理/协议层限制方向，适合历史、日志或文件上送。它不自动解决应用确认、补传、数据完整性和反向业务需求。很多 TCP 应用需要双向握手，单向系统通常在两侧使用代理重建会话，因此要验证代理缓存、确认和失败语义。

## 4. DMZ 常见服务与能力边界

| 服务 | 允许能力 | 不应隐含的能力 | 关键证据 |
| --- | --- | --- | --- |
| 历史镜像 | 复制过程数据供查询 | 企业端反向写控制历史或 PLC | 复制任务、质量、时间、断点 |
| 跳板/堡垒 | 受审计登录指定主机 | 任意网段路由、共享管理员 | 审批、MFA、会话录像、命令日志 |
| 文件交换 | 经检查投递批准文件 | 自动执行、任意双向共享目录 | 哈希、审批、扫描、接收确认 |
| 更新中继 | 分发批准补丁/特征库 | 直接连接互联网或自动全网安装 | 来源、签名、测试环、回退 |
| 日志汇聚 | 接收设备日志 | 用高权限代理反向管理设备 | 日志源、传输身份、完整性 |
| 应用代理 | 终止并重建特定业务 | 透明隧道任意协议 | 两侧会话、方法/对象策略 |

每项服务应使用独立服务账号并限制到必要目标。DMZ 服务器通常是高价值资产，因为同时了解两侧地址、证书和业务。其管理、EDR、备份和补丁路径也必须经过边界建模，不能为了安全管理重新打开企业到控制区的透明通道。

## 5. 数据流、控制流与状态流

以历史数据上送为例：

```text
数据流：PLC → 控制区历史库 → 复制客户端 → DMZ 镜像 → 企业查询
控制流：采集配置 → 复制白名单 → 服务账号 → 查询授权
状态流：Synchronized → Link Lost → Buffering → Reconnecting
       → Backfilling → Verified → Synchronized
```

网络会话恢复不是数据恢复。重连后要判断缓存是否完整、补传顺序、重复处理、源时间和质量是否保留。监测应区分正常实时流和恢复期高带宽补传，避免把补传误判为外传，也避免补传挤占报警通道。

文件交换的状态流可写为 `Submitted → Quarantined → Scanned → Approved → Released → Received → Verified → Archived`。任何跳步都可能造成未经审查文件进入控制区；审批后文件若被替换，应通过哈希或签名发现。

## 6. 身份、管理与信任边界

DMZ 应避免成为企业域和工业域的共同成员。可采用独立账号库、单向信任或经严格限制的联合身份，但必须分析认证失联和凭据泄露后果。服务账号禁止交互登录，密码/密钥轮换不应导致生产采集中断。

管理面要与业务面分离。管理员从受控管理工作站经堡垒访问 DMZ，不从普通办公终端直接 RDP/SSH。网络设备、防火墙、虚拟化、备份和安全平台的管理路径同样需要通信矩阵。应保留本地应急账号，但密封保管、使用告警并事后轮换。

证书信任也可能跨域扩散。若企业 CA 可无条件为控制区服务签发证书，企业 CA 被攻陷可能伪造 DMZ 或工业服务身份。需要限定模板、用途、签发权限和吊销/离线恢复方案。

## 7. 可用性、容量与故障模式

DMZ 故障不应直接导致控制器停止基本控制。历史上送、报表和远程支持通常可以降级，控制区应继续自治并本地缓存。若控制器运行依赖 DMZ 中的 DNS、AD、许可证或时间服务，DMZ 就进入关键控制依赖，需要更严格冗余和失联测试。

容量设计覆盖正常、报警洪泛、批量补传、补丁分发和事件取证。文件查杀、内容重构和代理会增加延迟与队列，队列满时应明确拒绝、缓存还是覆盖。恢复顺序通常先确认控制区稳定，再恢复基础服务、代理、数据复制和企业查询，避免上层批量连接冲击控制系统。

对高可用 DMZ 还要验证主备节点是否同步证书、策略、队列游标和缓存数据。仅有虚拟 IP 漂移不能证明文件、历史复制或远程会话可无损接管；切换后应重新确认身份、断点、数据质量以及审计链是否连续。

## 8. 安全风险与攻击链

常见安全风险包括企业到控制区端到端放通、跳板直达 PLC、共享文件夹双向长期挂载、DMZ 主机双域加入、服务账号权限过大、远程账号永不过期、更新中继直接联网、边界规则不同步、管理平台跨区以及日志只记录网络允许而无应用操作。

典型攻击链是：攻击者控制供应商账号，登录 DMZ 堡垒后利用开放的剪贴板上传工具；通过堡垒共享管理员凭据进入历史镜像，再利用历史复制服务账号访问控制区历史服务器；历史服务器残留工程软件和控制网接口，最终到达 PLC。若报警与历史显示同时被操纵，操作员可能延迟发现过程偏移。

防护要在每步设置屏障：临时账号与 MFA 限制入口；堡垒会话隔离和文件通道审批限制工具进入；DMZ 微分段限制横向；两侧独立身份和代理会话阻止凭据复用；控制区服务器最小软件和主机防火墙阻止到 PLC；工程下载审计、过程一致性和独立保护限制后果。

## 9. 工控场景：补丁中继成为跨区执行入口

某工厂在 DMZ 部署补丁服务器，企业侧同步软件包，控制区客户端自动下载并安装。为方便运维，补丁服务器加入企业域并使用域管理员服务运行；控制侧防火墙允许它访问所有 Windows 主机。攻击者取得企业域权限后替换未强制签名的内部包，在维护窗口被自动部署到工程站和操作站。

事件证据包括包来源、签名、哈希、审批、同步日志、客户端安装记录、服务账号和边界会话。整改应使用厂商签名与内部批准清单，企业侧下载、DMZ 检查、控制侧发布三阶段分离；先在兼容性测试区验证，再分批部署；发布账号仅能读取仓库，客户端安装由本地受控机制执行；保留回退和过程观察窗口。

## 10. 工程实施与验证

1. 从业务清单识别必须跨区的数据、方向、频率、身份和失联容忍度。
2. 优先选择代理、镜像或受控文件交换，避免透明端到端连接。
3. 为企业侧、DMZ 内部和控制侧分别建立通信矩阵与管理矩阵。
4. 审阅账号、证书、域信任、备份、更新和日志等隐式通道。
5. 定义正常、断链、缓存满、补传、主备切换和恢复验收条件。
6. 在实验或维护窗口验证业务，而不是只检查防火墙端口。
7. 定期核对实际流量、规则命中、资产软件和批准基线。

## 11. 面试表达

### 60 分答案

> 工业 DMZ 位于企业网与控制网之间，通过两侧防火墙部署跳板、历史镜像、文件交换和代理，减少企业到控制区的直接访问。

### 90 分答案

> 我判断 DMZ 是否有效，会看跨区会话是否在 DMZ 真正终止并以独立身份重建，而不是只看有没有两台防火墙。对历史复制会验证值、质量、源时间、缓存和补传；对文件交换会验证来源、签名、审批、隔离和接收确认；对远程访问会限制目标、时间、剪贴板和文件通道。同时检查身份、证书、管理、备份和更新等隐式信任，验证 DMZ 失效时控制区自治以及恢复期不会因补传冲击生产。

### 常见追问

**问：网闸或数据二极管是否能完全解决跨区风险？**

答：它限制网络方向，但仍要验证两侧代理、数据完整性、缓存补传、文件内容和管理面；有反向业务时更需明确替代流程。

**问：为什么不让跳板机直接访问 PLC？**

答：跳板用于受控进入管理目标，不应把外部会话直接延伸到基本控制。通常先访问指定工程站，再由工程流程和控制器权限约束操作。

## 12. 实操建议

### 实验一：端到端与代理终止对比

在隔离环境搭建企业客户端、DMZ 代理和控制服务，分别观察透明转发与双会话代理的源地址、证书、账号和日志差异。

### 实验二：历史缓存与补传

模拟链路中断并生成过程数据，恢复后检查补传顺序、重复、质量、源时间和带宽。记录网络恢复与数据完整恢复的不同时间点。

### 实验三：文件交换状态机

用无害测试文件执行提交、隔离、扫描、审批、哈希校验、投递和接收确认。审批后修改文件，验证完整性控制能否阻断。

### 实验四：DMZ 攻击路径桌面推演

以供应商账号、堡垒、历史镜像和工程站为节点，列出每步权限、证据、阻断点和过程后果。不对生产边界进行主动验证。

## 13. 总结与检查清单

- [ ] 能解释代理终止与透明转发的差异；
- [ ] 能设计历史、文件、远程和单向交换路径；
- [ ] 能描述跨区业务的数据流、控制流与状态流；
- [ ] 能审阅身份、证书、管理、备份和更新隐式通道；
- [ ] 能分析 DMZ 故障、缓存、容量和恢复顺序；
- [ ] 能把攻击链映射到工程权限与物理过程；
- [ ] 能在隔离环境验证跨区业务与恢复。
