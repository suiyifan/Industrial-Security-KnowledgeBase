# 工业网络评估、变更、恢复与验收

## 1. 原理解释：评估的终点是可安全改进和恢复

工业网络评估不只是找开放端口、过宽规则和旧固件，而是建立真实架构、业务流、控制能力和故障后果的证据，提出能在生产约束下实施的改进。未经过程理解的扫描或阻断可能比原风险更危险；没有恢复验证的整改，也可能在故障时失效。

完整生命周期包括准备、现场建模、证据采集、风险验证、方案设计、变更实施、故障/安全测试、生产验收和持续维护。每个阶段都要由网络、安全、自动化、工艺、设备和运营共同参与。

## 2. 评估准备与安全边界

评估前明确授权范围、资产、网络、时间、允许动作、禁止动作、联系人、停止条件、应急回退和证据处理。生产网默认采用访谈、配置审阅、日志和被动流量；主动扫描、协议写入、ARP 欺骗、环路、泛洪、冗余切换和控制器测试只在隔离环境或明确批准窗口执行。

收集资料包括网络图、资产清单、IP/VLAN/路由、防火墙规则、通信矩阵、远程访问、无线/蜂窝、工程项目、点表、备份、事件记录、变更流程和恢复预案。资料版本和现场实现可能不同，所有结论以当前证据为准。

评估工具笔记本应受管、加密、补丁更新，关闭桥接、共享和不必要服务。接入点、地址、时间和退出检查纳入工单；离场时删除临时账号、镜像、规则和证书。

对于多厂区或连续生产装置，还应约定分批范围和每日风险复盘，防止评估人员在尚未理解全局依赖时同时触碰多个冗余单元。现场发现超出授权范围的高风险问题时，先按正式升级路径通知责任人，不自行扩大任何验证范围和边界。

## 3. 现场调研：从过程到网络

先了解生产单元、关键设备、控制方式、联锁、SIS、启动停车和最大可信后果，再映射网络。仅看交换机配置难以判断某条 PLC 通信是否关键，也无法设计安全测试窗口。

访谈对象包括工艺、操作、仪表、自动化、电气、网络、安全、维护和供应商。问题围绕：谁使用系统、正常/故障怎样操作、哪些连接不能中断、维护如何进入、配置由谁保存、失联后设备做什么、过去发生过什么异常。

现场走查核对机柜、交换机端口、光路、双网卡、无线 AP、蜂窝天线、串口网关、USB/维护口、标签、电源和物理保护。拍照和采集配置需授权，敏感位置与信息按规定处理。

## 4. 建立权威当前状态

将资料、配置、被动流量和访谈合并为四张基础表：

1. **资产表**：工程身份、网络身份、功能、接口、版本、关键性、责任人；
2. **拓扑表**：物理/逻辑链路、VLAN、路由、NAT、冗余、管理面；
3. **通信矩阵**：业务、发起、协议操作、身份、周期、状态和恢复；
4. **依赖表**：AD、DNS、时间、PKI、虚拟化、备份、云和供应商。

每个字段记录来源、时间和置信度。网络观察到的 IP 不等于工程资产身份；未见流量不证明设备不存在；图纸存在的防火墙也不证明规则有效。冲突项保留并安排验证，不为得到“干净清单”而提前合并。

## 5. 非侵入验证方法

### 5.1 配置审阅

审阅 VLAN/Trunk、路由、ACL、防火墙、VPN、账号、日志、时间、SNMP、镜像、冗余和备份。使用离线配置副本，避免在生产设备执行高负载命令。敏感凭据先脱敏。

### 5.2 被动流量

验证实际对端、发起方向、周期、应用操作、重传、时间和异常路径。确认抓包点、方向、丢包和盲区。流量基线需覆盖稳态、启动、维护、故障与恢复。

### 5.3 日志与工程证据

结合防火墙会话、VPN/堡垒、交换变更、服务器、工程下载、控制器 SOE、报警和过程趋势。时间校正并记录误差，避免用单一日志归因。

### 5.4 离线与仿真验证

在同型号实验设备、虚拟网络、协议模拟器或数字孪生验证规则、超时、协议操作、故障切换和攻击假设。实验结果标注适用边界，不能无条件外推生产。

## 6. 风险分析与报告

风险描述采用：

```text
资产/业务 → 现状证据 → 攻击或故障前提 → 获得能力
→ 控制/状态变化 → 过程与业务后果 → 现有屏障
→ 可信度与建议验证 → 分阶段整改
```

严重度综合入口暴露、权限、可利用性、资产关键性、工况、检测、保护层和恢复复杂度。不要只用 CVSS 决定工业优先级。结论区分配置确认、流量观察、隔离实验和现场验证。

建议必须可实施：明确目标架构、规则/配置、责任人、窗口、前置测试、回退和验收证据。不能简单写“部署防火墙、加强监控、定期培训”。无法立即整改时给出临时访问限制、监测、现场许可和独立保护等补偿措施。

## 7. 变更工程与风险控制

网络变更可能影响实时流量、路由收敛、组播、时间、冗余和远程操作。变更单包含目的、受影响业务、配置差异、依赖、风险、执行步骤、验证、停止条件、回退、联系人和现场工况。

实施前备份当前配置并验证可读/可恢复，确认控制器和设备失联策略。高风险变更先在实验或同类非关键单元测试，再在维护窗口分阶段实施。避免同时修改两台冗余设备或两道边界，保留独立管理和回退路径。

实施中每步记录时间、操作者、命令/配置版本和状态。出现超出预期的丢包、质量下降、报警、主备异常或过程偏移立即按停止条件处置。禁止现场临时扩大 `any any` 后忘记回收。

## 8. 验收：从链路到过程

变更成功不能只看接口 Up 或设备提交成功。分层验收包括：

| 层次 | 验收内容 |
| --- | --- |
| 物理 | 链路、光功率、错误、电源、路径 |
| 二/三层 | VLAN、MAC/ARP、路由、组播、收敛 |
| 边界 | 允许与拒绝、NAT、会话、日志、HA |
| 应用 | 连接、订阅、事务、质量、时间、补传 |
| 控制 | HMI、PLC、模式、命令、反馈、报警 |
| 过程 | PV/MV、设备动作、联锁、生产质量 |
| 恢复 | 故障、回切、旧状态清理、备份可用 |

验收覆盖正常路径和关键拒绝路径，避免规则既放通业务也放通不必要工程操作。还要测试主备切换、链路中断、平台失联和恢复补传，但生产主动故障测试须专项批准。

## 9. 事件恢复原则

网络事件恢复不是“设备重启并能 ping”。应先稳定过程和现场安全，再保全证据、隔离攻击路径、恢复可信身份与配置，最后按受控顺序重建服务。

```text
发现 → 工况确认 → 风险稳定 → 证据保全 → 范围判定
→ 清除入口/凭据 → 恢复网络与平台 → 验证控制状态
→ 受控启动 → 观察 → 复盘改进
```

恢复前检查交换/防火墙/网关配置、固件、账号、证书、路由、NAT、时间、镜像和监测；同时检查工程项目、PLC 在线版本、强制、旁路、设备模式、阀位、库存和能量。网络恢复后可能存在缓存旧命令、历史补传和会话自动重连，需逐项控制。

## 10. 备份与恢复验证

备份对象包括网络设备配置、许可证、固件、证书/密钥、工业防火墙策略、VPN、网关映射、无线控制器、监测规则和资产数据库。仅保存配置文本可能不足以恢复专有设备；记录型号、硬件版本、引导程序和恢复工具。

备份应离线/不可变、访问受控、版本清晰并定期测试。恢复演练验证文件可读、工具兼容、密钥可用、依赖可满足和业务可重建。不能在生产设备上直接覆盖测试；使用备用设备或隔离环境。

恢复点应与批准基线和事件时间线匹配。过旧备份会丢失合法变更，过新备份可能包含入侵。通过差异审阅、签名/哈希、变更记录和多方确认选择可信版本。

## 11. 数据流、控制流与状态流

```text
数据流：资料/配置/流量/日志 → 证据模型 → 风险 → 变更与验收记录
控制流：授权 → 评估 → 批准 → 实施 → 验证 → 运营维护
状态流：Unknown → Observed → Verified → Designed → Changed
       → Validated → Monitored → Reassessed
恢复流：Compromised/Failed → Stabilized → Isolated → Restored → Trusted
```

“运行”不等于“可信”。事件后设备可能在线但配置未知；变更后业务可能正常但拒绝策略未验证。状态迁移必须有明确证据和批准人。

## 12. 安全风险与攻击链

评估和变更过程本身会产生攻击面：工具笔记本跨网、配置备份含凭据、临时 SPAN 泄露流量、测试账号/规则遗留、供应商远程窗口扩大、报告集中保存拓扑和漏洞。攻击者可针对这些高信息资产。

典型攻击链是：外部评估人员笔记本同时连接互联网和控制网络，恶意软件窃取交换配置与工程地址；利用临时放通规则进入管理 VLAN，修改核心交换和镜像配置；再隐藏后续 PLC 工程流量。防护包括受管隔离终端、单接口策略、临时访问自动到期、配置脱敏加密、操作双人复核和退出检查。

## 13. 工控场景：防火墙整改引发历史与报警中断

项目根据一周稳态流量删除“零命中”规则，却遗漏年度主备切换时备用报警服务器的主动连接。数月后主服务器故障，备机接管但无法向控制区订阅报警，操作员只看到部分实时值。网络链路正常，业务保护路径失效。

根因是基线未覆盖故障状态、规则删除无业务确认、验收只测主路径。整改需恢复最小备用规则，补充主备/故障通信矩阵，并把切换、报警、质量、历史和恢复纳入周期演练。

## 14. 面试表达

### 60 分答案

> 工业网络评估先确认授权和生产边界，通过资料、配置、被动流量和访谈建立资产、拓扑与通信矩阵，再分析分区、边界、远程和监测风险。整改要有变更、回退和验收。

### 90 分答案

> 我会从过程和控制功能开始，不直接扫描生产网。用资产、拓扑、通信和依赖四张表建立带来源和置信度的当前状态，再以攻击前提—控制能力—过程后果—屏障写风险。建议落实为配置差异、窗口、停止条件、回退和分层验收。验收从链路、路由、防火墙到应用质量、控制反馈和物理过程，并覆盖主备、失联和补传。事件恢复同时处理身份、配置、工程版本、强制旁路和现场状态，直到系统不仅在线而且重新可信。

### 常见追问

**问：为什么工业评估不直接做全网扫描？**

答：老旧或实时设备可能受异常并发影响，且扫描只能证明有限网络事实。优先配置、被动流量和工程证据，主动验证在隔离或批准窗口进行。

**问：如何证明整改有效？**

答：按预期业务、拒绝路径、故障切换、恢复和过程结果定义验收证据，而不是只看配置提交或端口连通。

## 15. 实操建议

### 实验一：离线配置审阅

用实验交换、防火墙配置建立资产/拓扑/通信矩阵，识别宽泛对象、管理面和备用路径，输出证据与建议。

### 实验二：变更与回退

在虚拟网络实施最小规则整改，预设停止条件，验证业务和拒绝，再故意触发失败并按回退恢复。

### 实验三：分层验收

构造 HMI—PLC—历史链路，分别验证物理、路由、会话、应用质量、变量反馈和过程趋势，记录每层通过标准。

### 实验四：事件恢复桌面演练

假设核心交换和工程站被篡改，设计稳定、取证、可信配置选择、现场核对、受控启动和观察流程。

## 16. 总结与检查清单

- [ ] 能定义授权范围、停止条件和安全测试边界；
- [ ] 能从过程建立资产、拓扑、通信和依赖模型；
- [ ] 能用非侵入证据并标注置信度与盲区；
- [ ] 能写出可验证、可实施、可回退的整改；
- [ ] 能从链路到物理过程完成分层验收；
- [ ] 能恢复身份、网络、工程和现场可信状态；
- [ ] 能保护评估工具、配置、PCAP 与临时访问。
