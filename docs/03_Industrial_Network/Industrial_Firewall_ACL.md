# 工业防火墙与通信矩阵

## 1. 原理解释：防火墙控制的是路径，不是物理后果

工业防火墙位于 Zone 之间，根据接口、地址、协议、端口、连接状态、用户或应用语义允许、拒绝和记录通信。它的目标是把跨域业务限制到已批准的最小集合，并让异常路径可见。防火墙不能理解所有控制逻辑，也不能保证通过的命令安全；被允许的工程站仍可能利用合法协议下载恶意逻辑。

通信矩阵是规则设计的业务依据。它描述哪些资产因什么生产功能而通信、谁主动发起、使用何种应用操作、正常频率和失联行为是什么。若矩阵只有“大网段—TCP 502—双向”，就无法区分 HMI 读取、PLC 主动上送、工程下载和未经授权写入。

## 2. 无状态 ACL、状态防火墙与工业 DPI

无状态 ACL 逐包匹配地址、协议和端口，不自动理解会话。状态防火墙跟踪 TCP 建连、UDP 伪会话和返回流，通常只需显式允许发起方向。应用代理或工业 DPI 进一步解析协议功能码、对象、服务或读写类型，但解析能力受协议版本、加密、分片、厂商扩展和设备性能限制。

工业 DPI 不能替代控制器授权。允许“只读功能码”有助于限制能力，但地址范围、控制器模式、封装协议和异常功能仍需验证；加密的 OPC UA、VPN 或 TLS 流量若无法解密，防火墙可能只能检查外层会话。设备宣称支持某工业协议，应通过文档、规则语法和隔离实验确认，而不能只看产品名称。

## 3. 通信矩阵的完整字段

| 字段 | 需要回答的问题 |
| --- | --- |
| 业务编号与责任人 | 为什么存在，谁批准和维护 |
| 源/目的 Zone、资产、进程 | 谁真正发起，目标是什么服务 |
| 协议、端口和应用操作 | 读、写、订阅、下载、文件还是管理 |
| 身份与认证 | 服务账号、证书、用户或设备身份 |
| 方向与会话 | 单向发起、双向独立会话还是回调 |
| 正常行为 | 周期、包长、并发、时延和时间窗口 |
| 状态与质量 | 心跳、超时、重连、缓存、补传 |
| 失联与恢复 | 保持、降级、人工确认或自动补传 |
| 安全控制 | 防火墙、代理、主机授权、监测 |
| 证据 | 配置、抓包、日志、工程文件和责任人确认 |

地址对象优先使用明确资产或受控资产组，而不是整个 `/16`。动态地址、HA 虚拟地址、NAT 前后地址和多接口主机必须记录。端口字段应区分客户端临时源端口和服务器监听端口，避免把临时端口范围错误开放为入站服务。

## 4. 从业务流生成规则

以 HMI 轮询 PLC 为例，先确定 HMI 是否直接连接 PLC、是否经过 SCADA 服务器、协议由谁发起、PLC 是否回连报警。再确认 HMI 只读还是也执行设定和启停，是否存在工程软件下载。最后才生成源、目的、服务、状态和日志规则。

```text
业务：操作员监视与控制泵站
发起：SCADA Server → PLC
应用：周期读取 + 经授权的操作写
网络：TCP 会话，PLC 不主动回连
防火墙：指定服务器到指定 PLC，指定服务，状态返回
补充控制：SCADA 用户权限、PLC 写保护、命令审计、过程反馈
```

若读写使用同一端口且防火墙无法解析命令，网络规则只能限制通信双方，最小权限必须由应用账号、控制器保护和监测补齐。报告中应明确控制边界，而不是宣称“防火墙已阻止写入”。

## 5. 规则顺序、对象与隐式行为

防火墙通常自上而下匹配首条规则，宽泛允许放在前面会遮蔽后续拒绝。对象组嵌套、地址重叠、服务别名、区域默认策略、NAT 和策略路由会改变实际效果。审阅不能只看一条规则，需要查看命中顺序、转换前后地址和路由路径。

高可用设备还要检查主备策略和对象同步。集中管理平台可能同时向多个厂区传播错误规则。变更前应做影响分析和配置备份，先在测试或低风险范围验证，再分阶段发布；变更后确认规则命中、业务状态和过程运行，而不只看提交成功。

隐式允许包括设备自身管理服务、同区域通信、VPN 解密后的流量、HA 同步、动态路由和默认系统规则。隐式拒绝若无日志，则排障时可能表现为无响应。需要为关键拒绝配置限速日志，避免攻击或故障时日志洪泛耗尽设备资源。

## 6. TCP、UDP 与状态超时

TCP 状态表记录五元组、序号和生命周期。工业长连接空闲但仍有效，若防火墙空闲超时短于应用心跳，连接可能周期性断开；过长超时又会保留大量陈旧状态。主备切换若未同步会话，现有连接会重置。

UDP 没有真正连接，防火墙通常用首包建立临时状态并在超时后删除。周期较长的遥测可能每次都被视为新会话；多播、广播和双向独立报文也需要专门策略。应根据真实周期和故障恢复设置超时，不盲目使用通用模板。

FTP、SIP 或某些厂商协议可能在控制连接中协商动态数据通道；简单放通固定端口可能失败，而宽泛开放高端口会扩大风险。应用感知助手应验证兼容性和解析边界。

## 7. 日志、命中与证据边界

允许日志可证明某五元组在某时刻匹配规则并尝试通过，不能证明目标收到、应用接受或设备动作。拒绝日志可证明策略阻断，但源地址可能被 NAT、代理或伪造。事件调查要关联接口、规则 UUID、NAT 映射、会话原因、字节数、TCP 标志和设备时间。

规则“零命中”不一定可删除：可能是备用链路、年度维护、报警回调或统计计数重置。删除前需要业务责任人确认并覆盖启动、故障、切换、检修和恢复场景。反之，高命中也不代表必要，可能是错误轮询或扫描。

日志时间必须与抓包、主机和控制器校正。防火墙集群的成员时钟或日志聚合延迟可能颠倒会话顺序。重要变更和事件应保全原始配置、导出文件、哈希和查询条件。

### 7.1 基线偏差的解释

实际流量与矩阵不一致时，不应自动把流量标为攻击，也不能自动修改矩阵迁就现状。偏差可能来自未登记业务、主备切换、临时维护、错误配置、资产冒用或真实入侵。处置流程应先识别源进程和目标功能，再核对变更、责任人和设备状态；未经确认的异常保持受限，并为业务补录提供正式审批路径。

矩阵也需要版本语义。配方系统上线、PLC 替换或历史架构调整后，旧规则可能仍被命中，却已不再符合当前设计。规则评审应把当前资产、工程版本和生产单元作为上下文，并记录规则新增、修改、停用与回退所对应的业务版本。

## 8. 数据流、控制流与状态流

```text
数据流：源应用 → 路由/NAT → 规则匹配 → 会话表 → 目标应用
控制流：业务批准 → 矩阵 → 对象/服务 → 规则 → 发布 → 验证
状态流：Proposed → Approved → Implemented → Observed
       → Reviewed → Modified/Expired → Removed
会话：New → Established → Idle → Closing/Expired/Reset
```

规则生命周期与网络会话生命周期必须分别治理。临时规则到期不应只依靠人工记忆；维护完成后应关闭访问、终止残余会话并验证控制器模式、强制与工程版本。规则删除也可能不影响已建立会话，需确认设备行为。

## 9. 安全风险与攻击链

常见安全风险包括 `any any allow`、大网段对象、双向规则、把源端口当服务端口、测试规则长期存在、管理面从办公网可达、工业 DPI 失败时默认放行、NAT 隐藏真实源、HA 策略漂移、共享管理员和日志未集中。

典型攻击链是：攻击者控制一台被允许访问控制区的历史服务器，发现防火墙规则按整个服务器网段放通 PLC 协议；在历史服务器启动协议客户端连接多台 PLC，读取地址后发送写请求。防火墙将其视为合法源和端口；若控制器无认证或写保护，攻击者可改变设定。过程后果仍取决于逻辑、模式、设备许可与 SIS。

防护要把规则缩到指定资产、应用和方向；分离历史与工程功能；支持时使用工业协议只读/写控制；监测新目标、新功能码和非窗口工程行为；在控制器与应用实施独立授权；用过程变量、反馈和保护事件检测合法通道滥用。

## 10. 工控场景：故障切换流量被白名单遗漏

正常情况下 SCADA 主服务器轮询 PLC，通信矩阵只记录主服务器 IP。主机故障后备服务器接管虚拟服务，但它以物理接口地址主动建立 PLC 会话，控制侧防火墙未放通。HMI 服务器本身显示正常，现场数据却逐步变为 Bad Quality，操作员无法远程控制。

处置要确认主备角色、源地址、虚拟地址行为、防火墙拒绝日志、PLC 会话和数据质量。修正规则时不能直接放通整个服务器网段，应记录主备物理/虚拟地址与发起模式，并测试主故障、网络故障、恢复回切和残余会话。业务验收包括数据新鲜、报警、操作反馈和历史连续性。

## 11. 规则设计与评审流程

1. 从流程和资产角色收集业务，不从现有规则反推“需求”。
2. 用抓包、配置和日志验证源、目的、方向、周期和异常路径。
3. 区分网络服务与工业操作能力，标注防火墙不能控制的部分。
4. 建立矩阵并由业务、自动化、网络和安全共同批准。
5. 生成对象、服务、NAT 与规则，执行影子、重叠和过宽分析。
6. 在隔离或维护窗口验证正常、拒绝、故障、切换和恢复。
7. 持续对比实际流量、命中、资产和变更，定期再认证。

## 12. 面试表达

### 60 分答案

> 工业防火墙按通信矩阵执行白名单，规则要明确源、目的、协议、端口、方向和业务。默认拒绝不必要通信，并记录关键允许和拒绝日志。

### 90 分答案

> 我会先从控制业务建立通信矩阵，记录资产进程、发起方向、工业操作、身份、周期、会话状态、失联和切换，而不是直接抄端口。然后结合路由、NAT、规则顺序和 HA 状态落地。防火墙允许只能证明网络路径，若同端口包含读写或工程下载，还需应用权限、控制器保护和语义监测。验收覆盖正常、主备切换、补传、维护和恢复，并用规则 UUID、会话日志、抓包与过程反馈交叉验证。

### 常见追问

**问：防火墙允许 Modbus TCP 502，怎样做到只读？**

答：若设备支持可靠的工业 DPI，可限制功能码；否则网络层只能限制双方，需通过应用代理、PLC 权限、只读数据路径和监测补偿，并明确控制边界。

**问：为什么零命中规则不能直接删？**

答：它可能服务备用、年度维护或故障状态。应结合统计周期、计数重置、业务确认和切换测试判断。

## 13. 实操建议

### 实验一：从抓包生成矩阵

在隔离仿真网收集 HMI、PLC、历史和工程流量，提取五元组、发起方、周期和应用操作，再补充身份、故障和恢复字段。比较自动发现与工程资料差异。

### 实验二：规则顺序与 NAT

配置宽泛规则和具体规则，观察命中顺序；加入 NAT 后比较两侧抓包和日志地址。建立可复现的会话关联表。

### 实验三：状态与切换

调整实验防火墙 TCP/UDP 超时并模拟 HA 切换，观察长连接、周期 UDP、应用重连和数据质量。禁止在生产边界直接试验。

### 实验四：工业能力验证

用仿真协议服务分别产生读取、写入和异常请求，验证五元组规则与 DPI 规则的能力差异，并将应用响应映射到仿真过程变量。

## 14. 总结与检查清单

- [ ] 能区分 ACL、状态防火墙、代理与工业 DPI；
- [ ] 能建立含业务、身份、状态和恢复的通信矩阵；
- [ ] 能审阅规则顺序、对象、NAT、隐式行为和 HA；
- [ ] 能解释 TCP/UDP 超时对工业会话的影响；
- [ ] 能限定日志和规则命中的证据边界；
- [ ] 能把合法通道滥用映射到控制与物理后果；
- [ ] 能安全验证正常、故障、切换和恢复场景。
