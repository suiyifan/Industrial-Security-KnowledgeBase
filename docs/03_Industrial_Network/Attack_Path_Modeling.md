# 工业攻击路径与屏障建模

## 1. 原理解释：攻击路径不等于网络连通图

网络拓扑回答设备怎样连接，攻击路径回答攻击者在给定能力、身份、知识和工况下能否从入口跨越屏障，获得某项工业控制能力并造成可信后果。`ping` 通、端口开放或存在漏洞，只能证明路径的一部分。

成熟路径必须逐层验证：网络是否可达，身份是否可用，应用操作是否允许，攻击者是否理解工程对象，控制器是否在合适模式，命令是否穿过逻辑和设备许可，保护层是否会阻断，物理过程是否支持推断后果。任何未验证环节都应标为假设。

## 2. 六层攻击路径模型

### 2.1 入口与执行环境

入口可能是办公钓鱼、远程运维、供应链文件、互联网边界、无线/蜂窝、现场端口、可移动介质或云平台。记录攻击者获得的是浏览器会话、普通用户、管理员、设备证书、物理接入还是工程文件。

### 2.2 网络路径

检查 Zone、VLAN、路由、NAT、防火墙、代理、VPN、双网卡和管理面。网络可达要具体到源资产/进程、目标接口、发起方向、协议和时间窗口，并考虑备用与故障路径。

### 2.3 身份与信任

分析域信任、共享管理员、服务账号、证书、API 密钥、本地设备密码和堡垒授权。网络隔离可能被集中身份或备份平台的高权限通道绕过；MFA 也不能阻止被控制的已登录终端。

### 2.4 应用与工程能力

区分观察、操作、维护、工程和管理能力。相同端口内可能包含读取、写入、程序下载、固件升级或模式切换。攻击者还需软件版本、驱动、项目密码、设备型号和工程语义。

### 2.5 控制与设备状态

验证控制器 Run/Program、手/自动、本地/远程、许可、联锁、命令源、任务覆盖和输出刷新。网络写入可能被程序下一扫描覆盖；设备就地模式可能拒绝远程命令；强制或旁路则可能跨越正常逻辑。

### 2.6 物理过程与保护

把数字动作映射到执行器、物料/能量、过程动态、报警、设备保护和 SIS。攻击后果取决于当前负荷、阀位、库存、时间尺度和人员响应。独立保护有效时，最坏结果可能是安全停车而非灾难事故。

## 3. 节点、边和前置条件

攻击图中的节点可以是“取得工程站管理员”“获得 PLC 下载能力”或“使进料阀打开”，边表示从一个能力到另一个能力所需动作。每条边至少记录：

| 字段 | 说明 |
| --- | --- |
| 前置条件 | 权限、网络、模式、工况、工具 |
| 行为 | 登录、横向、下载、写入、旁路等 |
| 获得能力 | 新身份、会话、工程或物理能力 |
| 证据 | 配置、流量、日志、工程文件、实验 |
| 屏障 | 防火墙、MFA、签名、联锁、SIS |
| 失败条件 | 规则拒绝、版本不符、模式错误等 |
| 可观测性 | 哪些系统能检测，存在什么盲区 |
| 恢复要求 | 凭据、程序、状态、现场过程怎样恢复 |

不要把资产本身当作唯一节点。一台工程站被攻陷后，攻击者可能仍缺少项目、控制器在线密码或维护窗口；一台 HMI 普通用户可能只能观察，却不能下载逻辑。

## 4. 数据流、控制流与状态流

攻击路径建模同时跟踪：

```text
数据流：凭据/项目/命令 → 网络与应用 → 控制变量 → 反馈/历史
控制流：身份授权 → 工程操作 → PLC 逻辑 → 输出 → 执行器
状态流：外部 → 企业立足点 → DMZ → 工程权限 → 控制权限
       → 过程偏移 → 保护动作 → 恢复
```

攻击者可能只操纵数据流，例如伪造历史值；也可能改变控制流，例如下载逻辑；或利用状态流，例如在维护模式植入参数，待恢复自动运行后生效。建模不能只搜索实时写命令。

## 5. 屏障分类与有效性

### 5.1 预防屏障

分区、防火墙、MFA、设备准入、应用白名单、程序签名、控制器写保护和物理机柜。判断其是否覆盖目标路径、是否启用、是否可被共享账号或旁路绕过。

### 5.2 检测屏障

EDR、VPN/堡垒审计、工业 IDS、配置变更、工程下载、过程一致性、报警和现场巡检。检测只降低暴露时间，不一定阻止动作；告警若无人响应不能获得完整风险降低信用。

### 5.3 响应与恢复屏障

会话撤销、隔离方案、可信备份、控制器恢复、现场手动控制和受控启动。响应动作本身可能影响生产，需要预先测试。

### 5.4 独立保护屏障

设备保护、机械限制、联锁和 SIS 限制物理后果。必须验证传感器、供电、网络、工程站和最终元件是否与被攻击系统独立。图上两套系统不代表没有共因。

屏障状态应记录为 `设计存在、配置存在、运行有效、定期验证`。仅在制度或图纸中存在的控制不能当成已验证屏障。

## 6. 证据等级与结论约束

可使用以下等级：

- **理论可能**：依据一般机制推断，未确认现场实现；
- **配置支持**：拓扑、规则、账号或项目显示路径存在；
- **行为观察**：日志/流量证明某步实际发生；
- **隔离验证**：在授权同类环境验证技术能力；
- **现场确认**：经批准确认目标实现和状态；
- **后果证实**：有真实过程响应和保护事件证据。

报告要为每条边标注等级。实验室 PLC 可写不能证明生产 PLC 可写；防火墙端口允许不能证明应用操作；历史案例也不能无条件外推到不同工艺。风险可以在证据不完整时提出，但必须公开假设和最坏/最可信边界。

## 7. 多路径、共同依赖与时间窗口

攻击者通常有多条路径：企业—DMZ—工程站、供应商 VPN—堡垒、备份服务器、无线网关、管理平台或现场笔记本。仅封堵最显眼路径可能把攻击转移到恢复、监测或管理通道。

共同依赖包括 AD、虚拟化、备份、网络管理、PKI、时间、更新和供应商云。它们横跨多个 Zone，失陷后可能同时削弱多道屏障。攻击图应把这些平台作为高影响节点，而不是背景资产。

时间窗口同样关键：工程下载只在维护时允许，SIS 旁路有期限，远程账号夜间启用，过程只有在满负荷或特定顺序才出现危险。路径概率和后果会随状态变化，不能用静态拓扑给出永久结论。

### 7.1 安全风险不能只按路径长度排序

路径短不一定风险最高，路径长也不一定难以利用。一个需要五次横向移动但全部复用域管理员的路径，可能比必须取得现场钥匙和工程密码的两跳路径更容易。安全风险应综合入口暴露、前置条件难度、身份复用、工程知识、检测概率、工况窗口、屏障独立性和最大可信后果。

优先治理能同时切断多条路径的控制，例如分离企业与工业身份、限制网络管理平台、删除双网卡旁路、保护工程项目和控制器下载。对不能立即消除的旧设备路径，使用边界白名单、受控工程站、行为监测、现场许可和独立保护进行补偿。

### 7.2 防护有效性的验证

防护验证不能用“配置存在”结束。防火墙规则需要命中和阻断测试，MFA 需要检查遗留协议和服务账号，程序签名需要验证控制器是否强制执行，SIS 需要验证传感器与最终元件独立。测试优先在数字孪生、仿真或实验环境完成；生产侧采用配置审阅、日志证据和经批准的低风险验证。

建模结果应形成屏障改进计划：明确责任人、实施条件、验证证据、失效监测和恢复方式。无法验证的屏障标记为“设计声明”，不用于降低结论等级。

## 8. 从漏洞到物理后果

漏洞分析按以下链条：

```text
漏洞可触达
→ 获得代码执行/认证绕过/报文能力
→ 获取工程资产与语义
→ 形成控制或显示能力
→ 在当前模式与工况生效
→ 执行器改变物料或能量
→ 报警/保护/人员响应
→ 安全、质量、环境或生产后果
```

RCE 在工程站上严重，是因为可能取得项目、凭据和可信下载路径；但仍需确认工程软件、网络、控制器模式和保护。PLC 未认证写入严重，是因为可能改变变量；但变量是否影响输出、是否被覆盖和当前工况仍需验证。

## 9. 攻击链案例一：办公网到工程站

1. 钓鱼取得办公用户执行；
2. 窃取共享生产服务账号；
3. 通过过宽企业—生产规则访问 MES；
4. 利用相同管理员横向到历史服务器；
5. 经历史服务器第二网卡到工程网；
6. 从工程站项目读取 PLC 地址和变量；
7. 等待维护窗口下载修改逻辑；
8. 操纵 HMI/历史延迟发现。

屏障包括身份分离、DMZ 代理、主机防火墙、禁止双网卡、工程项目签名、下载审批、流量监测和过程一致性。每一步都应有证据和失败条件。

## 10. 攻击链案例二：远程供应商到 VFD

供应商云账号被盗后进入蜂窝网关，通过串口服务器修改 VFD 命令源、最高频率和自动重启。攻击者仍需设备型号、参数语义和现场远程模式。电机保护、硬接线许可、STO、锁定挂牌和过程联锁可能限制后果。

检测可关联云登录、网关配置、Raw TCP、参数写入、驱动器事件、电流/速度和流量。恢复需撤销云会话、恢复可信参数、确认能量隔离并执行方向和保护测试。

## 11. 工控场景：看似可达但攻击路径不成立

评估发现办公测试机可 TCP 连接到 PLC 端口，初步报告称可直接控制反应器。进一步验证发现连接终止在 DMZ 协议代理，代理仅发布历史缓存且拒绝写功能；生产 PLC 地址经 NAT 映射为不同测试模拟器。原结论把网络外观误当真实控制路径。

修订后风险是边界命名和日志容易误导、代理配置仍需保护，而不是“办公网直控反应器”。该案例说明风险报告必须验证目标身份、会话终止点、应用操作和过程映射。

## 12. 恢复建模

攻击路径终点不应停在影响。恢复需沿路径逆向处理：撤销外部会话和凭据，隔离入口，保全边界证据，恢复主机与项目，比较控制器在线版本，清除强制/旁路/临时参数，确认执行器和物料状态，受控启动并持续观察。

若只重装工程站而不轮换 PLC 密码、验证网关映射和现场状态，攻击能力可能保留。恢复图应记录依赖、责任人、进入/退出条件和回退方案。

## 13. 面试表达

### 60 分答案

> 工业攻击路径从初始入口经过横向移动到工程站、HMI 或 PLC。分析要看网络分区、账号权限、协议能力和防火墙，并评估最终生产影响。

### 90 分答案

> 我不会把网络可达直接写成控制后果，而会按入口、网络、身份、应用工程、控制状态和物理过程六层建模。每条边记录前置条件、获得能力、证据等级、失败条件和预防/检测/保护屏障。对漏洞继续验证项目语义、控制器模式、设备许可、工况和 SIS 独立性。结论区分理论、配置、行为、实验和现场证实，并把恢复路径纳入模型，避免只画一条从办公网到 PLC 的箭头。

### 常见追问

**问：办公网能访问工程站，是否一定能控制 PLC？**

答：不一定。还需工程站权限、项目/工具、控制器网络、认证、模式、变量语义和工况；但这是需要优先阻断和验证的高风险路径。

**问：攻击图中的防火墙算不算有效屏障？**

答：只有确认规则、对象、运行状态、管理安全和定期验证后才算；图纸上存在不能直接计入。

## 14. 实操建议

### 实验一：六层路径表

用仿真 HMI/PLC 和分区网络建立从客户端到变量写入的路径，为每层填写前提、证据、屏障和失败条件。

### 实验二：屏障对照

依次加入防火墙白名单、独立账号、控制器写保护和过程联锁，观察每项控制阻断哪一步，不能阻断哪一步。

### 实验三：证据等级

对同一“PLC 可写”结论分别使用端口扫描、规则配置、抓包、仿真写入和过程反馈，练习限定结论强度。

### 实验四：恢复桌面演练

假设工程站与 PLC 项目被篡改，沿身份、网络、项目、在线状态和现场过程设计恢复与验收顺序。

## 15. 总结与检查清单

- [ ] 能区分网络连通图与工业攻击路径；
- [ ] 能按六层验证攻击前提和能力；
- [ ] 能为每条边记录证据、失败条件与屏障；
- [ ] 能评估共同依赖、时间窗口和多路径；
- [ ] 能从漏洞严谨推导到物理后果；
- [ ] 能区分理论、配置、行为、实验和现场证据；
- [ ] 能把恢复与验收纳入攻击路径模型。
